# The Privacy Problem

In the last few chapters, we reviewed how Lightning payments are sent across multiple hops and, crucially, what information each hop needs in order to set up an <g>HTLC</g> on its given channel.

Remember, since Alice is the one sending the payment, she's the one who looks at the entire path to calculate the fees and timelocks required for each hop.

So the natural next question is: how does Alice tell each hop how to set up its given HTLC? Recall that each individual hop isn't supposed to know where it is in the larger route, which means it will have no way of knowing which absolute timelock to set or how much to include in the HTLC amount unless Alice tells them directly.

One simple and naive solution (don't try this at home!) would be for Alice to send a plaintext message to Bob. Once Bob receives it, he simply reads his portion, removes it, and passes it on. Below is a visual that demonstrates what this would look like. Can you spot the problem here?

In the above example, Bob and Charlie both learn information they shouldn't! Namely, Bob learns who the next two hops are and Charlie learns the final destination. This kind of data leak is a big no-no if you're trying to build a global financial payment rail that also protects people's privacy.

Instead, we should strive to meet the following rule: **intermediate nodes should learn just enough to set up their HTLC, and nothing more.** If we can pull that off, then we're well on our way to building a privacy-preserving payment infrastructure.

## The properties we want

So, if the above isn't a good enough solution (it's not!), what is? Well, let's list out the properties an ideal payment-routing scheme would have, and what each participant should and shouldn't know after the payment completes.

**The sender (Alice):** Has to know the full route. Since she's the one constructing the payment, this is unavoidable. Note: if the recipient provides an introduction node for a blinded path, then the sender would only know the path up until the introduction node, but that is an advanced topic that is beyond the scope of this introductory example.

**A forwarder (Bob, Charlie):** Knows only:
- The hop *before* it, because that's who sent the payment info that just arrived.
- The hop *after* it, because that's who it needs to forward the payment to.
- The amount it should forward, and when the HTLC expires.

A forwarder should ***not*** know:
- Who started the payment.
- Who receives the final payment.
- How long the route is.
- Where in the route it sits (first hop, last hop, etc.).
- Anything (ex: identity) about any hops that aren't its immediate neighbors.

**The destination (Dave):** Knows that the payment is for him (the route ends with him), and he learns whatever payment metadata Alice included, like an invoice reference (`payment_secret`) or description. He does ***not*** know who paid him, unless Alice voluntarily reveals that or includes hints in the invoice.

## A first attempt: per-hop keys

Let's start to inch our way towards a more private way to transmit payment data to each hop along the path.

Since each Lightning node already has a public key that serves as its identity, Alice can take each hop's payload and encrypt it with *that hop's* public key. Bob's hop payload gets encrypted with Bob's public key. Charlie's with Charlie's. Dave's with Dave's. Then, when the message moves down the payment path, each hop can decrypt **only their own hop payload** with their private key. Everyone else's hop payloads look like noise to them. Progress!

<encrypted-slice-reveal></encrypted-slice-reveal>

Take a moment and verify the following:

- Bob's private key decrypts Bob's data only. Charlie's and Dave's hop payloads stay encrypted.
- When Bob forwards, he peels his own hop payload off the wire and passes Charlie's and Dave's hop payloads along, still encrypted.
- Charlie does the same. Dave decrypts the final hop payload with his own private key.

This is the *intuition* behind onion routing, which the rest of the course will focus on. The actual construction is much more clever (and complicated) than this, but the general idea is similar to what we just saw in the above visual: **per-hop encryption, per-hop keys, and only the intended recipient can decrypt their own payment data.**

## Houston, we (still) have a problem

Per-hop encryption fixes the most obvious problem. Namely, each hop's *contents* are now cryptographically hidden from everyone except the intended recipient. But look back at the visual and check it against our security properties list. We still have work to do!

<checkpoint id="cp-still-vulnerable-draft"></checkpoint>

There's a second piece we still need. A hop's node-identity key never changes, so if Alice just kept encrypting straight to it, every payment she ever routes through Bob would be locked with the same key, which makes her payments easy to tie together. What we'd really like is a **shared secret** that's unique to each payment, one that Alice and Bob can both compute on their own but that's used once and never again. Here's the tricky part: Alice only holds Bob's public key, and Bob only holds Alice's. So can you think of a way the two of them could each take their own private key, mix in the other's public key, and arrive at the very same secret, without ever exchanging a single message?

## Up next

We're making real progress: we've built a strong intuition for both the information each forwarder needs to receive *and* the privacy properties our payment path has to satisfy. To actually pull this off, though, we'll need to lean on some more advanced cryptography on our journey to **<g>Sphinx</g>**, the actual onion-routing construction Lightning uses.

In the next few chapters, we'll dig into the following two cryptography tools:

- A **shared-secret derivation** scheme so Alice and each hop can derive a fresh secret, unique to every payment, from nothing but the public keys on the gossip network.
- A **fixed-size packet** so the message doesn't shrink as hops peel off their hop payload, ensuring the message looks identical at every hop.
