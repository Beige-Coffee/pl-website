# BIP32 Key Derivation

## What Is BIP32?

**BIP32** (Bitcoin Improvement Proposal 32) defines a standard for **Hierarchical Deterministic (HD) wallets**. It specifies how to derive a tree of cryptographic key pairs from a single master seed. Each key in the tree is identified by a **derivation path** like `m/44h/0h/0h/0/0`.

The core idea is simple: from a parent key and an index number, you can deterministically compute a child key. By chaining multiple derivation steps together, you create a hierarchy of keys.

## Master Key Generation

Everything starts with a seed (typically 16-64 bytes of entropy). The master key is derived using HMAC-SHA512:

```python
import hmac
import hashlib

def master_key_from_seed(seed: bytes) -> tuple[bytes, bytes]:
    I = hmac.new(b"Bitcoin seed", seed, hashlib.sha512).digest()
    master_secret = I[:32]   # Left 32 bytes: the private key
    master_chain = I[32:]    # Right 32 bytes: the chain code
    return master_secret, master_chain
```

The HMAC-SHA512 output is 64 bytes. The left half becomes the master private key, and the right half becomes the **chain code**. The chain code is additional entropy that prevents child keys from being predictable even if the parent key is known.

## Child Key Derivation

Given a parent key pair (private key, chain code) and an index, we derive a child key pair. There are two types of derivation:

### Non-Hardened Derivation (index 0 to 2^31 - 1)

Non-hardened derivation uses the parent **public key** as input to the HMAC:

```python
def derive_child(parent_key: bytes, parent_chain: bytes, index: int) -> tuple[bytes, bytes]:
    # Non-hardened: HMAC input is parent public key + index
    parent_pubkey = private_to_public(parent_key)
    data = parent_pubkey + index.to_bytes(4, 'big')
    I = hmac.new(parent_chain, data, hashlib.sha512).digest()

    child_key = (int.from_bytes(I[:32], 'big') + int.from_bytes(parent_key, 'big')) % SECP256K1_ORDER
    child_chain = I[32:]
    return child_key.to_bytes(32, 'big'), child_chain
```

The child private key is computed by adding the left 32 bytes of the HMAC output to the parent private key (modulo the curve order). The right 32 bytes become the child's chain code.

### Hardened Derivation (index 2^31 to 2^32 - 1)

Hardened derivation uses the parent **private key** as input to the HMAC:

```python
def derive_child_hardened(parent_key: bytes, parent_chain: bytes, index: int) -> tuple[bytes, bytes]:
    # Hardened: HMAC input is 0x00 + parent private key + index
    # The index has bit 31 set (>= 0x80000000)
    data = b'\x00' + parent_key + index.to_bytes(4, 'big')
    I = hmac.new(parent_chain, data, hashlib.sha512).digest()

    child_key = (int.from_bytes(I[:32], 'big') + int.from_bytes(parent_key, 'big')) % SECP256K1_ORDER
    child_chain = I[32:]
    return child_key.to_bytes(32, 'big'), child_chain
```

The difference is in the HMAC input data. Hardened derivation prepends `0x00` followed by the parent private key, while non-hardened derivation uses the compressed parent public key.

## Hardened vs Non-Hardened: Why It Matters

The distinction between hardened and non-hardened derivation is critical for security:

**Non-hardened** derivation has a useful property: you can derive child **public keys** from the parent **public key** alone, without knowing the private key. This is great for watch-only wallets. However, it has a dangerous weakness: if an attacker learns both a child private key and the parent public key (along with the chain code), they can compute the parent private key.

**Hardened** derivation prevents this attack entirely. Since the parent private key is used as input (not the public key), knowing a child private key reveals nothing about the parent. However, you cannot derive child public keys without the parent private key.

In the Lightning key derivation path `m/1017h/0h/{family}h/0/{index}`:

- The first three levels (`1017h`, `0h`, `{family}h`) use hardened derivation. This means compromising a key at one family level cannot be used to derive keys in other families.
- The last two levels (`0`, `{index}`) use non-hardened derivation. This allows efficient derivation of channel-specific keys.

## The Derivation Algorithm

Putting it all together, deriving a key along a path like `m/1017h/0h/2h/0/5` works as follows:

1. Start with the master key pair from the seed.
2. Derive child at index `1017 + 0x80000000` (hardened) from master.
3. Derive child at index `0 + 0x80000000` (hardened) from step 2.
4. Derive child at index `2 + 0x80000000` (hardened) from step 3.
5. Derive child at index `0` (non-hardened) from step 4.
6. Derive child at index `5` (non-hardened) from step 5.

Each step produces a new (key, chain_code) pair that becomes the parent for the next step.

```python
def derive_path(seed: bytes, path: list[int]) -> bytes:
    key, chain = master_key_from_seed(seed)
    for index in path:
        if index >= 0x80000000:  # Hardened
            data = b'\x00' + key + index.to_bytes(4, 'big')
        else:  # Non-hardened
            pubkey = private_to_public(key)
            data = pubkey + index.to_bytes(4, 'big')
        I = hmac.new(chain, data, hashlib.sha512).digest()
        key = ((int.from_bytes(I[:32], 'big') + int.from_bytes(key, 'big'))
               % SECP256K1_ORDER).to_bytes(32, 'big')
        chain = I[32:]
    return key
```

## Important Implementation Details

### Compressed Public Keys

BIP32 uses **compressed** public keys (33 bytes). A compressed public key is the x-coordinate of the elliptic curve point prefixed with `0x02` (if y is even) or `0x03` (if y is odd).

### The secp256k1 Curve Order

All arithmetic is performed modulo the secp256k1 curve order:

```text
n = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141
```

If the derived key is zero or greater than or equal to the curve order, the index is considered invalid and the next index should be tried. In practice, this is astronomically unlikely.

### Chain Codes

Chain codes are 32 bytes of additional entropy that travel alongside the private key through each derivation step. They ensure that even if someone knows the parent key, they cannot derive child keys without also knowing the chain code.
