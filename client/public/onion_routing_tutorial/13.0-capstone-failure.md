# Capstone: Failure Path

Real Lightning isn't always green checkmarks. Hops go offline, channels run dry, timelocks turn out to be too tight. When something goes wrong, the failing hop wraps an error onion (chapter 10) and sends it back. Alice's job is to decrypt it, identify which hop failed, and decide what to do next (retry on a different route, surface the error to the user, or both).

This chapter exercises that failure path against the same 3-hop simulator we used in chapter 11. The simulator is configured to fail the payment at a specific hop with a specific BOLT 4 failure code. Your code submits the onion just like in chapter 11, but instead of a preimage you get a wrapped error packet on the way back.

## What happens

When you click **Send Payment** on the panel below:

1. **Builder phase** runs as before: your `OnionPacketBuilder.build` produces a 1,366-byte packet.
2. **Forward propagation**: the simulator's Bob and Charlie process the packet normally (peel, validate, forward).
3. **Failure injection**: the simulator decides which hop will fail (configurable; default is hop 1, Charlie). At that hop, instead of forwarding, the simulator builds an error using its `um` and `ammag` keys and sends 292 bytes back upstream.
4. **Error wrapping**: each upstream hop wraps the error with another `ammag` layer.
5. **Decryption phase** (your code): your `decrypt_error_onion` runs against the wrapped bytes plus the `(um, ammag)` chain you derive from the shared secrets. It returns either `(failing_hop_index, failure_message)` or `(None, None)` if no layer's HMAC verified.
6. **Display**: the panel shows the failing hop and the decoded failure code, plus a play-by-play of the error's path back through the route.

## What success looks like

The capstone is "successful" when your code:

- Builds the forward onion correctly (so the failure happens at the *injected* hop, not because of a bug in your builder).
- Derives the per-hop `um` and `ammag` keys from the shared secrets your builder computed.
- Iterates the `decrypt_error_onion` algorithm in the right order (outermost layer first) and reports the right `failing_hop_index`.
- Decodes the failure message bytes correctly using the u16 length prefix.

If your `decrypt_error_onion` returns the right hop index and message bytes that match what the simulator injected, the panel reports success on this chapter.

## Common failure-decryption bugs

A few things tend to trip people up on the error path:

- **Wrong order of hops.** The outermost layer corresponds to hop 0 (the first forwarder), not the failing hop. Iterate `for i in range(len(hop_keys))`, peeling each in order, and check each one's HMAC. The first match is the failing hop.
- **Wrong key labels.** It's `b"um"` and `b"ammag"`, not `b"mu"` and `b"rho"`. The forward-direction keys (`mu`, `rho`) are not used on the return path.
- **HMAC over the wrong bytes.** The HMAC tag is the leading 32 bytes of the peeled buffer; the message it covers is the trailing 260 bytes. Check `HMAC(um, peeled[32:]) == peeled[:32]`, not the other way around.
- **Forgotten length-prefix parse.** The padded message is 260 bytes; the actual failure message is shorter. Parse the u16 `failure_len` at offset 0 of the payload and use it to slice the real message bytes (don't just strip trailing zeros, since some failure messages contain internal zero bytes).

<details>
<summary>What does a real Lightning wallet do with a decoded failure?</summary>

A wallet that receives a decoded failure does several things:

- **Update the network graph.** If the failure is `temporary_channel_failure` with the `UPDATE` flag, parse the included `channel_update` message and replace the stale entry in the local graph. This avoids retrying the same broken edge.
- **Score the failing hop.** Implementations track per-hop reliability over time. A hop that fails frequently gets downweighted in future pathfinding.
- **Retry on a different route.** If the failure is recoverable (channel out of liquidity, expired, etc.), the wallet runs pathfinding again with the failing edge excluded and tries the new route.
- **Surface to the user.** Some failures are unrecoverable (invoice expired, wrong destination key). These get surfaced as a payment-failed error in the wallet UI.

The capstone doesn't simulate any of this; it just identifies the failing hop and the error code. But if you wanted to extend the exercise into a more realistic Lightning client, this is the next layer.

## You've finished the course

Once you've cleared this chapter, you've implemented the full BOLT 4 onion routing protocol from scratch. Your code can:

- Build a spec-compliant 1,366-byte Sphinx packet from a route plus a list of hop payloads.
- Peel a single layer of an onion, advancing the blinding chain and producing a packet for the next hop.
- Process an inbound onion end-to-end: verify integrity, parse the TLV, branch on forwarder-vs-destination, and produce the appropriate result.
- Build an encrypted error onion from a failing hop's perspective.
- Decrypt an error onion from the sender's perspective and identify which hop failed.
