# Failure Handling

Not every payment succeeds. Maybe Carol doesn't have enough liquidity in her channel with Dave. Maybe a channel was closed between the time Alice found the route and the time the onion arrived. Maybe the CLTV expiry is too close to the current block height.

Whatever the reason, the failing hop needs to tell Alice what went wrong. But there's a problem: the onion was designed for one-way communication from Alice to Dave. Error messages need to travel in the opposite direction, from the failing hop back to Alice. And they need to be just as private as the onion itself.

## The Problem with Plaintext Errors

Imagine Carol discovers she can't forward the payment to Dave. She could just send a plaintext error message back to Bob: "I don't have enough liquidity for channel 700000x3x0." Bob forwards it to Alice. Alice now knows that Carol failed and why.

But Bob also sees the error. He learns that Carol tried to forward to Dave via a specific channel. If errors are unencrypted, every intermediate hop learns the failure details, which leaks information about the route structure. Even worse, a malicious hop could modify the error message, blaming a different node to influence Alice's future routing decisions.

The solution: **encrypted error packets** that use the same key derivation from the forward path, but with two keys we haven't used yet.

## The Backward Keys: `um` and `ammag`

Remember when Alice derived five keys from each shared secret? Two of those keys, `um` and `ammag`, have been waiting for this moment.

- **`um` key**: Used to compute HMACs on error messages. This lets Alice verify which hop generated the error.
- **`ammag` key**: Used to encrypt (obfuscate) error messages. Each hop adds a layer of encryption as the error travels backward.

The names are hints: "um" is "mu" backwards, and "ammag" is "gamma" backwards. The forward path uses `mu` for HMAC and `rho` for encryption. The backward path uses `um` for HMAC and `ammag` for encryption. Symmetric in spirit, separate in practice.

## How Carol Constructs an Error

Let's say Carol discovers she doesn't have enough liquidity to forward to Dave. She needs to construct an error packet. Here's the process:

**Step 1: Build the error message.** Carol assembles a failure code (2 bytes, big-endian) and optional failure data. For a liquidity failure, the code is `0x1007` (`temporary_channel_failure`). She may include additional data (like a `channel_update` message), but for now let's keep it simple with just the code.

```python
failure_code = 0x1007  # temporary_channel_failure
failure_data = b""     # no additional data
error_msg = failure_code.to_bytes(2, 'big') + len(failure_data).to_bytes(2, 'big') + failure_data
```

**Step 2: Pad to 256 bytes.** The error message is padded with zeros to a fixed 256-byte size, regardless of the actual content. This prevents the error length from leaking information about which failure occurred.

```python
padded = error_msg.ljust(256, b'\x00')
```

**Step 3: Compute the HMAC.** Using the `um` key from Carol's shared secret, compute an HMAC over the padded error:

```python
error_hmac = HMAC-SHA256(um_key, padded)  # 32 bytes
```

**Step 4: Prepend the HMAC.** Combine the HMAC and padded error into a 288-byte packet:

```python
error_packet = error_hmac + padded  # 32 + 256 = 288 bytes
```

**Step 5: Obfuscate with `ammag`.** XOR the entire 288-byte packet with a cipher stream generated from the `ammag` key:

```python
stream = generate_cipher_stream(ammag_key, 288)
error_packet = xor(error_packet, stream)
```

Carol sends this 288-byte encrypted error packet back to Bob via `update_fail_htlc`.

## Intermediate Hop Wrapping

Bob receives the error packet from Carol. He doesn't know what's in it (he can't decrypt it), but he adds another layer of obfuscation before passing it back to Alice. This is much simpler than the initial construction:

```python
stream = generate_cipher_stream(bob_ammag_key, 288)
wrapped = xor(error_packet, stream)
```

Bob just XORs the packet with his own `ammag` cipher stream. No HMAC computation, no padding. Just one more layer of obfuscation. He sends the result to Alice.

If there were more intermediate hops, each one would add another XOR layer the same way.

## Alice Unwraps the Error

Alice receives the error and needs to figure out two things: which hop generated it, and what went wrong. She has the shared secrets for every hop (computed during packet construction), so she can reverse the obfuscation layer by layer.

For each hop in forward order (Bob, Carol, Dave), Alice:

1. **XOR with the hop's `ammag` stream** to remove one layer of obfuscation
2. **Check the HMAC with the hop's `um` key**: extract the first 32 bytes as the claimed HMAC, compute `HMAC-SHA256(um_key, remaining_256_bytes)`, and compare

If the HMAC matches, this hop generated the error. Alice extracts the failure code and failure data from the decrypted payload. If the HMAC doesn't match, Alice moves on to the next hop and removes another layer.

In our example:
- Alice XORs with Bob's `ammag` stream. Checks HMAC with Bob's `um` key. **No match** (Bob didn't generate this error).
- Alice XORs with Carol's `ammag` stream. Checks HMAC with Carol's `um` key. **Match!** Carol generated the error.
- Alice extracts the failure code: `0x1007` (temporary_channel_failure).

Now Alice knows: Carol couldn't forward the payment due to insufficient liquidity. Alice can retry with a different route that avoids the Carol-Dave channel.

<details>
  <summary>Why are error messages encrypted rather than sent in plaintext?</summary>

Encrypted errors serve two purposes: **privacy** and **integrity**. Without encryption, every intermediate hop can read the error, learning which downstream hop failed and why. This leaks route structure information. For example, if Bob sees Carol's plaintext error about channel 700000x3x0, he learns that Carol was forwarding to Dave on that specific channel.

Integrity is equally important. Without authentication (the `um` HMAC), a malicious hop could forge error messages to blame innocent nodes. If Bob wants to make Alice avoid Carol in the future, he could fabricate an error claiming Carol failed. The HMAC prevents this: Alice can verify exactly which hop generated the error because only that hop's `um` key produces a valid HMAC.
</details>

<details>
  <summary>What if Alice can't identify which hop failed?</summary>

If Alice iterates through all hops and no HMAC matches, the error packet is malformed. This could happen if an intermediate hop corrupted the packet (either by accident or maliciously). In this case, Alice knows the payment failed but can't determine where or why. She can still retry, but she has no information to improve her route selection. Most implementations treat this as a protocol violation and may penalize the route.
</details>
