# Shared Secrets per Hop

Hmm, so the last section left us with a bit of a dilemma. Even though Alice has identified the best path to route her payment through, she still doesn't have a way to encrypt the payment instructions so that only the intended recipient can read them. Or, does she!?

The *secret* (pun intended) to making this work is for Alice to derive a unique shared private key for *each hop* in the payment path that. Crucially, only Alice and the unique hop will be able to derive the shared private key. By doing this, she can ensure that each message stays encrypted, and only the intended recipient is able to decrypt it. To make this trick work, we'll use a tool that we reviewed in the Noise handshake course: **Elliptic Curve Diffie-Hellman (ECDH)**.


## A quick refresher on ECDH

Recall that a public key is just a large, private number multiplied by the elliptic curve generator point: `A = a · G`. It's worth pinning down the notation while we're here, since we'll lean on it for the rest of the course: for an elliptic-curve keypair, the lowercase letter is the private scalar and its uppercase twin is the public point (`a` private, `A` public). Watch the case, it's the only thing that tells `e` (a private key) apart from `E` (a public one). The resulting point `A` is also a point on the curve, but it has a special property such that you cannot recover `a` from `A`, even though both `A` and `G` are publicly known. To do so, you'd have to solve the **elliptic curve discrete log problem**, which is computationally infeasible (let's ignore the quantum debate for now).

Another neat property of elliptic curves is that scalar multiplication on the curve is **commutative**. For a simple example of commutativity: `3 · 5` is the same as `5 · 3`. If we extend that example to elliptic curves, given Alice's keypair `(a, A)` and Bob's `(b, B)`, the point `a · B` equals `b · A`, because both expand to `ab · G`. This means that, **if Alice and Bob trade public keys**, they can both compute a shared point that only they know. An observer may see `A` and `B` on the wire, but they can't combine them into `ab · G` without one of the private keys, which they don't have and cannot compute. If Alice and Bob take the SHA256 hash of this point, they end up with a **shared secret**, which they can use to encrypt and decrypt information sent between them.

## First attempt: Alice reuses her node key

Now, let's connect ECDH back to our payment example. Armed with the ECDH toolkit, **one option** Alice has is to use her Lightning node's existing keypair. She'd take her node private key and combine it with each forwarder's node public key (already broadcast on the gossip network) to derive a shared secret with that forwarder.

Then she'd embed her own node public key at the top of the packet. Each forwarder reads it, performs ECDH between their own node's private key and that public key, and derives the same shared secret. With that secret, they can decrypt the slice meant for them.

## Second attempt: a fresh ephemeral key per hop

So, our first attempt wasn't ideal. Let's see if we can do better!

Instead of using her Lightning node's private key, Alice should generate a fresh, throwaway keypair for each hop in the payment path. We call these **ephemeral** keypairs because they are short-lived, as they are only ever used for this specific payment.

For our example, Alice would generate one `(ephemeral private key, ephemeral public key)` pair per forwarder, keeping our lowercase-private, uppercase-public convention: `(e_Bob, E_Bob)`, `(e_Charlie, E_Charlie)`, and `(e_Dave, E_Dave)`.

> JUST TO BE CLEAR: In this scenario, Alice must create three ephemeral key pairs for this payment!

 With each keypair in hand, Alice performs ECDH between each ephemeral private key and the matching hop's Lightning node public key (same as in attempt one) to derive a shared secret with that hop. Then she attaches all three ephemeral pubkeys to the packet, `E_Bob`, `E_Charlie`, and `E_Dave`, one per hop. These pubkeys are included unencrypted, sitting in the packet next to the still-encrypted hop payloads. They have to be readable: a forwarder needs its ephemeral pubkey to perform ECDH and derive the shared secret, so the pubkey can't be hidden inside the very payload that secret is meant to decrypt.

Each hop retrieves its own ephemeral public key from the packet, performs ECDH between that key and its own node private key, and recovers the matching shared secret, which it then uses to decrypt its hop payload. The identity leak from the first attempt is now gone: the ephemeral keys aren't tied to any Lightning node. Yay!

## Third (and last!) attempt: a single ephemeral key, advanced by blinding

Okay, enough funny business. Let's review how things actually work in the Lightning Network.

Instead of creating a unique ephemeral keypair for each hop, Alice will generate **one** keypair, called the **session key**. This keypair is the starting point for a chain of key derivations. Each hop will deterministically derive its shared secret from the session key plus information that accumulates as the payment routes through the network. This way, only the *first* ephemeral key ever needs to be embedded in the packet. For every subsequent hop, the *next* hop's ephemeral key is computed on the fly by the current hop and written into the packet before forwarding.

If what we just reviewed above isn't immediately clear, don't worry! We're going to take it step by step below. Buckle up!

First, Alice will begin by generating a random one-time **session key**. Then, for each hop in the route, she will derive an ephemeral private key, an ephemeral public key, a shared secret with that hop, and (for every hop except the final one) a **blinding factor** that the next hop will use to advance the chain. Let's walk through this below.

> NOTE: If you don't know what a **blinding factor** is, don't worry. We'll review it shortly!

For Bob (hop 0):
- Alice sets the ephemeral private key equal to the <g>session key</g>, which is a random private key that Alice generated for this payment: <m>e_AB</m> = sessionkey
- She derives the ephemeral public key by multiplying <m>e_AB</m> by the generator point: <m>E_AB</m> = <m>e_AB</m> · <m>G</m>
- Next, she derives her shared secret with Bob using his node's public key: <m>ss_AB</m> = SHA256(<m>e_AB</m> · <m>B</m>)
- Finally, she derives the <g>blinding factor</g> for Bob: <m>bf_AB</m> = SHA256(<m>E_AB</m> ‖ <m>ss_AB</m>)

For Charlie (hop 1):
- Alice derives the next ephemeral private key by tweaking Bob's with the blinding factor: <m>e_AC</m> = <m>bf_AB</m> · <m>e_AB</m> (mod <m>n</m>, the curve's order, which keeps the result a valid private key)
- She derives Charlie's ephemeral public key by multiplying <m>e_AC</m> by the generator point: <m>E_AC</m> = <m>bf_AB</m> · <m>E_AB</m>
- Next, she derives her shared secret with Charlie using his node's public key: <m>ss_AC</m> = SHA256(<m>e_AC</m> · <m>C</m>)
- Finally, she derives the blinding factor for Charlie: <m>bf_AC</m> = SHA256(<m>E_AC</m> ‖ <m>ss_AC</m>)

For Dave (hop 2):
- Alice derives the next ephemeral private key by tweaking Charlie's with the blinding factor: <m>e_AD</m> = <m>bf_AC</m> · <m>e_AC</m> (mod <m>n</m>)
- She derives Dave's ephemeral public key by multiplying <m>e_AD</m> by the generator point: <m>E_AD</m> = <m>bf_AC</m> · <m>E_AC</m>
- Next, she derives her shared secret with Dave using his node's public key: <m>ss_AD</m> = SHA256(<m>e_AD</m> · <m>D</m>)
- No blinding factor for Dave: he's the final hop, so there's no successor that would need one.

Now, she just needs to include <m>E_AB</m> into the start of the onion, and each forwarder will be able to derive the rest of the ephemeral public keys themselves.

## Why the forwarders can advance the chain

Hopefully it's now clear how Alice can compute all the necessary cryptographic material (<m>E_AB</m>, <m>ss_AB</m>, <m>E_AC</m>, <m>ss_AC</m>, <m>E_AD</m>, <m>ss_AD</m>) in advance. But it's less obvious why Bob (or any other node in the payment path) can independently compute the same blinding factor and keep the rest of the packet routing on track.

Let's review what Bob does when he receives the onion:

1. First, Bob takes <m>E_AB</m> from the packet header and performs ECDH between it and his node private key: <m>ss_AB</m> = SHA256(bob_priv · <m>E_AB</m>). **Crucially, this is the same <m>ss_AB</m> thta Alice derived using <m>e_AB</m> · <m>B</m>**, because - remember - scalar multiplication on the curve is commutative.
2. Next, Bob uses the shared secret (<m>ss_AB</m>) and the ephemeral public key (<m>E_AB</m>) to derive the blinding factor: <m>bf_AB</m> = SHA256(<m>E_AB</m> ‖ <m>ss_AB</m>). Bob can do this because he has both variables needed: <m>E_AB</m> is included in the packet, and he just derived <m>ss_AB</m>.
3. He then uses the blinding factor to compute the next ephemeral public key, which he'll hand to Charlie: <m>E_AC</m> = <m>bf_AB</m> · <m>E_AB</m>.

When Charlie receives the onion, the ephemeral key hop payload now contains <m>E_AC</m>. He runs the same routine with his own private key, derives <m>ss_AC</m>, computes <m>bf_AC</m>, advances to <m>E_AD</m>, and forwards to Dave. Dave does the same with his private key to derive <m>ss_AD</m> (he doesn't need to advance the chain, since there's no next hop).

At this point, you may be wondering: "what if Bob gives Charlie a fake ephemeral public key?!" Great question! Technically, there is nothing stop him from doing this, but he is incentivized to play fair. Why? Bob is locking up his capital to route this payment, and he only earns a fee if the payment completes successfully. If he hands Charlie a bogus <m>E_AC</m>, Charlie's ECDH will derive the wrong <m>ss_AC</m>, his decryption will produce garbage, and the payment will fail. Bob would lose his fee and tie up his channel for nothing.
