# Key Derivation

Alice has computed a shared secret with each hop. But a single shared secret isn't enough. She needs to encrypt the payload, authenticate it with an HMAC, generate padding, and handle error messages, all for each hop. Using the same key for all of these would be a security risk.

Instead, Alice **derives** multiple purpose-specific keys from each shared secret. This is a standard cryptographic practice called **domain separation**: even though all the keys come from the same secret, each one is bound to a specific purpose and can't be confused with any other.

## The Five Per-Hop Keys

From each shared secret, Alice derives exactly five keys using HMAC-SHA256:

| Key | Derivation | Purpose |
|-----|-----------|---------|
| **rho** | `HMAC-SHA256(key=b"rho", msg=shared_secret)` | Generates the pseudo-random byte stream for encrypting/decrypting the onion payload (used as a ChaCha20 key) |
| **mu** | `HMAC-SHA256(key=b"mu", msg=shared_secret)` | Computes the HMAC that authenticates the onion packet (outgoing direction) |
| **um** | `HMAC-SHA256(key=b"um", msg=shared_secret)` | Computes the HMAC for error messages (return direction) |
| **pad** | `HMAC-SHA256(key=b"pad", msg=shared_secret)` | Generates filler bytes used to maintain the fixed packet size |
| **ammag** | `HMAC-SHA256(key=b"ammag", msg=shared_secret)` | Encrypts error messages traveling backward to the sender |

<key-tree></key-tree>

Notice something about the names? **"um" is "mu" backwards**, and **"ammag" is "gamma" backwards**. This isn't a coincidence! The `mu` and `rho` keys are used in the forward direction (sender to recipient). The `um` and `ammag` keys are used in the reverse direction (error messages traveling back from a failing hop to the sender). The reversed names are a nice mnemonic for their reversed roles.

<details>
  <summary>Why derive separate keys instead of using the shared secret directly?</summary>

Using the shared secret directly for multiple purposes would be a security risk. If one key is somehow leaked or compromised (say, through a side-channel attack on the encryption), it shouldn't compromise the authentication or error handling. Key derivation with distinct labels ensures **domain separation**: each key is cryptographically independent, even though they all come from the same shared secret. This is a fundamental principle in cryptographic protocol design. Even if an attacker learns the `rho` key, they can't derive `mu` from it without knowing the original shared secret.
</details>
