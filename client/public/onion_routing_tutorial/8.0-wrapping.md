## Wrap, step by step

Now that we've got a solid intuition, let's see how we can translate the visual into an algorithm we can code. NOTE: the steps below are numbered independently of the visual's steps above, so don't try to match them 1-to-1! Before Alice adds any payload data to the onion packet, she'll need to initialize a 1,300-byte buffer with `pad_key`. To do this, she derives a `pad_key` from her <g>session key</g> and uses it as a ChaCha20 key to generate 1,300 bytes of pseudorandom noise.

<python-snippet id="wrap-pad-init"></python-snippet>

<details>
<summary>Why noise instead of zeros?</summary>

If she started with zeros, then the destination's decrypted view would decrypt to a sequence of zeros, which would leak information about how long the route was! By starting with pseudorandom bytes, the destination is unable to decipher how long the route was.

</details>

Now that the buffer is initialized, Alice can iterate over each hop (in reverse order: Dave → Charlie → Bob). Each iteration will follow the same algorithm below.

#### Step 1. Add this hop's data to the front
Each iteration, Alice prepends this hop's data to the front of the buffer: the `payload` (the bigsize-prefixed TLV records), followed by the 32-byte `next_hmac`. To keep the buffer at exactly 1,300 bytes, she drops the same number of bytes (`hop_size = len(payload) + 32`) off the end. In the visual above, this is the "right-shift and write" step.

> Note, `payload` is the bigsize-prefixed TLV records (`bigsize_LEN ‖ TLV_bytes`), so `len(payload)` already includes the LEN prefix. The `+ 32` accounts for the HMAC field that comes after.

Notem `next_hmac` is **not this hop's own HMAC**! It's the HMAC of the layer *inside* this hop, which this hop's payload commits to. Or, from an algorithmic perspective, it's the HMAC from the previous iteration. For example:

- **Dave's iteration** (innermost, runs first): `next_hmac` is 32 zero bytes. Dave is the destination, so there's no inner layer for his hop payload to commit to.
- **Charlie's iteration**: `next_hmac` is `dave_hmac`, computed in Dave's Step 4 (below).
- **Bob's iteration**: `next_hmac` is `charlie_hmac`, computed in Charlie's Step 4 (below).

Each iteration's Step 4 produces *this* hop's HMAC, which then becomes the *next* iteration's `next_hmac`. That's the HMAC chain: each layer commits to the encrypted contents of the one beneath it.

<python-snippet id="wrap-write"></python-snippet>

#### Step 2. XOR the buffer with this hop's `rho` keystream
Now that Alice has added the payload and `next_hmac`, she's ready to perform some cryptography! She XORs the entire 1,300-byte buffer with the keystream generated from this hop's **rho** key (`rho_i`, derived from the shared secret `ss_AX` with the current hop). Remember, one XOR pass both encrypts the bytes Alice just wrote AND adds a fresh encryption layer on top of everything written in inner iterations.

<python-snippet id="wrap-xor"></python-snippet>

#### Step 3. Overlay the filler (innermost iteration only)
If you're wondering where the <g>filler</g> comes in, this is it! Before Alice is ready to calculate the HMAC for this packet, she needs to overwrite the trailing `len(filler)` bytes of the buffer with the filler we built in chapter 7. Note, this only needs to happen in the innermost layer because the filler was precomputed (back in chapter 7) to anticipate every later iteration's `rho` XOR. We place it once here, and Charlie's and Bob's subsequent XOR passes transform it into the bytes each forwarder will see at the trailing positions when they peel.

<python-snippet id="wrap-filler-overlay"></python-snippet>

#### Step 4. Compute this hop's HMAC
Finally, Alice computes the HMAC over the buffer concatenated with `associated_data` (the 32-byte `payment_hash`) using this hop's `mu` key. Save it as `next_hmac` for the iteration that builds the layer above.

Why do we include `associated_data` rather than hash the buffer alone? According to [BOLT 4](https://github.com/lightning/bolts/blob/master/04-onion-routing.md#packet-construction), the onion "commits to" this extra data, binding it to this specific payment. Why does that matter? If an attacker captured an in-flight onion and tried to re-attach it to a different HTLC (one carrying a different `payment_hash`), then the HMAC verification would fail.

<python-snippet id="wrap-hmac"></python-snippet>

After the outermost iteration finishes, the buffer goes in the packet's `hop_payloads` field, and the HMAC we just computed is the packet's outer `hmac` tag. Alice prepends the 1-byte version (`0x00`) and her 33-byte ephemeral pubkey `E_AB`, appends the 32-byte `bob_hmac`, and ships the 1,366-byte packet to Bob.

## Wrap one hop, then loop

Let's get to coding! For this exercise, we'll complete the following two functions.
- First, a per-hop function `wrap_hop` that performs the steps we just reviewed for a single iteration.
- Second, a top-level `build` function that wraps the destination first (handing `wrap_hop` the filler), then each forwarder outward, threading the `next_hmac` along the way, and finally assembling the 1,366-byte packet.

Note that the first exercise (`wrap_hop`) handles a single iteration of the loop, including the filler overlay (step 3) when it's handed a filler. For the second exercise (`build`), the setup code is provided for you. It re-uses the functions you already wrote and were tested on (`derive_keys`, `generate_filler`) to derive the per-hop keys, build the filler, and seed the pad-noise buffer. Your job in `build` is the part that's new this chapter: wrapping the destination first (the only hop that gets the filler), then the forwarders outward (threading `next_hmac`), and assembling the 1,366-byte packet.
