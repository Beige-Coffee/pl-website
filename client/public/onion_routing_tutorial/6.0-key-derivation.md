# Key Derivation

At the end of chapter 5 we made a small confession. We'd been pretending all the onion magic happens with just one shared secret, but in reality <g>Sphinx</g> uses **named keys** with separate purposes. Four are derived from each hop's shared secret, and one (`pad`) is derived from Alice's <g>session key</g> to initialize the packet buffer.

Before we learn how to derive each key, let's first review the cryptographic operations that the onion construction actually performs as a packet flows across the route. This will help us contextualize the purpose of each key.

## Cryptographic operations across the roundtrip

As the packet flows through the route, it has to support a handful of distinct cryptographic operations. Many of the below examples should look familiar from the last chapter:

- **Encrypt the forward payload** with a stream cipher (in this case, ChaCha20) so that only the intended hop can read the payment instructions.
- **Authenticate the forward layer** with a message authentication code (MAC) (HMAC-SHA256) so the forwarder can detect tampering *before* processing. Without it, an attacker could flip bytes mid-flight and force the forwarder to act on wrong information.
- **Encrypt return-path errors** with a stream cipher so only Alice can read the failure reason. Without it, intermediate hops on the return path would learn what went wrong and where, which would be a serious privacy leak!
- **Authenticate return-path errors** with a MAC so Alice can verify the error actually came from a specific hop on her route.
- **Initialize the empty buffer** with a stream cipher so unused space at the end of the packet doesn't leak path length.

Now, here's the kicker... **each one of these operations needs its own dedicated key.** If we reused one key across multiple operations, we'd break the security guarantees these cryptographic primitives rely on. Below are a few specific reasons:

- **Encryption and authentication are different jobs.** ChaCha20 (the stream cipher) and HMAC-SHA256 (the MAC) each come with their own security guarantees, but those guarantees only hold when each key is used for *one* job. If we used the same key for both, those guarantees would no longer apply.

- **Forward and backward directions can't share a key.** [BOLT 4](https://github.com/lightning/bolts/blob/master/04-onion-routing.md#pseudo-random-byte-stream) uses ChaCha20 with the same nonce (zero) everywhere, including for both forward and backward encryption. This is safe only because every operation gets its own key: the spec notes that *"the use of a fixed nonce is safe, since the keys are never reused."* As we learned in the Noise Protocol course, reusing the same nonce with the same key is a very bad idea. If forward and backward encryption shared a key, an attacker could XOR the forward and backward payload ciphertexts together and potentially recover the original plaintext.

So, using *just* one raw secret won't be sufficient for our purposes. Instead, we'll use a key-generation function, defined in BOLT 4, with different labels to derive distinct 32-byte keys. ## Deriving the five keys

Now that we've seen each cryptographic operation and learned why each needs its own key, let's review the specification, defined in [BOLT 4's Key Generation section](https://github.com/lightning/bolts/blob/master/04-onion-routing.md#key-generation), for how to derive them. The BOLT writes the formula as:

<spec-formula source="BOLT 4">generate_key(name, secret) = HMAC-SHA256(key=name, msg=secret)</spec-formula>

You may be wondering why mixing the secret with a public ASCII label produces something safe to use as another key... Well, due to the properties of <g>HMAC</g>, as long as the shared secret stays a secret and is pseudorandom, then the output of this function will be another 32-byte pseudorandom value. The beauty of this construction is that, by using standardized names, the derivation becomes deterministic and both Alice and each subsequent hop can derive the exact same keys from the same shared secret. In other words, Alice and Bob (who have the same shared secret), can independently derive new secrets, using just ASCII labels.

In the formula, `name` is a short ASCII label used to derive unique keys for each of the purposes we reviewed above. Four labels (`rho`, `mu`, `um`, and `ammag`) are used with each hop's shared secret <m>ss_i</m>, making them unique to that hop. The fifth label (`pad`) is used with Alice's session key, making it unique to this specific payment and useful for initializing the empty packet buffer.

<kdf-pipeline></kdf-pipeline>

Below is a description of each key and what it's used for:

- **`rho`**: the **forward encryption** key. A ChaCha20 stream-cipher key. When Alice builds the packet, she XORs each hop payload with `rho`'s keystream to encrypt it; the forwarder later derives the *same* `rho` and XORs again to strip that layer off as it peels. (One key both ways, because XOR-ing twice with the same keystream cancels out.)
- **`mu`**: the **forward authentication** key. An HMAC key. Alice computes an integrity tag over each hop's payload with it; the forwarder recomputes that tag and checks it *first*, before decrypting anything, to confirm no one tampered with the bytes.
- **`um`**: the **return authentication** key. An HMAC key. A hop that fails signs its error with `um`; Alice later recomputes the tag with the same key to confirm which hop the error really came from.
- **`ammag`**: the **return encryption** key. A ChaCha20 stream-cipher key. The failing hop (and each hop on the way back) XORs the error with `ammag`'s keystream to wrap it; Alice peels those layers off with the matching keys, so only she can read the failure.
- **`pad`**: the **buffer-setup** key. Derived from Alice's session key (not a per-hop secret). Alice alone uses it, once, to pre-fill the empty payload area with random-looking bytes before adding any layers.

Note: each hop has its own `rho`, `mu`, `um`, and `ammag` key. This should make intuitive sense, since each is derived from that hop's shared secret with Alice. The `pad` key is different: Alice derives it from the session key once while building the packet.

> **Fun Fact!** The names come from cryptographic convention plus a few wordplay-y choices. `rho` and `mu` match the Sphinx paper's stream-cipher (`h_ρ`) and MAC (`h_μ`) hash oracles. `ammag` is `gamma` reversed, the return-direction counterpart of an encryption key. `um` is `mu` reversed, the return-direction MAC key. `pad` does what it says, so that one is kinda boring.

<summary>What about attribution data: isn't there a sixth key now?</summary>

In November 2025, BOLT 4 added an optional **attribution data** feature for return-path errors, along with a fifth per-hop key called `ammagext` (derived the same way: ammagext_i = HMAC-SHA256("ammagext", <m>ss_i</m>)). Attribution data lets Alice identify *which* hop in the route failed even when intermediate hops misbehave.

The construction is identical to the other four per-hop keys, so once you have `rho`, `mu`, `um`, and `ammag` down, adding `ammagext` is a one-line extension. We're scoping this chapter to the original four because attribution data is still rolling out across implementations.

</details>
