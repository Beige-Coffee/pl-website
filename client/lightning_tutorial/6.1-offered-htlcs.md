# Offered HTLCs

## What Are HTLCs?

**HTLCs (Hashed Time-Locked Contracts)** are the mechanism that transforms payment channels from simple two-party constructs into a global payment network. An HTLC is a conditional payment: "I will pay you X satoshis if you can provide the preimage of this hash within Y blocks."

HTLCs enable **multi-hop payments**. Alice can pay Carol through Bob by setting up a chain of HTLCs: Alice offers an HTLC to Bob (conditional on a hash), and Bob offers an HTLC to Carol (conditional on the same hash). When Carol reveals the preimage, both HTLCs settle atomically.

## Offered vs Received

From the perspective of a commitment transaction holder:

- An **offered HTLC** is one where the holder is the **payer** -- they are offering to send funds if the preimage is revealed. The HTLC amount is going out.
- A **received HTLC** is one where the holder is the **payee** -- they will receive funds when they reveal the preimage. The HTLC amount is coming in.

The scripts are different because the timeout and claim paths are reversed. In this chapter, we focus on offered HTLCs.

## The Offered HTLC Script

BOLT 3 specifies the offered HTLC witness script as:

```text
# To remote node with revocation key
OP_DUP OP_HASH160 <RIPEMD160(SHA256(revocationpubkey))> OP_EQUAL
OP_IF
    OP_CHECKSIG
OP_ELSE
    <remote_htlcpubkey>
    OP_SWAP OP_SIZE 32 OP_EQUAL
    OP_NOTIF
        # Below 32 bytes: timeout path (no preimage)
        OP_DROP 2 OP_SWAP <local_htlcpubkey> 2 OP_CHECKMULTISIG
    OP_ELSE
        # Exactly 32 bytes: claim path (has preimage)
        OP_HASH160 <RIPEMD160(payment_hash)> OP_EQUALVERIFY
        2 OP_SWAP <local_htlcpubkey> 2 OP_CHECKMULTISIG
    OP_ENDIF
OP_ENDIF
```

This script has three spending paths:

### 1. Revocation Path

If the commitment transaction is a revoked state, the counterparty can sweep the HTLC output immediately using the revocation key. The witness is:

```text
<revocation_sig> <revocationpubkey>
```

The script checks: is this the revocation key? If the hash of the provided pubkey matches the revocation hash, verify the signature.

### 2. Preimage Path (Remote Claims via HTLC-Success)

If the remote party knows the preimage, they can claim the HTLC. However, they cannot claim directly -- they must go through a **second-stage transaction** (HTLC-success). The witness for the commitment output is:

```text
0 <remote_htlc_sig> <local_htlc_sig> <payment_preimage>
```

The script checks: is the provided value exactly 32 bytes? If so, verify it hashes to the expected payment hash, then check the 2-of-2 multisig.

### 3. Timeout Path (Local Reclaims via HTLC-Timeout)

If the HTLC expires without the preimage being revealed, the local party reclaims their funds. This also uses a second-stage transaction (HTLC-timeout). The witness for the commitment output is:

```text
0 <remote_htlc_sig> <local_htlc_sig> <>
```

The empty byte string (less than 32 bytes) triggers the timeout branch. The script checks the 2-of-2 multisig.

## Why Second-Stage Transactions?

You might wonder why the preimage and timeout paths require second-stage transactions instead of allowing direct spending. The reason is the **revocation delay**.

When a commitment transaction is broadcast, the counterparty needs time to check if it is a revoked state. If the HTLC output could be spent immediately (without going through a second-stage transaction), the broadcaster could claim the HTLC funds before the counterparty could react with a revocation.

The second-stage transactions (HTLC-timeout and HTLC-success) have their output encumbered with the same revocation-and-delay structure as the `to_local` output. This ensures the revocation mechanism works consistently across all outputs.

## The HTLC-Timeout Transaction

The **HTLC-timeout** transaction is a second-stage transaction that the local party uses to reclaim funds from an expired offered HTLC. It has the following structure:

| Field | Value |
|-------|-------|
| Version | `2` |
| Input | The offered HTLC output from the commitment transaction |
| Input sequence | `0x00000000` |
| Output | A single output with the revocation-and-delay script |
| Locktime | The HTLC's `cltv_expiry` |

The **locktime** is set to the HTLC's CLTV expiry. This means the HTLC-timeout transaction cannot be broadcast until the HTLC has actually expired. The `OP_CHECKLOCKTIMEVERIFY` in the overall scheme ensures the timeout cannot be triggered prematurely.

### HTLC-Timeout Output Script

The output of the HTLC-timeout transaction uses the same revocable delay structure as `to_local`:

```text
OP_IF
    <revocationpubkey>
OP_ELSE
    <to_self_delay>
    OP_CHECKSEQUENCEVERIFY
    OP_DROP
    <local_delayedpubkey>
OP_ENDIF
OP_CHECKSIG
```

This output gives the counterparty a window to use the revocation key (if this was a revoked commitment) before the local party can claim the funds after the CSV delay.

### HTLC-Timeout Witness

The witness for the HTLC-timeout transaction input (spending the offered HTLC output) is:

```text
0 <remotehtlcsig> <localhtlcsig> <> <offered_htlc_script>
```

The empty byte string `<>` selects the timeout branch (less than 32 bytes). Both HTLC signatures are required (2-of-2 multisig). The `0` at the start is the NULLDUMMY for `OP_CHECKMULTISIG`.

### Building the HTLC-Timeout Transaction

```python
def build_htlc_timeout_tx(
    commitment_txid: bytes,
    htlc_output_index: int,
    amount_sat: int,
    cltv_expiry: int,
    revocation_pubkey: bytes,
    to_self_delay: int,
    local_delayed_pubkey: bytes,
    fee_sat: int
):
    """Build an HTLC-timeout transaction."""
    # Input: the offered HTLC output
    input_txid = commitment_txid
    input_vout = htlc_output_index

    # Output: revocable delay script (same as to_local)
    output_script = to_local_script(revocation_pubkey, to_self_delay, local_delayed_pubkey)
    output_amount = amount_sat - fee_sat

    return {
        'version': 2,
        'inputs': [(input_txid, input_vout, 0x00000000)],  # sequence 0
        'outputs': [(output_amount, script_to_p2wsh(output_script))],
        'locktime': cltv_expiry,  # Cannot broadcast before HTLC expiry
    }
```

## Summary

The offered HTLC flow:

1. The local party offers an HTLC (conditional payment going out).
2. The HTLC output is included in the commitment transaction with the offered HTLC script.
3. If the remote party provides the preimage, they claim via an HTLC-success transaction.
4. If the HTLC expires, the local party reclaims via an HTLC-timeout transaction (with locktime = cltv_expiry).
5. If the commitment was revoked, the counterparty sweeps using the revocation key.
