# A First (Broken) Attempt

We've built a solid cryptographic foundation. Our `SphinxPacketBuilder` can compute shared secrets with every hop, derive per-hop keys, and generate the full chain of ephemeral public keys. Now it's time to actually use these keys to build the **onion packet**.

Let's start with the obvious approach and see where it breaks.

## The Naive Construction

Suppose Alice wants to send instructions to Bob, Carol, and Dave. She has each hop's TLV-encoded payload (the forwarding instructions we built in Section 2) and a unique encryption key for each hop (the `rho` key from Section 3). The simplest construction would be:

1. Take each hop's plaintext payload
2. Encrypt each payload with that hop's `rho` key
3. Concatenate all the encrypted payloads together
4. Send the result to Bob

<naive-packet></naive-packet>

Bob decrypts his portion, reads his instructions, strips his payload from the front, and forwards the rest to Carol. Carol does the same for Dave. Simple, right?

This construction has three fatal problems.

## Problem 1: The Packet Shrinks

When Bob strips his payload and forwards the remainder, the packet gets smaller. Carol receives a packet that's shorter than what Bob received. Dave receives an even shorter one.

Any intermediate node can measure the packet size and estimate how many hops remain. If Bob sees a 300-byte packet and knows each hop's payload is about 65 bytes, he can guess there are roughly 4-5 hops in total. Combined with knowing he's the first hop (Alice sent it to him directly), Bob can estimate the route length and, depending on the network topology, potentially narrow down the destination.

**Fixed-size packets** are essential for privacy. Every hop must see a packet that looks identical in size, regardless of its position in the route.

<naive-vs-fixed></naive-vs-fixed>

## Problem 2: No Layered Encryption

In the naive approach, each hop's payload is encrypted independently. The payloads sit side by side in the packet. If Bob can somehow break or bypass his decryption (or if there's an implementation bug), he can see the structure of the remaining encrypted payloads. Even without decrypting them, he learns how many remain.

Worse, the payloads aren't **nested**. In true onion routing, each layer of encryption wraps ALL the inner layers. Bob's layer covers everything inside it. When Bob decrypts his layer, he reveals only Carol's encrypted layer, not the raw structure underneath.

## Problem 3: No Authentication

Nothing in this naive construction prevents a malicious node from modifying the packet. Bob could flip bits in Carol's encrypted payload, and Carol would have no way to detect the tampering. She'd decrypt garbage and either crash or forward corrupted data.

We need an **HMAC** (Hash-based Message Authentication Code) at each layer to authenticate the packet. If any byte has been tampered with, the HMAC check fails and the packet is rejected immediately.

## The Fixed-Size Packet Structure

To solve all three problems, we need the **Sphinx** packet construction. Before we dive into the algorithm, let's look at the packet format we're building toward.

An onion packet is always exactly **1,366 bytes**:

| Field | Size | Description |
|-------|------|-------------|
| **Version** | 1 byte | Protocol version (currently `0x00`) |
| **Ephemeral Public Key** | 33 bytes | Session public key for this hop's ECDH |
| **Encrypted Payload** | 1,300 bytes | All hop payloads, layered and encrypted |
| **HMAC** | 32 bytes | Authentication tag for the current hop |

The 1,300-byte encrypted payload contains the TLV-encoded instructions for every hop, encrypted layer by layer. When Bob peels his layer, the payload stays exactly 1,300 bytes (he decrypts, extracts his portion, and pads the rest). The HMAC lets each hop verify the packet hasn't been tampered with before processing it.

This fixed structure means Bob sees a 1,366-byte packet, peels one layer, and forwards a 1,366-byte packet to Carol. Carol sees the same size, peels, forwards. Dave sees the same size. Nobody can determine their position from the packet size alone.

<details>
  <summary>Why 1,300 bytes for the payload?</summary>

The 1,300-byte payload size comes from a practical tradeoff. Each hop's TLV payload is typically 40-65 bytes (depending on whether it's an intermediate or final hop). With 1,300 bytes of space and roughly 65 bytes per hop (including the 32-byte per-hop HMAC), the protocol supports up to **20 hops**. This is well above the practical limit for Lightning payments (most real payments use 3-6 hops). The extra space provides room for future extensions (like route blinding metadata) without changing the packet format.
</details>

## Building the Hop Payloads

Before we can wrap the onion, we need to prepare the actual payloads that go inside it. Each hop's payload is a TLV-encoded set of instructions (the same format we implemented in Exercise 2), but now we need to build them as a method on our `SphinxPacketBuilder`.

For each hop, the payload contains:
- **Type 2**: `amt_to_forward` (how much to forward, in millisatoshis)
- **Type 4**: `outgoing_cltv_value` (the CLTV expiry for the outgoing HTLC)
- **Type 6**: `short_channel_id` (which channel to forward on, only for intermediate hops)
