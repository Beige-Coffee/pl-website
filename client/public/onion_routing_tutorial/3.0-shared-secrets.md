# Shared Secrets

Alice has her route (Alice -> Bob -> Carol -> Dave) and she's calculated the per-hop forwarding amounts and timelocks. Now she needs to build the **onion packet**, the encrypted, layered structure that delivers each hop's instructions without revealing anything about the rest of the route.

The foundation of this construction is a cryptographic primitive called **Elliptic Curve Diffie-Hellman (ECDH)**. If you've completed the Noise Protocol tutorial, you'll recognize this immediately. If not, here's the key idea.

## ECDH in 30 Seconds

Every Lightning node has a **keypair**: a private key (a large random number) and a corresponding public key (a point on the secp256k1 elliptic curve). If Alice's private key is `a`, her public key is `A = a * G`, where `G` is the curve's generator point.

The magic of ECDH is this: if Alice knows Bob's public key `B` and Bob knows Alice's public key `A`, they can each independently compute the **same shared secret** without ever exchanging it directly.

- Alice computes: `shared_point = a * B`
- Bob computes: `shared_point = b * A`

These produce the same point because `a * B = a * (b * G) = b * (a * G) = b * A`. Nobody who only knows the public keys `A` and `B` can compute this shared point. This is what makes ECDH secure.

To turn the shared point into a usable 32-byte secret, we hash the x-coordinate: `shared_secret = SHA256(shared_point.x)`.

## Why Alice Uses a Session Key

Here's a problem. If Alice performs ECDH using her real Lightning node public key, every hop in the route would see that same public key. Bob, Carol, and Dave would all learn Alice's identity, destroying sender privacy.

The solution: Alice generates a fresh **session key** (also called an **ephemeral key**) for each payment. This is a brand new keypair that exists only for the duration of this one onion packet construction. The session public key is included in the onion packet header so each hop can compute the shared secret, but it reveals nothing about Alice's identity.

From the perspective of Bob, Carol, or Dave, the session public key is just a random curve point. It could belong to anyone. There's no way to link it back to Alice's node.

<details>
  <summary>What if Alice reuses the same session key for multiple payments?</summary>

Reusing a session key would allow correlation. If Bob sees the same ephemeral public key in two different onion packets, he knows both payments came from the same sender. Worse, if any hop stores ephemeral keys it has seen, it could link payments across time. Fresh session keys per payment prevent this entirely. Each payment looks completely independent, even to the same forwarding node.
</details>
