# Inspecting the HTLC Commitment Transaction

Alright, if you've put in the Proof-of-Work and completed all of the exercises up to this point, then you'll be able to generate your very own commitment transaction with one "outgoing" HTLC. It's called "outgoing" because it represents a payment from us (Alice) to Bob, so, if successful, bitcoin will be deducted from our channel balance and added to Bob's.

## Generate an HTLC Commitment Transaction

Click the button below to generate a signed commitment transaction with an offered HTLC output. You'll need to paste your **Funding Transaction ID**, which you can find in the **Transactions** notebook (open it via **TOOLS** in the top-right corner).

<tx-generator id="gen-htlc-commitment"></tx-generator>

Under the hood, this generator is running all of the functions we created earlier! It creates two sets of keys (one for us, Alice, and one for Bob), fetches the funding UTXO using the Funding TxID, and then calls:

- `create_commitment_tx`: This creates an unsigned commitment transaction. **Crucially, it's now updated to add any relevant HTLCs too!**
- `create_funding_script`: This creates the 2-of-2 multisig script, which is needed to generate a signature and pass into the witness.
- `finalize_commitment_tx`: This generates our (Alice's) signature and adds it to the witness (along with Bob's signature and the witness script), resulting in a fully signed transaction ready to be broadcast!

## Three Outputs

The most obvious difference from the basic commitment transaction is that we now have **three outputs** instead of two. Previously, we only had `to_local` and `to_remote`. Now, we also have an **HTLC output** representing the in-flight payment.

Let's identify each one. Once again, we can see the `to_local` and `to_remote` outputs! Can you tell which is which?

Even without looking at the amount, we should be able to identify the `to_remote` output pretty easily. It's the only output that is locked to a `"type": "witness_v0_keyhash"` (Pay-To-Witness-Public-Key-Hash).

Since the `to_local` and HTLC outputs are both Pay-To-Witness-Script-Hash, we will have to tell them apart by their amount (or recreate the script and hash it).

### Output 1: to_remote (P2WPKH)

```
{
  "value": 0.00000500,
  "n": 0,
  "scriptPubKey": {
    "asm": "0 cc1b07838e387deacd0e5232e1e8b49f4c29e484",
    "type": "witness_v0_keyhash"
  }
}
```

This is the **to_remote** output, representing our counterparty's balance. We can spot this immediately because it's the only output with a type of `"witness_v0_keyhash"` (P2WPKH). As we learned earlier, the `to_remote` output locks funds directly to the counterparty's public key hash. No timelocks, no complex scripts. They can spend it right away if the commitment transaction is broadcast.

### Output 2: HTLC Output (P2WSH)

```
{
  "value": 0.00405000,
  "n": 1,
  "scriptPubKey": {
    "asm": "0 4adb4e2f00643db396dd120d4e7dc17625f5f2c11a40d857accc862d6b7dd80e",
    "type": "witness_v0_scripthash"
  }
}
```

This is the **HTLC output**. It's a P2WSH output, just like the `to_local`, so we can't distinguish them by type alone. However, we can identify it by its value: 405,000 satoshis (0.00405000 BTC), which is the amount locked in the HTLC.

Under the hood, this output is locked to the offered (or received) HTLC script we built earlier, which contains the revocation path, the preimage path, and the timeout/success path. The hash in the `asm` field is the SHA256 of that full HTLC witness script.

As you can see, our outputs are sorted properly, so our HTLC is right there in the middle (index 1)!

### Output 3: to_local (P2WSH)

```
{
  "value": 0.04593500,
  "n": 2,
  "scriptPubKey": {
    "asm": "0 4adb4e2f00643db396dd120d4e7dc17625f5f2c11a40d857accc862d6b7dd80e",
    "type": "witness_v0_scripthash"
  }
}
```

This is the **to_local** output, representing our own balance (minus the HTLC amount and fees). Like the HTLC output, it's a P2WSH, so we tell them apart primarily by their values. The `to_local` holds the larger balance, while the HTLC output holds the specific payment amount.

## Output Ordering

As we learned when implementing `create_commitment_tx`, BOLT 3 specifies a deterministic output ordering:

1. Outputs are sorted by value, smallest first.
2. If two outputs have the same value, they are sorted by the lexicographic order of their `scriptPubKey`.
3. For HTLC outputs with the same value and script, they are further sorted by increasing `cltv_expiry`.

In our decoded transaction, you can see the outputs are sorted properly: the `to_remote` (0.00000500 BTC) comes first, followed by the HTLC (0.00405000 BTC), and finally the `to_local` (0.04593500 BTC). The values increase from smallest to largest, exactly as the BOLT 3 sorting rules specify.

The important takeaway is that output ordering is deterministic. Both Alice and Bob will independently produce the same ordering, which means they will compute the same transaction ID and can exchange valid signatures. If either party used a different ordering, the signatures would be invalid and the commitment transaction could not be broadcast.

## Identifying Output Types

Since both `to_local` and HTLC outputs are P2WSH, it's worth reviewing the strategies for telling them apart:

1. **By amount**: If you know the HTLC amount and channel balances, you can simply match by value. This is the easiest method when inspecting a transaction you built yourself.
2. **By type field**: The `to_remote` output is always `"witness_v0_keyhash"` (P2WPKH), while `to_local` and HTLC outputs are `"witness_v0_scripthash"` (P2WSH). So you can always identify `to_remote` by its type.
3. **By recreating the script**: If you have the public keys and payment hash, you can rebuild the expected HTLC script, hash it with SHA256, and compare the result against the output's script hash. This is the most reliable method and is what Lightning implementations use in practice.

As more HTLCs are added to a commitment transaction (for example, if Alice is routing multiple payments through Bob), you'll see additional P2WSH outputs. Each one represents a separate HTLC, and they will all be sorted among the other outputs according to the BOLT 3 ordering rules.
