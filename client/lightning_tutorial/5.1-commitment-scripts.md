# Commitment Scripts

## Overview

A commitment transaction has two main outputs that distribute the channel's funds between the two parties:

1. **to_local** -- The broadcaster's own balance, encumbered with a time delay and a revocation path.
2. **to_remote** -- The counterparty's balance, immediately spendable.

These outputs use different scripts depending on whose perspective we are looking at. Remember that commitment transactions are **asymmetric**: Alice's version and Bob's version have different scripts, even for the same balance split.

## The to_remote Output

The `to_remote` output is the simpler of the two. With `option_static_remotekey` (which is now mandatory in the specification), the to_remote output is a standard **P2WPKH** (Pay-to-Witness-Public-Key-Hash) output that pays directly to the counterparty's `payment_basepoint`.

### Script

The output script is a standard P2WPKH:

```text
OP_0 <20-byte-hash-of-payment-basepoint>
```

In bytes:

```python
import hashlib

def to_remote_script(remote_payment_pubkey: bytes) -> bytes:
    """Create the to_remote output script (P2WPKH).

    Uses the remote party's payment_basepoint directly (static remotekey).
    """
    # HASH160 = RIPEMD160(SHA256(pubkey))
    sha = hashlib.sha256(remote_payment_pubkey).digest()
    h160 = hashlib.new('ripemd160', sha).digest()

    # P2WPKH: OP_0 <20-byte-hash>
    return bytes([0x00, 0x14]) + h160
```

### Why Static Remotekey?

Before `option_static_remotekey`, the to_remote output used a per-commitment derived key. This created a serious recovery problem: if the remote party lost their channel state database, they would not know which per-commitment point was used and could not spend their output.

With static remotekey, the remote party can always recover their funds using just their `payment_basepoint_secret` -- no channel state required. This is a major reliability improvement that was later made mandatory in the specification.

## The to_local Output

The `to_local` output is more complex. It is a **P2WSH** (Pay-to-Witness-Script-Hash) output containing a script with two spending paths:

1. **Revocation path**: The counterparty can spend immediately using the revocation key (penalty for cheating).
2. **Delayed path**: The broadcaster can spend after a CSV delay using their delayed key (normal spending).

### The Witness Script

BOLT 3 specifies the to_local witness script as:

```text
OP_IF
    <revocationpubkey>
OP_ELSE
    <to_self_delay>
    OP_CHECKSEQUENCEVERIFY
    OP_DROP
    <local_delayedpubkey>
OP_ENDIF
OP_CHECKSIG
```

Let us trace through both spending paths:

### Revocation Path (counterparty detects cheating)

The witness for spending via revocation is: `<revocation_sig> 1`

1. `1` is pushed onto the stack (selecting the `OP_IF` branch).
2. `OP_IF` pops the `1` and enters the true branch.
3. `<revocationpubkey>` is pushed onto the stack.
4. `OP_ENDIF` closes the conditional.
5. `OP_CHECKSIG` verifies `<revocation_sig>` against `<revocationpubkey>`.

If the signature is valid, the spend succeeds. The revocation path has **no time delay** -- the counterparty can sweep immediately once they detect a cheating attempt.

### Delayed Path (broadcaster spends their own funds)

The witness for spending via the delayed path is: `<local_delayed_sig> 0`

1. `0` is pushed onto the stack (selecting the `OP_ELSE` branch).
2. `OP_IF` pops the `0` and enters the false (else) branch.
3. `<to_self_delay>` is pushed (e.g., 144 blocks, roughly 1 day).
4. `OP_CHECKSEQUENCEVERIFY` verifies that at least `to_self_delay` blocks have passed since the transaction was confirmed.
5. `OP_DROP` removes the delay value from the stack.
6. `<local_delayedpubkey>` is pushed onto the stack.
7. `OP_ENDIF` closes the conditional.
8. `OP_CHECKSIG` verifies `<local_delayed_sig>` against `<local_delayedpubkey>`.

The `OP_CHECKSEQUENCEVERIFY` (CSV) delay is the critical enforcement mechanism. It gives the counterparty time to detect the cheating and use the revocation path before the broadcaster can claim their funds.

### Building the Script in Python

```python
def to_local_script(revocation_pubkey: bytes, to_self_delay: int, local_delayed_pubkey: bytes) -> bytes:
    """Build the to_local witness script.

    OP_IF
        <revocationpubkey>
    OP_ELSE
        <to_self_delay>
        OP_CHECKSEQUENCEVERIFY
        OP_DROP
        <local_delayedpubkey>
    OP_ENDIF
    OP_CHECKSIG
    """
    script = bytes([0x63])  # OP_IF
    script += bytes([0x21]) + revocation_pubkey  # Push 33 bytes + key
    script += bytes([0x67])  # OP_ELSE

    # Encode to_self_delay as minimal script number
    delay_bytes = to_self_delay.to_bytes(
        (to_self_delay.bit_length() + 7) // 8, 'little'
    )
    # Add high byte if needed to prevent negative interpretation
    if delay_bytes[-1] & 0x80:
        delay_bytes += b'\x00'
    script += bytes([len(delay_bytes)]) + delay_bytes

    script += bytes([0xb2])  # OP_CHECKSEQUENCEVERIFY
    script += bytes([0x75])  # OP_DROP
    script += bytes([0x21]) + local_delayed_pubkey  # Push 33 bytes + key
    script += bytes([0x68])  # OP_ENDIF
    script += bytes([0xac])  # OP_CHECKSIG

    return script
```

## Wrapping in P2WSH

The to_local witness script is not placed directly in the transaction output. Instead, it is hashed and wrapped in a P2WSH output:

```python
def script_to_p2wsh(witness_script: bytes) -> bytes:
    """Wrap a witness script in a P2WSH output script."""
    script_hash = hashlib.sha256(witness_script).digest()
    return bytes([0x00, 0x20]) + script_hash
```

The full witness script is only revealed when the output is spent. Until then, an on-chain observer sees only the 32-byte hash, providing some privacy about the channel's structure.

## Key Derivation Recap

The keys used in these scripts are:

| Script Key | Derivation |
|-----------|-----------|
| `revocationpubkey` | Two-component: `revocation_basepoint` + `per_commitment_point` |
| `local_delayedpubkey` | `derive_pubkey(delayed_payment_basepoint, per_commitment_point)` |
| `remote_payment_pubkey` | `payment_basepoint` directly (static remotekey) |

Remember: "local" and "remote" are always relative to the holder of the commitment transaction.
