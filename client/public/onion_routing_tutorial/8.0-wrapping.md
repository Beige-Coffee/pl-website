# Wrapping Layer by Layer

We have everything we need to actually build the onion. Shared secrets per hop (chapter 4), five named keys per hop (chapter 5), the precomputed filler (chapter 6), and a clear picture of the 1,366-byte packet shape (chapter 6). This chapter brings them all together. By the end, you'll have written the function that produces a complete BOLT 4 onion packet from a route plus a list of hop payloads.

Heads up: the construction has one big counterintuitive move. We're going to build the onion **in reverse**, starting with the destination's hop payload and working outward toward Bob's. The reason has to do with how each per-hop HMAC commits to the layer beneath it. Let's see why.

## A first attempt: build forward

Let's start with what *seems* like the natural ordering. Alice builds Bob's layer first, then Charlie's, then Dave's. From Alice's vantage point, she wraps Bob's hop payload, encrypts it, computes Bob's HMAC, and moves on to wrap Charlie next. Then Dave.

The trouble is: **what does Bob's HMAC commit to?**

When Bob receives the onion, he verifies the HMAC against the `hop_payloads` he sees on the wire. If Bob's HMAC is supposed to authenticate the *entire* packet contents (including the layers underneath his), then Alice has to know what Charlie's and Dave's encrypted layers look like *before* she can compute Bob's HMAC. Otherwise the HMAC won't match what Bob ultimately receives.

So building forward doesn't work. Bob's HMAC depends on the inner layers. Charlie's HMAC depends on Dave's layer. The dependency runs from the outside in, which means construction has to run **from the inside out**.

## The real algorithm: build in reverse

Alice starts with the destination (Dave's hop payload), wraps it inside an outer layer (Charlie's hop payload), and then wraps that inside the outermost layer (Bob's hop payload). Each step encrypts the buffer with the current hop's `rho` key and computes that hop's HMAC over the result, threading the HMAC value into the *next* iteration as the value the outer layer will commit to.

Conceptually, each iteration does this dance:

1. **Shift the buffer right** by `(this hop's hop payload size)` bytes. This pushes the existing contents back to make room at the front.
2. **Write this hop's hop payload at the front**: bigsize-prefixed TLV payload, followed by the 32-byte HMAC pointing to the next layer (which we computed in the *previous* iteration; for Dave, it's all zeros since there's no next layer).
3. **XOR the entire 1,300-byte buffer with this hop's `rho` keystream**. This is the encryption step.
4. **(Innermost iteration only)** Overwrite the trailing `len(filler)` bytes of the buffer with the filler we computed in chapter 6. This places the filler exactly where each forwarder will expect to see it.
5. **Compute this hop's HMAC** over the buffer concatenated with `associated_data` (the 32-byte payment_hash) using this hop's `mu` key. Save it as the `next_hmac` value for the iteration that builds the layer above.

After the last iteration (the outermost hop, Bob), the buffer is what goes in the packet's `hop_payloads` field, and the HMAC we just computed is what goes in the packet's `hmac` field. Alice assembles the final packet as:

```
version (1 byte) || E_AB (33 bytes) || hop_payloads (1300 bytes) || hmac (32 bytes)
```

That's the 1,366 bytes she ships to Bob.

## Two BOLT 4 details that matter

Two specifics from the BOLT 4 spec that are easy to miss in simpler descriptions:

### `associated_data`

The HMAC at every layer is computed over `hop_payloads || associated_data`, where `associated_data` is a 32-byte value (the **payment_hash** for an HTLC) that gets concatenated with the buffer. It binds the onion to a specific payment.

If a forwarder sees an onion attached to a different payment_hash than the one Alice baked in, the HMAC won't verify and the packet is rejected. Without this, an attacker who captured an in-flight onion could attach it to a different HTLC and try to fool forwarders into routing it.

### Pad-key noise (not zeros)

The buffer is initialized with **pad-key noise** instead of all zeros. Specifically, Alice computes:

```
pad_key = HMAC-SHA256(b"pad", session_key)
buffer = chacha20_keystream(pad_key, 1300)
```

This gives her a 1,300 bytes of pseudorandom noise to start with. If she started with all zeros, the destination's decrypted view would have detectable trailing structure (the parts of the buffer no hop ever wrote to would XOR back to zero against the pad keystream). Starting with pseudorandom bytes makes the unused tail look like random noise to the destination, hiding "this packet was built for a short route."

## Walking through a 3-hop build

Let's trace this for our route Alice → Bob → Charlie → Dave with hypothetical hop payload sizes of 65, 65, and 100 bytes (so the destination's hop payload is bigger because it carries `payment_data`). The buffer is 1,300 bytes throughout.

**Initial state.** The buffer is filled with 1,300 bytes of pad-key keystream:

```
[ ... 1300 bytes of pad-key keystream ... ]
```

**Iteration 1: Dave (the innermost).** Shift the buffer right by 100 bytes. The leading 100 bytes are about to be overwritten:

```
[ 100 bytes about to be written | 1200 bytes of shifted noise ]
```

Write Dave's bigsize-prefixed TLV payload (68 bytes of payment_data, etc.) at the front. Append a 32-byte all-zero HMAC (Dave is the destination; there's no further layer to commit to). The buffer's first 100 bytes now contain `dave_payload || 0x00 * 32`.

XOR the entire 1,300-byte buffer with Dave's `rho` keystream. *Then* (and only on this innermost iteration) overwrite the trailing 130 bytes of the buffer with the filler we computed in chapter 6:

```
[ encrypted Dave layer | encrypted noise | 130 bytes of filler ]
```

Now compute Dave's HMAC: `dave_hmac = HMAC-SHA256(dave_mu, buffer || associated_data)`. Save it as `next_hmac` for Charlie's iteration.

**Iteration 2: Charlie.** Shift the buffer right by 65 bytes. The original contents shift back; the leading 65 bytes are now about to be overwritten. Write Charlie's TLV payload (~33 bytes) at the front, then append `dave_hmac`. The buffer's first 65 bytes now contain `charlie_payload || dave_hmac`.

XOR the whole buffer with Charlie's `rho` keystream. Compute Charlie's HMAC over the result concatenated with `associated_data`, using `charlie_mu`. Save as `next_hmac` for Bob. (We don't overlay filler again; that step only happens on the innermost iteration.)

**Iteration 3: Bob.** Shift right by 65. Write Bob's TLV payload (~33 bytes) and append `charlie_hmac`. XOR with Bob's `rho` keystream. Compute Bob's HMAC with `bob_mu` over the buffer concatenated with `associated_data`. Done.

The buffer at this point is the `hop_payloads` field of the packet. The HMAC we just computed is the packet's `hmac` field. Bob will be the first to receive this onion, and his HMAC verification will use the packet's `hmac` (which is `bob_hmac`).

## What the forwarder sees

When Bob receives the packet, he'll:

1. Verify `HMAC-SHA256(bob_mu, hop_payloads || associated_data) == packet.hmac`. (Yes, because we just computed exactly that.)
2. XOR the `hop_payloads` field with his `rho` keystream. This undoes the encryption from iteration 3.
3. Read his TLV payload from the front. Read `charlie_hmac` from the bytes right after his TLV.
4. Shift the buffer left by Bob's hop payload size (to remove his own hop payload). Extend the trailing position with bytes from his rho keystream (we'll see how in chapter 8).
5. Forward to Charlie with the shifted-left `hop_payloads`, the next ephemeral pubkey <m>E_AC</m>, and `charlie_hmac` as the `hmac` field.

When Charlie receives Bob's forwarded packet, the `hmac` field is now `charlie_hmac`. His `mu` key will verify that against the `hop_payloads` he sees (concatenated with the same `associated_data`), and the chain continues.

The clever observation is that the per-hop HMACs are a **commitment from the outside in**. Bob's HMAC commits to Charlie's layer. Once Bob has verified his HMAC, he knows the packet is authentic and unmodified.

<details>
<summary>Why is the HMAC over the encrypted bytes, not the plaintext?</summary>

If the HMAC were over the plaintext, the forwarder would have to decrypt before it could verify integrity. That's bad practice: the forwarder would be processing untrusted data with its `rho` keystream before knowing whether the packet was tampered with. By computing the HMAC over the encrypted bytes (the `hop_payloads` field as it appears on the wire), the forwarder verifies integrity *first* and only proceeds to decrypt if verification passes.

This matches the encrypt-then-MAC pattern from the [Noise course's primitives chapter](/noise-tutorial/crypto-primitives), where the same principle applies to AEAD constructions: never feed unauthenticated data into the next stage of processing.

</details>

## Why filler goes after Dave's XOR

You might wonder why the filler overlay (step 4 above) happens *after* Dave's `rho` XOR, rather than before. Here's the reason.

When Dave's `rho` keystream XORs the buffer, it XORs *everything*: Dave's payload, the previously-noise-filled trailing positions, all of it. After that XOR completes, we directly overwrite the trailing `len(filler)` bytes with the filler value. From this point onward, those trailing bytes are exactly the bytes that need to appear at the back of each forwarder's view of the buffer (the filler we computed in chapter 6 was designed to be those bytes).

If we tried to pre-place the filler in the initial buffer and let the wraps propagate it forward, the math wouldn't line up. Dave's right-shift drops the last `dave_slot_size` bytes off the back of the buffer. If `dave_slot_size` is large (say, the destination payload is much bigger than the forwarder payloads), those dropped bytes can include the filler we tried to pre-place. The filler ends up at the wrong offset and HMAC verification breaks.

By overwriting the trailing positions *after* Dave's XOR, we put the filler exactly where it needs to be regardless of the destination's hop payload size.

## Wrap one hop, then loop

The construction has two pieces. First, a per-hop function `wrap_hop` that does the shift-write-XOR-HMAC dance for a single iteration. Second, a top-level `build` function that runs `wrap_hop` for each hop in reverse order, threading the `next_hmac` through the loop, applying the filler overlay on the innermost iteration, and finally assembling the 1,366-byte packet.

You'll write both. The first exercise (`wrap_hop`) gives you a clean primitive for one iteration. The second exercise (`build`) uses your earlier `derive_shared_secrets`, your `derive_keys`, your `generate_filler`, and your new `wrap_hop` to produce a complete spec-compliant packet. Don't worry: the `build` exercise comes with a structured starter code skeleton so you can focus on the substantive moves (the reverse loop and the filler overlay) rather than bookkeeping.
