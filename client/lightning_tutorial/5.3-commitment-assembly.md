# Commitment Transaction Assembly

## Overview

Now we have all the building blocks: scripts, keys, and the obscured commitment number. It is time to assemble the complete commitment transaction. This transaction spends the funding output and creates the `to_local` and `to_remote` outputs that distribute the channel funds.

## Transaction Structure

A commitment transaction has the following structure:

| Field | Value |
|-------|-------|
| Version | `2` |
| Input | The funding outpoint (txid + vout) |
| Input sequence | Encodes upper 24 bits of obscured commitment number |
| Output(s) | to_local, to_remote, and any HTLC outputs |
| Locktime | Encodes lower 24 bits of obscured commitment number |

### The Single Input

The commitment transaction always has exactly **one input**: the funding output. This input references the funding transaction's txid and the output index (vout) of the P2WSH multisig output.

The `nSequence` field of this input carries the upper 24 bits of the obscured commitment number (with the `0x80000000` base), as we covered in the previous chapter.

## Creating the Outputs

### The to_local Output

The `to_local` output pays the broadcaster (local party) through the revocable delayed script:

```python
def create_to_local_output(
    revocation_pubkey: bytes,
    to_self_delay: int,
    local_delayed_pubkey: bytes,
    amount_sat: int
) -> tuple:
    """Create the to_local output (P2WSH)."""
    witness_script = to_local_script(revocation_pubkey, to_self_delay, local_delayed_pubkey)
    output_script = script_to_p2wsh(witness_script)
    return (amount_sat, output_script)
```

### The to_remote Output

The `to_remote` output pays the counterparty (remote party) directly via P2WPKH:

```python
def create_to_remote_output(
    remote_payment_pubkey: bytes,
    amount_sat: int
) -> tuple:
    """Create the to_remote output (P2WPKH with static remotekey)."""
    output_script = to_remote_script(remote_payment_pubkey)
    return (amount_sat, output_script)
```

## Fee Deduction

The commitment transaction fee is paid by the **channel funder** (the party who initiated the channel open). The fee is deducted from the funder's output.

If the local party is the funder, the fee comes from the `to_local` output:

```python
def compute_to_local_amount(local_balance_msat: int, fee_sat: int, is_funder: bool) -> int:
    """Compute the to_local output amount in satoshis."""
    amount_sat = local_balance_msat // 1000
    if is_funder:
        amount_sat -= fee_sat
    return amount_sat
```

The fee is calculated based on the **feerate** (satoshis per 1000 weight units) and the transaction's weight:

```text
fee = feerate * weight / 1000
```

The base weight of a commitment transaction (with no HTLCs) is **724 weight units**. This includes:
- Transaction overhead (version, locktime, etc.)
- One input (the funding outpoint)
- Two outputs (to_local and to_remote)

Each HTLC output adds **172 weight units** to the commitment transaction.

## Dust Limit Filtering

An output is considered **dust** if its value is below the dust limit. Dust outputs are uneconomical to spend because the fee to spend them would exceed their value. BOLT 3 specifies that:

- The `to_local` output is omitted if its amount (after fee deduction) is below the dust limit.
- The `to_remote` output is omitted if its amount is below the dust limit.

```python
DUST_LIMIT_SAT = 546  # Default dust limit for P2WSH and P2WPKH

def should_include_output(amount_sat: int) -> bool:
    """Check if an output amount is above the dust limit."""
    return amount_sat >= DUST_LIMIT_SAT
```

If an output is omitted due to being dust, its value effectively goes to the miner as part of the transaction fee.

## Output Ordering

BOLT 3 requires commitment transaction outputs to be sorted using a BIP69-style ordering:

1. **By value** (ascending): Outputs with smaller amounts come first.
2. **By script** (lexicographic, ascending): If two outputs have the same value, sort by their output script bytes.

This deterministic ordering ensures both parties independently produce the same transaction:

```python
def sort_outputs(outputs: list[tuple[int, bytes]]) -> list[tuple[int, bytes]]:
    """Sort outputs by (amount, script) per BOLT 3 / BIP69."""
    return sorted(outputs, key=lambda o: (o[0], o[1]))
```

Note: When HTLC outputs are present, the sorting also considers `cltv_expiry` as a tiebreaker after script comparison. We will cover this in the HTLC section.

## Assembling the Transaction

Putting it all together:

```python
def build_commitment_tx(
    funding_txid: bytes,
    funding_vout: int,
    commitment_number: int,
    obscure_factor: int,
    local_amount_sat: int,
    remote_amount_sat: int,
    to_local_script_bytes: bytes,
    remote_payment_pubkey: bytes,
    dust_limit_sat: int
):
    # 1. Compute obscured commitment number
    locktime, sequence = set_obscured_commitment_number(commitment_number, obscure_factor)

    # 2. Create outputs
    outputs = []

    # to_local (P2WSH)
    if local_amount_sat >= dust_limit_sat:
        to_local_output_script = script_to_p2wsh(to_local_script_bytes)
        outputs.append((local_amount_sat, to_local_output_script))

    # to_remote (P2WPKH)
    if remote_amount_sat >= dust_limit_sat:
        to_remote_output_script = to_remote_script(remote_payment_pubkey)
        outputs.append((remote_amount_sat, to_remote_output_script))

    # 3. Sort outputs (BIP69-style)
    outputs = sort_outputs(outputs)

    # 4. Build the transaction
    # Version 2, single input (funding outpoint), sorted outputs
    # locktime and sequence carry the obscured commitment number
    return serialize_tx(
        version=2,
        inputs=[(funding_txid, funding_vout, sequence)],
        outputs=outputs,
        locktime=locktime
    )
```

## Transaction Serialization

The commitment transaction is serialized in the standard Bitcoin transaction format:

```text
version:     02000000
marker:      00
flag:        01
input_count: 01
  txid:      <funding_txid, 32 bytes, little-endian>
  vout:      <funding_vout, 4 bytes, little-endian>
  script:    00 (empty scriptSig for SegWit)
  sequence:  <obscured upper 24 bits + 0x80000000>
output_count: <number of non-dust outputs>
  [for each output:]
    amount:  <satoshis, 8 bytes, little-endian>
    script:  <output script, length-prefixed>
witness:     (empty for now -- filled during finalization)
locktime:    <obscured lower 24 bits + 0x20000000>
```

The witness field is left empty at this stage. It will be filled in during the commitment finalization step, after both parties have exchanged their signatures.
