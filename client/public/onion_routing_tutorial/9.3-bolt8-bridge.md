# BOLT 8 and BOLT 4: The Two Encryption Layers

If you've completed the **Noise Protocol tutorial** on this site, you've already built one of Lightning's two encryption layers: the transport layer. In this course, you've built the other: the routing layer. Let's take a step back and see how these two layers fit together.

## Two Different Problems

Lightning needs to solve two distinct encryption problems:

1. **Point-to-point encryption.** When Bob sends a message to Carol over their TCP connection, nobody eavesdropping on the wire should be able to read it. This is the same problem that TLS solves for web traffic. In Lightning, it's handled by **BOLT 8**, which uses the Noise Protocol framework (specifically, the Noise_XK handshake pattern) to establish an encrypted channel between two directly connected peers.

2. **End-to-end encryption.** When Alice constructs payment instructions that will traverse multiple hops, each intermediate node should only see its own instructions, not anyone else's. This is the problem we've been solving throughout this course. It's handled by **BOLT 4**, which uses the Sphinx construction to build layered onion packets.

These are fundamentally different problems with different trust models. BOLT 8 protects against external eavesdroppers. BOLT 4 protects against the intermediate nodes themselves.

## How They Connect

When Alice sends a payment through Bob and Carol to Dave, here's what happens at the message level:

Alice constructs a Sphinx onion packet (BOLT 4). This packet contains encrypted layers for Bob, Carol, and Dave. Alice then sends an `update_add_htlc` message to Bob. This message contains the onion packet, the payment hash, the amount, and the CLTV expiry.

But Alice doesn't just drop this message onto a raw TCP socket. The `update_add_htlc` message is itself encrypted using the Noise session that Alice and Bob established when they first connected (BOLT 8). So the message is encrypted twice:

- The **inner encryption** (BOLT 4, Sphinx): The onion packet encrypts each hop's payload so that only the intended hop can read it. This is end-to-end, spanning the full route.
- The **outer encryption** (BOLT 8, Noise): The entire `update_add_htlc` message (including the onion) is encrypted for the direct peer connection. This is point-to-point, covering a single link.

When Bob receives the message, he first decrypts the Noise layer (BOLT 8) to read the `update_add_htlc` message. Then he processes the onion packet (BOLT 4), peeling his layer to learn that he should forward to Carol. He constructs a new `update_add_htlc` message for Carol (with the processed onion), and encrypts it using his Noise session with Carol.

At every hop, the same pattern repeats: decrypt the transport layer, process the routing layer, re-encrypt for the next transport link.

## The Layer Cake

If you think about this in terms of a network stack, the layers look like this:

Each layer has a distinct job and a distinct scope. The Noise layer encrypts everything between two peers, so an ISP or network observer sees nothing but encrypted bytes. The Sphinx layer encrypts payment instructions across the full route, so intermediate nodes can only read their own hop. The application layer manages the channel state machine (the commitment dance we covered earlier).

## Double Encryption in Practice

Let's trace a concrete moment. Alice is sending `update_add_htlc` to Bob. The message contains an onion packet of exactly 1,366 bytes. Inside that onion, Bob's layer is encrypted with a shared secret derived from Alice's session key and Bob's node public key. Carol's layer (deeper in the onion) is encrypted with a different shared secret. Dave's layer is encrypted with yet another.

The entire message (including the 1,366-byte onion) is then encrypted using the Noise session keys that Alice and Bob negotiated during their BOLT 8 handshake. These are symmetric keys (ChaCha20-Poly1305) derived from a Diffie-Hellman exchange performed when the peers first connected.

So at the moment the message leaves Alice's node and travels over the wire to Bob, it's wrapped in two completely independent encryption schemes, built from different key material and serving different purposes.

## What You've Built

Between the two tutorials on this site, you've built two of the four layers in that stack:

- In the **Noise Protocol tutorial**, you built the transport layer. You performed the Noise_XK handshake, derived encryption keys, and encrypted messages between two peers. That's BOLT 8.
- In **this course**, you built the routing layer. You computed shared secrets, derived onion keys, filled packets with encrypted hop payloads, and peeled layers on the receiving end. That's BOLT 4.

The Lightning protocol emerges from these layers working together. A payment is an application-layer concept (HTLCs, commitment transactions, preimages). The instructions for routing that payment are encrypted end-to-end by Sphinx. And every message carrying those instructions is encrypted point-to-point by Noise.

You've now seen how both encryption layers work from the inside. The rest of the protocol (channel management, gossip, invoices, watchtowers) builds on top of the foundation you've explored here.
