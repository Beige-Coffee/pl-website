# Per-Commitment Key Derivation

## Why Per-Commitment Keys?

Each commitment transaction uses different keys, even though the base keys (the basepoints) remain the same throughout the channel's lifetime. This is achieved through **per-commitment key derivation**: combining a static basepoint with the per-commitment point to produce a unique key for each state.

This design serves two purposes:

1. **Privacy**: Each commitment transaction uses different public keys, making it harder to correlate channel states on-chain.
2. **Revocation**: The per-commitment derivation ties each key to a specific state, enabling the revocation mechanism when the per-commitment secret is revealed.

## The derive_pubkey Formula

BOLT 3 defines the per-commitment public key derivation as:

```text
derived_pubkey = basepoint + SHA256(per_commitment_point || basepoint) * G
```

Where:
- `basepoint` is the static public key (e.g., `delayed_payment_basepoint`)
- `per_commitment_point` is the public key for this specific commitment
- `G` is the secp256k1 generator point
- `||` is concatenation
- `*` is scalar-point multiplication
- `+` is elliptic curve point addition

In Python:

```python
import hashlib

def derive_pubkey(basepoint: bytes, per_commitment_point: bytes) -> bytes:
    """Derive a per-commitment public key from a basepoint and per-commitment point.

    derived_key = basepoint + SHA256(per_commitment_point || basepoint) * G
    """
    # Compute the tweak: SHA256(per_commitment_point || basepoint)
    tweak = hashlib.sha256(per_commitment_point + basepoint).digest()

    # Multiply the tweak by the generator point G
    tweak_point = scalar_multiply_G(tweak)

    # Add the basepoint
    return ec_point_add(basepoint, tweak_point)
```

### How It Works

The SHA256 hash produces a 32-byte scalar value (the "tweak"). Multiplying this scalar by the generator point `G` produces a new curve point. Adding this to the basepoint effectively "shifts" the basepoint by a deterministic amount that depends on the per-commitment point.

Since the per-commitment point changes with each commitment, the derived key is different for every state, even though the basepoint stays the same.

## The derive_privkey Formula

The corresponding private key derivation follows naturally:

```text
derived_privkey = basepoint_secret + SHA256(per_commitment_point || basepoint)
```

This is simply adding the same tweak scalar to the base private key (modulo the curve order):

```python
def derive_privkey(basepoint_secret: bytes, per_commitment_point: bytes, basepoint: bytes) -> bytes:
    """Derive a per-commitment private key.

    derived_privkey = basepoint_secret + SHA256(per_commitment_point || basepoint)
    """
    CURVE_ORDER = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141

    tweak = hashlib.sha256(per_commitment_point + basepoint).digest()

    privkey = (
        int.from_bytes(basepoint_secret, 'big') +
        int.from_bytes(tweak, 'big')
    ) % CURVE_ORDER

    return privkey.to_bytes(32, 'big')
```

### Mathematical Consistency

Note the mathematical relationship: if `basepoint = basepoint_secret * G`, then:

```text
derived_pubkey  = basepoint + tweak * G
                = basepoint_secret * G + tweak * G
                = (basepoint_secret + tweak) * G
                = derived_privkey * G
```

The derived public key is indeed the public key corresponding to the derived private key. This is a fundamental property of elliptic curve arithmetic.

## Which Keys Use This Derivation?

Not all channel keys use per-commitment derivation. Here is the breakdown:

| Key | Derivation Method |
|-----|------------------|
| `local_delayed_pubkey` | `derive_pubkey(delayed_payment_basepoint, per_commitment_point)` |
| `local_htlc_pubkey` | `derive_pubkey(local_htlc_basepoint, per_commitment_point)` |
| `remote_htlc_pubkey` | `derive_pubkey(remote_htlc_basepoint, per_commitment_point)` |
| `revocation_pubkey` | Special two-component formula (covered in previous chapter) |
| `local_payment_pubkey` | Uses `payment_basepoint` directly (static remotekey) |
| `funding_pubkey` | Static -- no per-commitment derivation |

The `payment_basepoint` uses static remotekey (`option_static_remotekey`), meaning it is used directly without per-commitment derivation. This simplifies recovery: the remote party can always sweep their `to_remote` output with just their base private key.

## Derivation in Context

Let us trace through how per-commitment key derivation fits into building a commitment transaction:

### For Commitment #42 (Alice's version)

1. Alice derives `per_commitment_secret_42` from her commitment seed.
2. Alice computes `per_commitment_point_42 = per_commitment_secret_42 * G`.
3. Alice sends `per_commitment_point_42` to Bob.

Now both parties can derive the keys for this commitment:

```text
local_delayed_pubkey = derive_pubkey(Alice.delayed_payment_basepoint, per_commitment_point_42)
  -> Used in Alice's to_local output (with CSV delay)

revocation_pubkey = derive_revocation_pubkey(Bob.revocation_basepoint, per_commitment_point_42)
  -> Used in Alice's to_local output (revocation path)

local_htlc_pubkey = derive_pubkey(Alice.htlc_basepoint, per_commitment_point_42)
  -> Used in HTLC scripts on Alice's commitment

remote_htlc_pubkey = derive_pubkey(Bob.htlc_basepoint, per_commitment_point_42)
  -> Used in HTLC scripts on Alice's commitment
```

### After Revocation

When Alice revokes commitment #42, she reveals `per_commitment_secret_42` to Bob. Bob can now:

1. Derive `revocation_privkey_42` (using his `revocation_basepoint_secret` and Alice's `per_commitment_secret_42`).
2. If Alice broadcasts commitment #42, Bob uses this private key to sweep Alice's to_local output.

## A Note on Notation

In the BOLT specifications and Lightning implementations, you will see these keys referred to with different naming conventions:

- **Basepoints** are the static public keys: `revocation_basepoint`, `htlc_basepoint`, etc.
- **Basepoint secrets** are the corresponding private keys: `revocation_basepoint_secret`, etc.
- **Per-commitment keys** are derived using the formulas above: `local_delayed_pubkey`, `local_htlc_pubkey`, etc.

The "local" and "remote" prefixes are always relative to the holder of the commitment transaction. In Alice's commitment, "local" means Alice and "remote" means Bob.
