# Hop Payloads and TLV Encoding

Alice has calculated the exact amounts and timelocks for every hop in her route. Now she needs to package this data into **hop payloads**, the per-hop instructions that each node will read when it peels its layer of the onion.

Each hop payload answers one simple question for the receiving node: "What should I do with this payment?"

For an intermediate node like Bob, the answer is: "Forward 50,003,000 msat on channel `700000x2x0` with outgoing CLTV 700,048."

For the final recipient Dave, the answer is: "This payment is for you. The amount is 50,000,000 msat with CLTV 700,018."

## TLV Encoding

Hop payloads are encoded using **TLV (Type-Length-Value)** format, a flexible binary encoding defined in BOLT 4. TLV is the standard way Lightning encodes structured data when the set of fields may vary or grow over time.

Each field in a TLV stream consists of three parts:

1. **Type**: a number identifying what this field is
2. **Length**: how many bytes the value occupies
3. **Value**: the actual data

The type and length are encoded using **bigsize** encoding, a variable-length integer format:
- Values 0 to 252 use a single byte
- Values 253 to 65,535 use 3 bytes (a `0xfd` prefix followed by 2 bytes big-endian)
- Values 65,536 to 4,294,967,295 use 5 bytes (a `0xfe` prefix followed by 4 bytes big-endian)
- Larger values use 9 bytes (a `0xff` prefix followed by 8 bytes big-endian)

For the field types we'll encounter (2, 4, 6, 8), all type numbers fit in a single byte.

## The Key TLV Fields

BOLT 4 defines four TLV field types for hop payloads:

**Type 2: `amt_to_forward`** (variable length, typically 4-8 bytes)
The number of millisatoshis this hop should forward to the next node. Encoded as a big-endian `tu64` (truncated unsigned 64-bit integer, meaning leading zero bytes are stripped).

**Type 4: `outgoing_cltv_value`** (variable length, typically 4 bytes)
The CLTV expiry for the outgoing HTLC. Also encoded as a truncated `tu64`.

**Type 6: `short_channel_id`** (exactly 8 bytes)
The channel this hop should use to forward the payment. Encoded as the standard 8-byte short channel ID format.

**Type 8: `payment_data`** (exactly 32 + variable bytes)
Only present on the **final hop**. Contains the `payment_secret` (32 bytes) followed by `total_msat` (truncated `tu64`). The payment_secret proves to Dave that this payment matches a specific invoice he generated.

## Intermediate vs. Final Hop Payloads

The TLV fields differ depending on whether the hop is intermediate or final:

**Intermediate hops** (Bob, Carol) include:
- Type 2: `amt_to_forward`
- Type 4: `outgoing_cltv_value`
- Type 6: `short_channel_id`

**Final hop** (Dave) includes:
- Type 2: `amt_to_forward`
- Type 4: `outgoing_cltv_value`
- Type 8: `payment_data` (payment_secret + total_msat)

Crucially, the final hop does **not** include `short_channel_id` (type 6). Dave has no channel to forward on, since he is the destination. Instead, he gets `payment_data` (type 8), which lets him verify the payment matches his invoice.

<details>
  <summary>Why use TLV encoding instead of a fixed format?</summary>

TLV is extensible. New field types can be added in future protocol versions without breaking existing implementations. A node that doesn't recognize a field type can skip over it (using the length to jump ahead). This is also how application-level features work: custom TLV records with type numbers above 65,536 let applications embed their own data in payments. This is how features like **keysend** (spontaneous payments without an invoice) and **message attachments** work. The sender includes extra TLV fields that the recipient's application knows how to interpret.
</details>

## Length Prefixing

One final detail: the complete hop payload is prefixed with its own length (as a bigsize value). This tells the onion-processing code how many bytes to read for this hop. The payload format shown in the diagram above illustrates this structure: a length prefix followed by the concatenated TLV fields.

This length prefix is important for the fixed-size onion trick we discussed in Section 1. After reading the payload, any remaining space in the 1,366-byte onion is filled with padding, so the packet size never changes.
