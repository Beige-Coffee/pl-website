### Is There Actually a Server?

Good question. Let's find out.

Below is a simple probe that opens a raw WebSocket connection to the course server and sends a plaintext "hello." There is no encryption and no handshake - it's just 5 bytes of ASCII over the wire.

The server is expecting a 50-byte Noise Protocol Act 1 message (1-byte version + 33-byte ephemeral public key + 16-byte MAC). When it gets plain English instead, it has no idea what to do, so it will reject the connection immediately.

### What Your Code Will Do

The dashboard at the bottom of this page will take all of the functions you've written throughout this course and run them against Programming Lightning's server. Under the hood, your Python code will open a real WebSocket connection, perform the XK handshake, and establish an encrypted channel, just as two Lightning nodes would.

Here's the flow:

1. **Pre-flight check**: When you click "Begin Handshake", the Lab will first verify that all 13 of your exercise solutions produce correct outputs. If they don't, you'll see a link to jump back and fix it.

2. **Handshake**: Once pre-flight checks pass, you'll watch each Act execute in real time. The color-coded bars on the arrows break down the raw bytes: version byte, ephemeral key, encrypted payload, MAC tag. Click any Act's bar to inspect individual bytes in the grid below.

3. **Encrypted Terminal**: After the handshake completes, you'll have a fully encrypted channel, and a terminal will appear where you can send messages back and forth to the server. Every message you type gets encrypted by your `encrypt_message()`, sent over the wire, and decrypted by the server. Responses come back the same way.

### Real Lightning Messages

The quick command buttons in the terminal send real **Lightning BOLT protocol messages**, the same message types that Lightning nodes exchange after establishing a Noise connection:

- **`init`** (BOLT 1, type 16): The mandatory first message after connecting. Both peers announce which protocol features they support.
- **`ping`** (BOLT 1, type 18): A keepalive message. The server responds with `pong`. This is the simplest round trip, and it proves your encryption and decryption are working end-to-end.
- **`node_ann`** (BOLT 7, type 257): Requests the server's node announcement. Lightning nodes use this broadcast to advertise their identity, alias, and network addresses to the gossip network.

### Watching Key Rotation

The **Cipher State** panel shows the live internal state of your `CipherState` objects: the current encryption key, the chaining key, and the nonce counter. These are the exact `self.key`, `self.chaining_key`, and `self.nonce` attributes from the class you wrote.

Every time you send or receive a message, the nonce increments by 2 (one for the length prefix, one for the body). **Watch the progress bar fill toward 1,000**.

When the nonce hits 1,000, your `_maybe_rotate()` method fires. It calls `hkdf_two_keys(self.chaining_key, self.key)` to derive a completely new encryption key, then resets the nonce to zero. The server does the same thing on its end. **The panel will flash gold and the key value will change before your eyes**.

If you don't want to send 500 messages manually (totally understandable), you can use the **Burst** control in the terminal. It auto-calculates how many pings are needed to reach the next rotation (500 messages = 1,000 nonces). When it rolls over, that's your code performing a live key rotation on a real encrypted connection!

<details>
  <summary>Why does key rotation happen every 1,000 nonces instead of, say, every message?</summary>

There's a tradeoff. Rotating too often adds computational overhead (each rotation requires an HKDF derivation). Rotating too rarely means a single compromised key can decrypt a long stretch of messages. The BOLT 8 spec settled on 1,000 nonces (500 messages) as a balance between security and performance.

</details>

<details>
  <summary>What if the server and I get out of sync on key rotation?</summary>

They can't, as long as both sides follow the protocol correctly. Both your `CipherState` and the server's `CipherState` increment nonces identically for each encrypt/decrypt operation. Since they derive the same keys from the same chaining key, they always rotate at the same point. If something goes wrong (a dropped message, a decryption failure), the connection breaks immediately. There's no way to silently fall out of sync.

</details>

<details>
  <summary>Is this the same encryption that real Lightning nodes use?</summary>

Yes, exactly. BOLT 8 specifies the Noise XK handshake pattern with `secp256k1` for ECDH and `ChaCha20-Poly1305` for authenticated encryption. The handshake you just performed, the transport framing, and the key rotation logic are all identical to what LND, CLN, and LDK implement.

</details>
