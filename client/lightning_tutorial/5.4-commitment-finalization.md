# Commitment Finalization

## From Unsigned to Signed

We have assembled the commitment transaction with its inputs, outputs, locktime, and sequence. But the transaction cannot be broadcast without a valid witness -- the signatures that prove both funding key holders authorized this spend.

The funding output is a 2-of-2 P2WSH multisig, so the witness needs both parties' signatures plus the funding script itself.

## The Witness Structure

The witness for spending the funding output is:

```text
[empty] [signature_1] [signature_2] [funding_script]
```

Specifically:
1. An **empty byte string** (`0x00` or `""`)
2. The **first signature** (from the party whose pubkey sorts first lexicographically)
3. The **second signature** (from the party whose pubkey sorts second)
4. The **funding script** (the full 2-of-2 multisig witness script)

### The OP_CHECKMULTISIG Bug

Why is there an empty element at the beginning? This is due to a well-known **off-by-one bug** in Bitcoin's `OP_CHECKMULTISIG` implementation that has existed since the earliest versions of Bitcoin.

When `OP_CHECKMULTISIG` executes, it pops one more item from the stack than the number of signatures specified. In a 2-of-2 multisig, it expects 2 signatures but pops 3 items. The extra item is consumed but never validated.

By consensus, this extra item **must be an empty byte string** (`OP_0`). If it is any other value, modern Bitcoin nodes reject the transaction as non-standard (this was enforced by BIP147 as a softfork called NULLDUMMY).

This bug was never fixed in the traditional sense because changing the behavior of `OP_CHECKMULTISIG` would require a hard fork. Instead, the community standardized the workaround: always provide an empty element.

## Signature Ordering

The two signatures in the witness must be in the **same order** as their corresponding public keys in the funding script. Since the funding script sorts public keys lexicographically, the signatures must follow the same order:

```python
def order_signatures(
    local_pubkey: bytes,
    remote_pubkey: bytes,
    local_sig: bytes,
    remote_sig: bytes
) -> tuple[bytes, bytes]:
    """Order signatures to match the lexicographic pubkey order in the funding script."""
    if local_pubkey < remote_pubkey:
        return local_sig, remote_sig
    else:
        return remote_sig, local_sig
```

If Alice's pubkey is lexicographically smaller than Bob's pubkey:
- `signature_1` = Alice's signature
- `signature_2` = Bob's signature

If Bob's pubkey is smaller:
- `signature_1` = Bob's signature
- `signature_2` = Alice's signature

Getting this order wrong will cause `OP_CHECKMULTISIG` to fail, even if both signatures are individually valid.

## Building the Witness

The witness is serialized as a list of items, each prefixed by its length:

```python
def build_commitment_witness(
    local_sig: bytes,
    remote_sig: bytes,
    local_pubkey: bytes,
    remote_pubkey: bytes,
    funding_script: bytes
) -> bytes:
    """Build the witness for a commitment transaction spending the funding output."""
    # Order signatures to match pubkey order in funding script
    sig1, sig2 = order_signatures(local_pubkey, remote_pubkey, local_sig, remote_sig)

    # Witness: [item_count] [empty] [sig1] [sig2] [funding_script]
    witness = bytes([0x04])  # 4 witness items

    # Item 1: empty (NULLDUMMY for OP_CHECKMULTISIG bug)
    witness += bytes([0x00])

    # Item 2: first signature
    witness += bytes([len(sig1)]) + sig1

    # Item 3: second signature
    witness += bytes([len(sig2)]) + sig2

    # Item 4: funding script (witness script)
    witness += encode_varint(len(funding_script)) + funding_script

    return witness
```

## The Signature Exchange Protocol

In practice, commitment transaction signing follows a specific protocol:

### During Channel Open

1. Both parties exchange their `funding_pubkey` and construct the funding script.
2. The funder creates the funding transaction (but does not broadcast yet).
3. Both parties create commitment transaction #0 and sign it.
4. They exchange signatures via `funding_created` and `funding_signed` messages.
5. Only after both parties have valid commitment signatures does the funder broadcast the funding transaction.

### During State Updates

1. One party proposes a new state (e.g., after a payment).
2. Both parties construct the new commitment transaction.
3. They exchange signatures for the new commitment via `commitment_signed` messages.
4. They acknowledge receipt via `revoke_and_ack`, which reveals the per-commitment secret for the old state.

This order ensures that both parties always have a valid, signed commitment transaction they can broadcast.

## Each Signature is for the Counterparty's Version

An important subtlety: when Alice signs a commitment transaction, she is signing **Bob's version** (the one Bob holds). Alice uses her `funding_privkey` to create a signature, which she sends to Bob. Bob combines it with his own signature to finalize his commitment transaction.

Similarly, Bob signs Alice's version and sends the signature to Alice.

This means each party holds a commitment transaction signed by both parties, ready to broadcast at any time.

## Signature Format

Each signature in the witness is a DER-encoded ECDSA signature with the `SIGHASH_ALL` byte appended:

```text
<DER-encoded-signature> <0x01>
```

The `0x01` at the end indicates `SIGHASH_ALL`, meaning the signature covers the entire transaction (all inputs and all outputs).

## Finalization Summary

The complete finalization process:

1. **Compute the sighash**: Use BIP143 to compute the signature hash for the funding input.
2. **Sign**: Each party signs the counterparty's commitment transaction with their `funding_privkey`.
3. **Exchange signatures**: Signatures are sent via Lightning protocol messages.
4. **Order signatures**: Place them in the witness in the same order as the pubkeys in the funding script.
5. **Build witness**: Construct `[empty, sig1, sig2, funding_script]`.
6. **Attach witness**: Insert the witness into the commitment transaction.

The result is a fully valid, broadcastable Bitcoin transaction that can settle the channel on-chain at any time.
