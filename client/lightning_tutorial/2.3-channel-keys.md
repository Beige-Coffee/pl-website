# Channel Keys & Public Keys

## Deriving a Complete Key Set

Now that we have our keys manager and understand BIP32 derivation, it is time to derive the full set of keys for a Lightning channel. Each channel needs six private keys, one from each key family, and their corresponding public keys.

Recall the six key families and their derivation paths for channel index `n`:

```text
Family 0 (MultiSig):        m/1017h/0h/0h/0/n  ->  funding_privkey
Family 1 (RevocationBase):  m/1017h/0h/1h/0/n  ->  revocation_basepoint_secret
Family 2 (HtlcBase):        m/1017h/0h/2h/0/n  ->  htlc_basepoint_secret
Family 3 (PaymentBase):     m/1017h/0h/3h/0/n  ->  payment_basepoint_secret
Family 4 (DelayBase):       m/1017h/0h/4h/0/n  ->  delayed_payment_basepoint_secret
Family 5 (CommitmentSeed):  m/1017h/0h/5h/0/n  ->  commitment_seed
```

## From Private Keys to Public Keys

Each private key (except the commitment seed) needs to be converted to a **compressed public key** for use in scripts and protocol messages. The conversion uses elliptic curve multiplication on the secp256k1 curve:

```python
from cryptography.hazmat.primitives.asymmetric.ec import (
    SECP256K1, derive_private_key
)

def privkey_to_pubkey(secret: bytes) -> bytes:
    """Convert a 32-byte private key to a 33-byte compressed public key."""
    private_key = derive_private_key(
        int.from_bytes(secret, 'big'),
        SECP256K1()
    )
    public_numbers = private_key.public_key().public_numbers()
    x = public_numbers.x.to_bytes(32, 'big')
    prefix = b'\x02' if public_numbers.y % 2 == 0 else b'\x03'
    return prefix + x
```

A compressed public key is 33 bytes: a single prefix byte (`0x02` for even y-coordinate, `0x03` for odd) followed by the 32-byte x-coordinate. This is the standard format used throughout the Lightning protocol.

## The Channel Key Set

After derivation and public key computation, each party in a channel has:

### Private Keys (kept secret)

| Key | Usage |
|-----|-------|
| `funding_privkey` | Signs commitment transactions (spends the funding output) |
| `revocation_basepoint_secret` | Used to derive revocation private keys for old states |
| `htlc_basepoint_secret` | Signs HTLC-timeout and HTLC-success transactions |
| `payment_basepoint_secret` | Spends the to_remote output (with static remotekey) |
| `delayed_payment_basepoint_secret` | Spends the to_local output after CSV delay |

### Public Keys (shared with counterparty)

| Key | Shared During |
|-----|---------------|
| `funding_pubkey` | Channel open (used in funding script) |
| `revocation_basepoint` | Channel open (used in commitment scripts) |
| `htlc_basepoint` | Channel open (used in HTLC scripts) |
| `payment_basepoint` | Channel open (used in to_remote output) |
| `delayed_payment_basepoint` | Channel open (used in to_local output) |

### The Commitment Seed (special case)

The `commitment_seed` (family 5) is different from the other five keys. It is not converted to a public key. Instead, it serves as the root from which all **per-commitment secrets** are derived using the shachain algorithm. We will cover this in detail in the Commitment Keys section.

## Key Exchange During Channel Open

When two nodes open a channel, they exchange their public keys in the `open_channel` and `accept_channel` messages:

```text
Alice -> Bob:  open_channel {
    funding_pubkey: <Alice's funding pubkey>,
    revocation_basepoint: <Alice's revocation basepoint>,
    htlc_basepoint: <Alice's htlc basepoint>,
    payment_basepoint: <Alice's payment basepoint>,
    delayed_payment_basepoint: <Alice's delayed basepoint>,
    ...
}

Bob -> Alice:  accept_channel {
    funding_pubkey: <Bob's funding pubkey>,
    revocation_basepoint: <Bob's revocation basepoint>,
    htlc_basepoint: <Bob's htlc basepoint>,
    payment_basepoint: <Bob's payment basepoint>,
    delayed_payment_basepoint: <Bob's delayed basepoint>,
    ...
}
```

After this exchange, both parties have all the public information they need to independently construct commitment transactions. The private keys never leave the node that generated them.

## Why Each Key Family Exists

You might wonder why Lightning needs six different key families instead of just one. The answer is **separation of concerns and security isolation**:

- **MultiSig** is the most critical key -- it controls the funding output. It is kept completely separate from the others.
- **RevocationBase** enables the penalty mechanism. Isolating it means a compromised HTLC key cannot be used to construct revocation keys.
- **HtlcBase** is used frequently for signing HTLC transactions. If it were the same as the funding key, a vulnerability in HTLC handling could compromise the entire channel.
- **PaymentBase** is used directly in the to_remote output (with static remotekey). It provides a simple recovery path.
- **DelayBase** is used for the to_local output. The delay mechanism requires its own key derivation path.
- **CommitmentSeed** generates per-commitment secrets. It is the foundation of the revocation system and needs the strongest isolation.

By keeping these key families separate with hardened derivation at the family level, a compromise of any single key family does not affect the others.
