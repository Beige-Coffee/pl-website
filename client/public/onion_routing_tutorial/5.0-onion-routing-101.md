# Onion Routing 101

In the last chapter we worked out how Alice can derive a shared secret with each hop in the route, one secret per hop, all rolling out deterministically from a single session key. This chapter walks through what she does with those secrets — end to end. Alice builds the onion, ships it, and we follow it from forwarder to forwarder until it lands at Dave.

By the end of this chapter you'll have a complete mental model of how a Sphinx onion gets constructed, transmitted, and peeled. Subsequent chapters refine specific pieces: chapter 6 splits the single shared secret into the five distinct keys an honest implementation needs, chapter 7 explains why packets are exactly 1,300 bytes long, and chapter 8 covers the clever trick that keeps the HMAC chain valid as the buffer shrinks.

For this whole chapter we'll keep some details fuzzy on purpose, with explicit forward references where things get refined later. The goal here is the *picture*, not yet the *precision*.

## Encryption is XOR with a keystream

The first piece is the encryption itself. Sphinx uses a stream cipher (specifically ChaCha20), which produces a deterministic stream of pseudo-random bytes given a key. Encryption is just **XOR-ing** the plaintext with that keystream byte-by-byte — and the same operation works both ways. Alice XORs plaintext with the keystream to encrypt; Bob XORs the ciphertext with the *same* keystream to decrypt. The keystream cancels itself out. That's why Alice and a forwarder only need to share one secret for encryption to work in both directions; they never have to exchange anything else.

## Alice builds the onion inside-out

Alice now constructs the packet. She doesn't write Bob's slot first and then Charlie's after. She writes the **innermost** slot first (Dave's, since he's the destination), wraps it in a layer of encryption, then wraps *that* in Charlie's slot + Charlie's encryption, then wraps *that* in Bob's slot + Bob's encryption.

This reverse-build pattern isn't an aesthetic choice. Each slot carries a 32-byte HMAC field that commits to the *next inner* layer. Bob's outer HMAC is computed over Charlie's encrypted layer (which itself contains Dave's encrypted layer underneath). So before Alice can compute Bob's HMAC, Charlie's layer has to already be built. Before she can compute Charlie's HMAC, Dave's layer has to already be built. The dependencies run from the outside in, which forces the construction to run from the inside out.

When the construction finishes, Alice has a 1,300-byte payload area with three nested encryption layers, plus an outer HMAC (`bob_hmac`) appended at the end. She ships the packet to Bob.

## Each hop peels its layer

Now follow the packet across the route. Each forwarder runs the same three-step ritual on whatever they receive:

1. **Verify the outer HMAC** — does it match what's expected for this hop's shared secret? If not, drop the packet.
2. **Decrypt with XOR** — the same operation Alice used to apply this layer of encryption.
3. **Read the slot at the front** — extract the forwarding info (amount, expiry, next channel) and the next-hop HMAC, then forward the rest.

The visual below mirrors the construction one exactly, but in reverse direction (route order: Bob, then Charlie, then Dave) and with the inverse semantics: encryption hatches *disappear* as each hop XORs their layer off, and slots get *removed* from the front as each hop reads and forwards.

<peel-primer></peel-primer>

Notice what happens at each XOR step: the hop's encryption layer comes off **every region** in the buffer, not just their own slot. Bob's XOR removes one layer from Bob's slot (revealing the plaintext TLV), one layer from Charlie's slot (which still has Charlie's layer underneath), one layer from Dave's slot (which still has Dave's and Charlie's layers underneath), and one layer from the padding. The next forwarder sees a buffer that looks freshly-encrypted by exactly two layers; the same recipe runs again with their key, and so on.

When the packet finally reaches Dave, his slot's HMAC field contains 32 zero bytes. That's the universal "you are the destination" signal. Dave checks the payment hash, verifies the amount, and claims the funds.

## The HMAC chain is the spine

The thing that holds all of this together is the **HMAC chain**. Read it from outermost to innermost:

- The packet's outer HMAC is `bob_hmac`.
- Inside Bob's slot, the HMAC field is `charlie_hmac`.
- Inside Charlie's slot (visible only after Bob decrypts and forwards), the HMAC field is `dave_hmac`.
- Inside Dave's slot, the HMAC field is `0x00…` (32 zeros).

Each link commits to *everything* below it in the onion: the slot, the next-inner HMAC, and all the encrypted layers underneath. A forwarder can't tamper with what's below their own slot without invalidating the HMAC the *next* forwarder will check. That's why the HMAC chain is what makes onion routing safe in the presence of malicious intermediate hops: any modification anywhere in the inner layers gets detected at the next hop and the packet is dropped.

## A small confession

Throughout this chapter we've been quietly using a *single* key per hop — Alice's shared secret with that hop, somehow doing every job at once: encrypting the forward payload, authenticating it with an HMAC, and (we'll see in a later chapter) handling the return path for errors too.

Here's the trouble: using one key for everything is a serious cryptographic mistake. Encryption and authentication are different jobs with different security models. Forward and backward directions need different keys to avoid catastrophic attacks. Reusing a single key across all four roles would break standard security proofs and open up real-world vulnerabilities.

Sphinx fixes this by deriving **five distinct named keys** from each shared secret, each with a specific role. The next chapter walks through what those roles are, why each one needs its own key, and how the derivation works.
