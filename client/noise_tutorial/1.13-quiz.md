# Quiz: Lightning's Noise Protocol

Think you've mastered Lightning's Noise Protocol? Let's find out! Below are 10 questions covering everything from the cryptographic primitives to the handshake process and message encryption. Each question has exactly one correct answer. Good luck!

### Question 1: Why does the Noise Protocol use ECDH rather than digital signatures for authentication?

- **A)** ECDH is more resistant to quantum computing attacks than signatures
- **B)** ECDH is more computationally efficient, and the resulting shared secrets can be directly used to derive encryption keys, simplifying the protocol
- **C)** The secp256k1 curve doesn't support signature operations, only key agreement
- **D)** Signatures would require a certificate authority, which contradicts Bitcoin's decentralized design

<details>
  <summary>Answer</summary>

**B)** As covered in the Noise Overview, Diffie-Hellman is more computationally efficient than signatures. When two parties perform ECDH and derive matching shared secrets, it implicitly proves each party knows their private key. These shared secrets can then be reused to derive encryption keys, combining authentication and key agreement into one step. This is a key design choice of the Noise Framework — signatures aren't needed because ECDH accomplishes both goals at once.

</details>

### Question 2: In Act 1, Alice encrypts a zero-length plaintext with ChaCha20-Poly1305. Why doesn't she encrypt any actual data?

- **A)** The Act 1 message must be exactly 50 bytes, and there's no room for additional plaintext after the version byte and Ephemeral Public Key
- **B)** ChaCha20-Poly1305 requires an empty plaintext for the initial use of any new key
- **C)** Alice's goal is only to prove she knows Bob's public key — the MAC alone, created with a key derived from their ECDH shared secret, is sufficient to prove this
- **D)** The plaintext is encrypted in a separate step that happens between Act 1 and Act 2

<details>
  <summary>Answer</summary>

**C)** Alice's goal in Act 1 is to prove she knows Bob's Static Public Key without revealing her own identity. Since `temp_k1` was derived from the ECDH shared secret (which used Bob's Static Public Key), the MAC will only verify correctly if Alice used the correct key material. She doesn't need to encrypt any data — she just needs Bob to validate the MAC. When Bob independently derives `temp_k1` using his static private key and Alice's Ephemeral Public Key, a successful MAC verification proves Alice computed the ECDH with his correct public key.

</details>

### Question 3: At what point during the Noise handshake does Bob first learn Alice's identity?

- **A)** Act 1 — when Alice proves she knows Bob's Static Public Key
- **B)** Act 2 — after the ephemeral-ephemeral (`ee`) ECDH exchange
- **C)** Act 3 — when Bob decrypts Alice's encrypted Static Public Key, but before he has verified the final MAC
- **D)** Act 3 — only after Bob successfully verifies the final MAC with `temp_k3`

<details>
  <summary>Answer</summary>

**C)** In Act 3 Step 3, Bob decrypts Alice's Static Public Key using `temp_k2` and nonce=1. This is the first time Bob knows who he is *apparently* talking to. However, he hasn't yet *authenticated* Alice — that happens later in Step 7 when he verifies the final MAC with `temp_k3`. The distinction matters: Bob learns Alice's claimed identity in Step 3, but doesn't *confirm* it until Step 7. If the MAC fails, Bob must discard the identity and terminate the connection.

</details>

### Question 4: Why does Alice use her Ephemeral Public Key — rather than her Static Public Key — for the ECDH in Act 1?

- **A)** Her Static Public Key uses a different curve than Bob's Ephemeral Key, making ECDH impossible
- **B)** Sending her Static Public Key in plaintext would reveal her Lightning node identity to eavesdroppers
- **C)** The Noise Protocol requires all Act 1 ECDH operations to use ephemeral keys for both parties
- **D)** Her Static Public Key is too large to fit in the 50-byte Act 1 message

<details>
  <summary>Answer</summary>

**B)** Alice's Static Public Key is her Lightning node identity. If she sent it in plaintext in Act 1, any eavesdropper monitoring the network would immediately know who is connecting to Bob. By using an Ephemeral Public Key instead, Alice can participate in ECDH while keeping her true identity hidden. This is the "X" in Lightning's XK handshake pattern — Alice's identity is transmitted in *encrypted* form later in Act 3, after enough key material has been established to protect it.

</details>

### Question 5: Why is forward secrecy NOT established until Act 2?

- **A)** Forward secrecy requires at least three ECDH operations, and only two are complete after Act 1
- **B)** Act 1 only uses hash functions — the first ECDH doesn't happen until Act 2
- **C)** The `es` ECDH in Act 1 involves Bob's Static Key, so if Bob's static private key were compromised, an attacker could recompute the shared secret using Alice's Ephemeral Public Key (which was sent in plaintext)
- **D)** Act 1 doesn't encrypt any data, so there's nothing that forward secrecy would protect

<details>
  <summary>Answer</summary>

**C)** In Act 1, Alice computes `es = ECDH(alice_ephemeral_private, bob_static_public)`. If Bob's static private key were later compromised, an attacker could compute the same shared secret: `es = ECDH(bob_static_private, alice_ephemeral_public)`, since Alice's Ephemeral Public Key was transmitted in the clear. Forward secrecy is established in Act 2 when both parties compute `ee = ECDH(alice_ephemeral_private, bob_ephemeral_public)`. Since *both* ephemeral private keys are deleted after the handshake, an attacker would need at least one ephemeral private key to compute `ee` — which is no longer possible.

</details>

### Question 6: What would happen if Alice reused the same nonce with the same key for two different ChaCha20 encryptions?

- **A)** The decryption would fail because ChaCha20 detects nonce reuse automatically
- **B)** Both messages would decrypt to the same plaintext regardless of the original content
- **C)** An attacker could XOR the two ciphertexts together to learn information about both plaintexts, potentially recovering them entirely
- **D)** The MACs would be identical, but the ciphertexts would remain secure

<details>
  <summary>Answer</summary>

**C)** Using the same key and nonce produces an identical keystream. Since ChaCha20 encrypts by XORing the plaintext with this keystream, an attacker who obtains two ciphertexts encrypted with the same keystream can XOR them together. This cancels out the keystream, revealing the XOR of the two plaintexts. The attacker can then exploit patterns — for example, identical bytes at the same position produce `0x00` in the XOR result. Combined with frequency analysis or partial knowledge of one plaintext, the attacker can potentially recover both messages and even the keystream itself.

</details>

### Question 7: What is the primary purpose of the handshake hash throughout the Noise handshake?

- **A)** It stores a compressed version of all messages sent between Alice and Bob so they can be replayed if the connection drops
- **B)** It acts as a running transcript of the handshake, and is included as associated data in each MAC so that both parties can verify they are progressing through the protocol in sync
- **C)** It is used as the encryption key for each act's ChaCha20-Poly1305 operation
- **D)** It serves as a checksum that Alice and Bob compare directly at the end of the handshake to verify success

<details>
  <summary>Answer</summary>

**B)** The handshake hash acts as a "running transcript" that both Alice and Bob maintain independently. Each time a public key or ciphertext is exchanged, it gets mixed into the hash. When either party creates a MAC, they include the current handshake hash as associated data. This means the MAC doesn't just authenticate the key material — it also commits to the entire sequence of operations performed so far. If anything is out of sync (different Noise variant, wrong keys, tampered messages), the handshake hashes will diverge and the MAC verification will fail.

</details>

### Question 8: During key rotation, what serves as the Input Key Material (IKM) for the HKDF function?

- **A)** A zero-length input, similar to Act 3's final key derivation
- **B)** The original ECDH shared secret from the handshake phase
- **C)** The current encryption key (`sk` or `rk`)
- **D)** A freshly computed ephemeral ECDH shared secret between Alice and Bob

<details>
  <summary>Answer</summary>

**C)** The key rotation formula is `sck', sk' = HKDF(sck, sk)`, where the Send Chaining Key (`sck`) is the salt and the current Sending Key (`sk`) is the IKM. This is different from the handshake phase (where ECDH shared secrets are the IKM) and different from the final key derivation in Act 3 (where the IKM is an empty string). Each context in the protocol uses HKDF with different input key material, and understanding what feeds into each derivation is important for reasoning about the protocol's security properties.

</details>

### Question 9: Why does Lightning encrypt the 2-byte message length prefix before transmission?

- **A)** The length field contains a checksum that must be protected from tampering
- **B)** ChaCha20-Poly1305 requires all inputs — including metadata — to be encrypted before the algorithm can produce a valid MAC
- **C)** Without length encryption, an attacker could perform traffic analysis to learn message types and potentially track payments across the network
- **D)** The encrypted length doubles as a session identifier that Bob uses to match messages to the correct connection

<details>
  <summary>Answer</summary>

**C)** If the length were sent in plaintext, any eavesdropper could observe message sizes. Since different Lightning message types have characteristic sizes (e.g., `update_add_htlc` vs. `channel_reestablish`), an attacker could infer which operations are being performed and potentially track payments as they route across the network. Encrypting the length ensures that even the *size* of each message is hidden from observers, providing stronger traffic analysis resistance.

</details>

### Question 10: After the handshake, Alice computes `sk, rk = HKDF(ck, "")` and Bob computes `rk, sk = HKDF(ck, "")`. Why are the variable assignments reversed?

- **A)** Alice and Bob use different Chaining Keys at this point, so the HKDF outputs are different and must be assigned accordingly
- **B)** The HKDF function produces outputs in a different order depending on which party calls it
- **C)** The reversal ensures that Alice's Sending Key is the same key as Bob's Receiving Key (and vice versa), so messages encrypted by one party can be decrypted by the other
- **D)** Bob reverses the assignment to account for the different nonce ordering between sender and receiver

<details>
  <summary>Answer</summary>

**C)** The HKDF function is deterministic — given the same Chaining Key and empty input, it always produces the same two 32-byte keys in the same order. The "reversal" is simply how Alice and Bob *label* those keys. The first output becomes Alice's Sending Key and Bob's Receiving Key. The second output becomes Bob's Sending Key and Alice's Receiving Key. This creates **complementary keys**: when Alice encrypts a message with her `sk`, Bob decrypts it with his `rk` — which is the exact same key. This elegant assignment ensures bidirectional encrypted communication without any additional key exchange.

</details>
