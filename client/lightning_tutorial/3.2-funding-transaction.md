# Funding Transaction

## Overview

The **funding transaction** is the on-chain Bitcoin transaction that opens a Lightning channel. It takes bitcoin from the channel opener's wallet and sends it to the 2-of-2 multisig P2WSH output that both parties control. Once this transaction is confirmed on the blockchain, the channel is open and the parties can begin exchanging commitment transactions off-chain.

## Transaction Structure

BOLT 3 specifies that the funding transaction must follow these rules:

- **Version**: Must be `2` (to enable relative timelocks via BIP68/BIP112)
- **Inputs**: At least one input from the channel opener's wallet
- **Outputs**: Must include the funding output (P2WSH of the multisig script). May include a change output
- **Locktime**: Set to `0` for the funding transaction

### Why Version 2?

Bitcoin transaction version 2 enables `nSequence`-based relative timelocks, which are required by `OP_CHECKSEQUENCEVERIFY` (CSV) in commitment transaction scripts. While the funding transaction itself does not use CSV, the commitment transactions that spend it do. Using version 2 from the start ensures compatibility.

## Building the Transaction

Let us walk through the construction step by step.

### Step 1: Create the Funding Output

The funding output is a P2WSH wrapping of the 2-of-2 multisig script:

```python
import hashlib

def create_funding_output(local_pubkey: bytes, remote_pubkey: bytes, amount_sat: int):
    # Sort keys lexicographically
    keys = sorted([local_pubkey, remote_pubkey])

    # Build multisig script
    funding_script = (
        bytes([0x52, 0x21]) + keys[0] +
        bytes([0x21]) + keys[1] +
        bytes([0x52, 0xae])
    )

    # P2WSH: OP_0 <SHA256(script)>
    script_hash = hashlib.sha256(funding_script).digest()
    output_script = bytes([0x00, 0x20]) + script_hash

    return output_script, amount_sat, funding_script
```

### Step 2: Serialize the Transaction

A Bitcoin transaction is serialized in a specific binary format. For a SegWit transaction, the format is:

```text
[version:4] [marker:1] [flag:1] [input_count:varint] [inputs] [output_count:varint] [outputs] [witness] [locktime:4]
```

The key fields for the funding transaction:

| Field | Value | Size |
|-------|-------|------|
| version | `0x02000000` | 4 bytes (little-endian) |
| marker | `0x00` | 1 byte (SegWit marker) |
| flag | `0x01` | 1 byte (SegWit flag) |
| locktime | `0x00000000` | 4 bytes (little-endian) |

### Step 3: Encode Inputs

Each input references a previous transaction output (UTXO) by its txid and output index:

```text
[txid:32] [vout:4] [script_length:varint] [script_sig] [sequence:4]
```

For SegWit inputs, the `script_sig` is empty (length 0) because the signature data goes in the witness section instead. The `sequence` is typically set to `0xfffffffd` to signal RBF (Replace-by-Fee) compatibility.

### Step 4: Encode Outputs

Each output specifies an amount and a locking script:

```text
[amount:8] [script_length:varint] [script]
```

The amount is in **satoshis** (1 BTC = 100,000,000 satoshis), encoded as an 8-byte little-endian integer.

For the funding output, the script is the 34-byte P2WSH output:

```text
[amount:8] [0x22] [0x00 0x20 <32-byte-hash>]
```

## Computing the Funding TxID

The funding transaction's **txid** is critical because it is referenced by every commitment transaction that spends from it. The txid is computed as the double-SHA256 of the **non-witness** serialization:

```python
def compute_txid(tx_bytes_no_witness: bytes) -> bytes:
    """Compute txid from non-witness serialization (result is reversed)."""
    return hashlib.sha256(hashlib.sha256(tx_bytes_no_witness).digest()).digest()[::-1]
```

Note that the txid uses the serialization **without** the SegWit marker, flag, and witness data. This is the key property of SegWit: the witness is not part of the txid, which prevents transaction malleability.

### Why Transaction Malleability Matters

Before SegWit, signatures were part of the txid computation. This meant a third party could modify the signature format (without invalidating it) and change the txid. For Lightning, this would be catastrophic: commitment transactions reference the funding txid, so if the txid changed after broadcast, all pre-signed commitment transactions would become invalid.

SegWit solved this by moving signatures to the witness section, which is excluded from the txid computation. This is why Lightning channels require SegWit -- it is not just an optimization, it is a fundamental requirement.

## The Funding Output Point

The combination of the funding txid and the output index (vout) is called the **funding output point** or **channel point**. It uniquely identifies the channel on the blockchain:

```text
funding_outpoint = (funding_txid, funding_vout)
```

This outpoint is used as the input for every commitment transaction. Both parties must agree on it before the channel can be used.

## A Complete Example

Here is the complete flow for creating a funding transaction:

1. The channel opener selects UTXOs from their wallet to fund the channel.
2. They construct the P2WSH output from the sorted multisig script.
3. They create a change output for any excess funds.
4. They serialize the transaction with version 2, locktime 0.
5. They compute the txid from the non-witness serialization.
6. Both parties exchange the funding txid and begin constructing commitment transactions.

The funding transaction is not broadcast until both parties have exchanged their first commitment transaction signatures. This ensures that neither party can lose funds if the counterparty disappears -- they can always broadcast the pre-signed commitment transaction to recover their balance.
