# Payment Channels Overview

## The Bitcoin Scaling Problem

Bitcoin's blockchain can process roughly 7 transactions per second. For comparison, Visa handles around 1,700 transactions per second on average. If Bitcoin is going to serve as a global payment system, it needs a way to handle far more transactions without making every single one compete for block space.

One approach is to make blocks bigger, but this has trade-offs: larger blocks take longer to propagate, increase storage requirements, and make it harder to run a full node. The Lightning Network takes a different approach entirely.

## What Is a Payment Channel?

A **payment channel** is a mechanism that allows two parties to make an unlimited number of payments between each other using only **two** on-chain transactions:

1. A **funding transaction** that opens the channel by locking funds into a shared account.
2. A **closing transaction** that settles the final balances back on-chain.

Everything in between happens **off-chain** -- just the two parties exchanging signed messages. No miners, no block confirmations, no fees for each individual payment. A channel could process millions of payments and the blockchain would only ever see the open and close.

### A Simple Example

Imagine Alice and Bob open a channel with 1 BTC total (0.5 BTC each):

```text
Opening State:  Alice: 0.5 BTC  |  Bob: 0.5 BTC

Alice pays Bob 0.1 BTC:
  State 1:      Alice: 0.4 BTC  |  Bob: 0.6 BTC

Bob pays Alice 0.05 BTC:
  State 2:      Alice: 0.45 BTC |  Bob: 0.55 BTC

Alice pays Bob 0.2 BTC:
  State 3:      Alice: 0.25 BTC |  Bob: 0.75 BTC
```

Three payments happened, but the blockchain sees none of them. When they eventually close the channel, the final state (Alice: 0.25, Bob: 0.75) is settled on-chain in a single transaction.

## The Channel Lifecycle

Every payment channel goes through three phases:

### 1. Fund

The channel opener creates a **funding transaction** that sends bitcoin to a 2-of-2 multisig address controlled by both parties. This is the only transaction that touches the blockchain during normal operation.

Both parties must agree on the channel parameters before the funding transaction is broadcast. They exchange public keys, negotiate the channel capacity, and pre-sign the first commitment transactions.

### 2. Update

Once the channel is open, the parties can update the balance split by creating new **commitment transactions**. Each commitment transaction is a valid Bitcoin transaction that could be broadcast at any time, but during normal operation, neither party broadcasts -- they just keep updating.

The critical challenge is: how do you prevent someone from broadcasting an old commitment transaction that gives them more money? This is where the **revocation mechanism** comes in, which we will explore in detail later.

### 3. Close

The channel can be closed in two ways:

- **Cooperative close**: Both parties agree on the final balances and create a simple closing transaction together. This is the cheapest and fastest option.
- **Unilateral close**: One party broadcasts their latest commitment transaction. This is more expensive (extra outputs, time delays) but ensures that either party can always get their funds back, even if the other party disappears.

## Commitment Transactions

The heart of a payment channel is the **commitment transaction**. This is a pre-signed Bitcoin transaction that spends the funding output and distributes the funds according to the current balance split.

### Why Asymmetric?

Here is the key insight that makes Lightning channels work: **each party holds a different version of the commitment transaction**. Alice's version is different from Bob's version, even though they represent the same balance split.

In Alice's commitment transaction:
- **Alice's output** (to_local) is encumbered with a time delay and is revocable.
- **Bob's output** (to_remote) is immediately spendable by Bob.

In Bob's commitment transaction:
- **Bob's output** (to_local) is encumbered with a time delay and is revocable.
- **Alice's output** (to_remote) is immediately spendable by Alice.

The rule is: **the broadcaster's own output is always delayed and revocable**. This means the person who publishes the transaction takes on the risk. If they published an old (revoked) state, the counterparty has time (during the delay period) to use the revocation key and claim everything.

This asymmetry is what makes the penalty mechanism possible. If both parties held identical commitment transactions, there would be no way to distinguish who broadcast it, and therefore no way to penalize cheating.

## From Channels to a Network

Payment channels between two parties are useful, but the real power comes from **connecting channels into a network**. If Alice has a channel with Bob, and Bob has a channel with Carol, then Alice can pay Carol by routing through Bob -- without Alice and Carol ever opening a direct channel.

This routing is made possible by **HTLCs (Hashed Time-Locked Contracts)**, which we will build in the later sections of this course. HTLCs ensure that multi-hop payments are atomic: either the entire payment succeeds across all hops, or it fails and everyone gets their money back.

## What We Need to Build

To implement a working payment channel, we need several pieces:

| Component | Purpose |
|-----------|---------|
| Key management | Derive all the cryptographic keys for channel operations |
| Funding script | Create the 2-of-2 multisig that locks funds |
| Funding transaction | Build and sign the on-chain transaction that opens the channel |
| Revocation keys | Enable the penalty mechanism for old states |
| Commitment scripts | Define the spending conditions for each output |
| Commitment transaction | Assemble the full transaction with proper structure |
| HTLC scripts | Enable conditional payments for routing |

Each of these components is specified in detail in the BOLT specifications (the Lightning Network's protocol documents). We will implement them one by one, validating against the official BOLT 3 test vectors at each step.
