# Failure Codes

Now that you can construct and wrap error packets, let's look at the bigger picture: what kinds of failures can occur, how they're classified, and how Alice unwraps the error to learn what happened.

## The Failure Code Flag System

BOLT 4 defines a structured system for failure codes. Each code is a 2-byte integer where specific bits act as flags that describe the nature of the failure:

| Bit | Flag | Meaning |
|-----|------|---------|
| 15 (0x8000) | `BADONION` | The onion packet itself is malformed |
| 14 (0x4000) | `PERM` | Permanent failure (don't retry this route) |
| 13 (0x2000) | `NODE` | Problem with a specific node |
| 12 (0x1000) | `UPDATE` | Contains a `channel_update` message |

These flags combine with a base error number to form the full failure code. For example, `temporary_channel_failure` is `0x1007`: the `UPDATE` flag (0x1000) is set (indicating the error includes channel information), combined with base code 7.

The flags tell Alice what kind of action to take without even looking at the specific error:

- **`BADONION`**: Something is wrong with the onion packet. A node couldn't parse it, or the HMAC failed. Alice should check her packet construction.
- **`PERM`**: This failure won't resolve itself. Don't retry this route; find an alternative.
- **`NODE`**: The problem is with a node, not a channel. Alice might avoid this node entirely.
- **`UPDATE`**: The error contains a fresh `channel_update` message. Alice should update her routing graph with the new fee or capacity information, then retry.

## Common Failure Types

Here are the failure codes you'll encounter most often in practice:

**Liquidity failures:**
- `temporary_channel_failure` (0x1007): The most common error. The forwarding node doesn't have enough outbound liquidity in the specified channel. The `UPDATE` flag means Alice should refresh her view of this channel.

**Fee and timelock issues:**
- `fee_insufficient` (0x100C): The forwarding fee in the onion is lower than the node's current policy. Alice's routing graph may be stale.
- `incorrect_cltv_expiry` (0x100D): The CLTV delta doesn't meet the node's requirements.
- `expiry_too_soon` (0x100E): The outgoing CLTV is too close to the current block height. The node doesn't have enough time margin to safely forward.

**Routing failures:**
- `unknown_next_peer` (0x400A): The forwarding node doesn't have a channel to the specified next hop. The channel may have been closed.

**Onion failures:**
- `invalid_onion_hmac` (0x8002): HMAC verification failed. The `BADONION` flag means the onion is corrupted.
- `invalid_onion_key` (0x8003): The ephemeral public key is invalid.

## Stuck Payments

There's one failure scenario that error packets can't help with: **stuck payments**. A stuck payment occurs when an intermediate hop holds an HTLC without either forwarding it or sending back an error. The hop simply goes silent.

Consider this scenario: Alice sends a payment through Bob and Carol to Dave. Carol receives the HTLC from Bob and the onion tells her to forward to Dave. But Carol's node crashes, or her network connection drops, or she's running buggy software that hangs. Carol neither forwards the HTLC to Dave nor sends an error back to Bob.

Now everyone is stuck:
- **Bob** committed funds to Carol via an HTLC, and he's waiting for either a preimage (success) or an error (failure)
- **Alice** committed funds to Bob and is also waiting
- Neither Alice nor Bob can unilaterally cancel the payment

The only resolution is **time**. Each HTLC has a CLTV timeout. When the timeout expires, the HTLC can be resolved on-chain: Bob can broadcast his commitment transaction and claim the timed-out HTLC output after the timelock. But until that timeout arrives, the funds are locked.

This is one of Lightning's known pain points. In the worst case, a stuck payment can lock funds for hours or even days (depending on the CLTV delta). Alice can't retry through a different route without risking a double payment if Carol eventually wakes up and forwards the original. Some mitigations exist (like probing routes before paying, or using multi-part payments to limit exposure), but the fundamental issue remains: if a hop goes silent, you have to wait.

<details>
  <summary>Can Alice force-cancel a stuck payment?</summary>

No. Once Alice has committed the HTLC with Bob (via the commitment dance), she can't unilaterally remove it. The HTLC is part of the signed commitment transaction. Alice would need Bob's cooperation to update the channel state and remove the HTLC, but Bob won't agree because he has a committed HTLC with Carol that he can't cancel either.

Alice's only option is to wait for the CLTV timeout. After the timeout, she can either cooperatively resolve the HTLC with Bob (if Bob is online and responsive) or go on-chain by broadcasting her commitment transaction and claiming the timed-out HTLC.

## Looking Ahead: PTLCs

One limitation of the current HTLC design is that the payment hash is the same at every hop. If Bob and Dave collude, they can correlate their HTLCs (both use the same payment hash) and determine that they're part of the same payment. This leaks information about the route.

**Point Time-Locked Contracts (PTLCs)** are a proposed improvement that replaces hash-based locks with adaptor signatures. Instead of a single payment hash, each hop would use a different "point" on the elliptic curve, unlinkable to points at other hops. This would break the correlation between hops and significantly improve payment privacy.

PTLCs require Schnorr signatures (available via Bitcoin's Taproot upgrade), and several Lightning implementations are working toward adopting them. The error handling mechanism we've built would work essentially the same way with PTLCs; the main change is in how HTLCs are constructed and fulfilled, not in how errors propagate.
