# Forwarding & Validation

Bob can now peel a layer and produce the packet for the next hop. But peeling is only part of the job. Before he forwards anything, Bob needs to verify the packet is authentic, parse his TLV instructions, sanity-check the forwarding parameters, and decide whether to proceed or fail. This chapter covers that full loop.

By the end you'll have written `OnionForwarder.process`, a function that takes an inbound packet plus context (the inbound HTLC's amount and CLTV) and returns either a `ForwardInstruction` (Bob is forwarding) or a `FinalDelivery` (Bob is the destination) or a rejection (something's wrong).

## The order of operations matters

A subtle but important pattern: verify integrity *before* doing any decryption or parsing. If we processed an attacker-supplied packet's contents before checking its HMAC, a malicious peer could feed us malformed bytes that exploit some bug in our parser before we'd had a chance to reject the packet outright.

The right order is:

1. **Length and version checks.** If the packet isn't 1,366 bytes or the version byte isn't `0x00`, reject immediately.
2. **HMAC verification.** Recompute `HMAC-SHA256(mu_i, hop_payloads)` and compare against the packet's HMAC field. Constant-time comparison if you can; if it fails, reject with an `invalid_onion_hmac` error.
3. **Peel.** This is the chapter-8 algorithm: derive `rho`, decrypt with extended keystream, extract the slot. Do this only after HMAC verification passes.
4. **Parse the TLV payload.** Determine if Bob is a forwarder (has `short_channel_id`) or the destination (has `payment_data` instead).
5. **Sanity-check fees and timelocks.** If forwarding, make sure the incoming HTLC's amount and CLTV are consistent with what the TLV asks Bob to forward.
6. **Forward or settle.** If everything checks out, hand the outgoing HTLC to the appropriate channel; if not, fail with the appropriate error code.

## Why we verify the HMAC over encrypted bytes

Step 2 verifies the HMAC against `hop_payloads` *as they appear on the wire*, before any decryption. This is the encrypt-then-MAC pattern from the Noise course's [crypto primitives chapter](/noise-tutorial/crypto-primitives). A few reasons it's the right design:

- **No untrusted data hits the parser.** If an attacker tampered with even one byte of `hop_payloads`, the HMAC will fail and we reject before we run our TLV parser. The parser is a piece of code that has to handle adversarial inputs; reducing the surface it sees is a defensive win.
- **No keystream extraction work for bogus packets.** Generating 2,600 bytes of ChaCha20 keystream isn't free. By gating on the HMAC first, we avoid spending CPU on packets we're going to discard.
- **Clear failure mode.** A peer who modifies a packet between us and the upstream hop produces an HMAC failure. We can rate-limit or disconnect the peer cleanly. If we caught the failure later (e.g., after parsing), the diagnostic would be murkier.

## Parsing the TLV

The peel step gave us back `payload_bytes`, a chunk of bigsize-prefixed TLV records. Inside it are the four kinds of fields we covered in chapter 2:

- Type 2: `amt_to_forward` (msat).
- Type 4: `outgoing_cltv_value` (block height).
- Type 6: `short_channel_id` (intermediate hops only).
- Type 8: `payment_data` (final hop only): payment_secret + total_msat.

Bob walks the TLV records in order (TLV requires monotonically increasing types) and extracts whatever is present. If `short_channel_id` is there, Bob is forwarding. If `payment_data` is there but `short_channel_id` is not, Bob is the destination. Both being present is malformed; neither being present is also malformed.

For our exercise, we'll keep the parser short and ignore unknown TLV types (which BOLT 4 mandates: forward-compatibility through ignoring unknown even types, rejecting unknown odd types).

## Fee and CLTV sanity checks

Bob's incoming HTLC carries an amount and a CLTV. The TLV in his payload says how much to forward and what CLTV to use on the outgoing HTLC. Bob has to confirm:

- **Incoming amount is enough.** The incoming amount must cover `amt_to_forward` plus Bob's own fee. If Alice computed everything correctly back in chapter 2's backward math, this will hold. If she didn't (or if a previous hop forwarded less than instructed), Bob fails with `amount_below_minimum` or `fee_insufficient`.
- **Incoming CLTV gives enough cushion.** The incoming CLTV must be greater than `outgoing_cltv_value` by at least Bob's published CLTV delta. If not, Bob fails with `incorrect_cltv_expiry` or `expiry_too_soon`.
- **Outgoing CLTV is in the future.** Even if the delta math works, the outgoing CLTV has to be a block height that hasn't passed yet, and ideally far enough in the future that downstream hops have time to settle.

If all checks pass, Bob hands the outgoing HTLC to the channel identified by `short_channel_id`, with amount `amt_to_forward` and timelock `outgoing_cltv_value`.

For the destination (payment_data present), the validation is different: confirm the `payment_secret` matches the invoice, and the total amount is what the receiver asked for. We won't dive into invoice validation here; that's a different protocol layer.

<details>
<summary>What if Bob's downstream channel is offline or has insufficient liquidity?</summary>

Bob fails the HTLC with `temporary_channel_failure`. The error onion (chapter 10) returns to Alice, who tries a different route on retry. This is one of the most common failure modes on real Lightning networks: pathfinding can't always know with certainty whether a particular channel has the liquidity in the right direction at the moment of payment, so retries are normal.

`temporary_channel_failure` is signaled by including the `UPDATE` flag (0x1000) and a `channel_update` message in the failure data, telling Alice the current state of the channel so her pathfinder can avoid stale information.

## What `process` returns

We'll model the result with three cases:

- **`ForwardInstruction(next_packet, short_channel_id, amt_to_forward, outgoing_cltv, ss)`**: Bob should send `next_packet` to the channel identified by `short_channel_id`, with the given amount and CLTV. The `ss` (shared secret) is kept for use on the error path if something downstream goes wrong.
- **`FinalDelivery(amt_to_forward, outgoing_cltv, payment_data, ss)`**: Bob is the destination. He compares `payment_data` against his pending invoice and either settles (revealing the preimage) or fails the HTLC.
- **`Rejection(code, ss)`**: Something's wrong. Build an error onion using the appropriate failure code and `um` key. (We'll do this in chapter 10.)

The `ss` propagates through every result type because the error path always needs it: either to wrap an error from this hop, or (for forwarders) to unwrap an inbound error from downstream and re-wrap it for the upstream peer.

## The exercise

Implement `OnionForwarder.process`. The tests will exercise:

- HMAC verification: feeds in a tampered packet, expects `Rejection`.
- Forwarding path: feeds in a valid packet for an intermediate hop, expects `ForwardInstruction` with the right values.
- Destination path: feeds in a packet whose final TLV omits `short_channel_id`, expects `FinalDelivery`.

Keep the fee/CLTV checks light in this exercise; the test won't simulate full economic validation. The point is to wire together the steps cleanly.
