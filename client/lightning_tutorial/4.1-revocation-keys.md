# Revocation Keys

## The Revocation Mechanism

Revocation is the core enforcement mechanism in Lightning channels. When Alice and Bob update their channel state, the old state must be invalidated -- otherwise, Alice could broadcast an old commitment transaction that gives her more money than she currently has.

Bitcoin itself has no concept of "revoking" a transaction. Any valid, signed transaction can be broadcast at any time. So Lightning uses a clever cryptographic construction: when a state is revoked, the revoking party shares a secret that allows the counterparty to **claim all channel funds** if the old state is ever broadcast.

This is the penalty mechanism we described in the fairness protocol chapter. The revocation key is the tool that makes it work.

## Two-Component Key Derivation

The revocation public key is derived from **two** components:

1. The **revocation basepoint** -- a static public key that belongs to the counterparty (the non-broadcaster).
2. The **per-commitment point** -- a public key derived from the per-commitment secret for this specific commitment number.

Neither component alone is sufficient to spend the revocation path. This is the key insight: the revocation key can only be fully constructed when both parties' secrets are combined.

## The Formula

BOLT 3 defines the revocation public key as:

```text
revocation_pubkey = revocation_basepoint * SHA256(revocation_basepoint || per_commitment_point)
                  + per_commitment_point * SHA256(per_commitment_point || revocation_basepoint)
```

Where:
- `||` denotes concatenation
- `*` denotes elliptic curve point multiplication (scalar times point)
- `+` denotes elliptic curve point addition

In Python, using elliptic curve operations:

```python
import hashlib

def derive_revocation_pubkey(revocation_basepoint: bytes, per_commitment_point: bytes) -> bytes:
    # First component: revocation_basepoint * SHA256(revocation_basepoint || per_commitment_point)
    hash1 = hashlib.sha256(revocation_basepoint + per_commitment_point).digest()
    point1 = ec_multiply(revocation_basepoint, hash1)

    # Second component: per_commitment_point * SHA256(per_commitment_point || revocation_basepoint)
    hash2 = hashlib.sha256(per_commitment_point + revocation_basepoint).digest()
    point2 = ec_multiply(per_commitment_point, hash2)

    # Revocation pubkey = point1 + point2
    return ec_add(point1, point2)
```

## Why This Construction?

The two-component construction has an elegant security property:

### Before Revocation

Before a state is revoked, only the counterparty knows the `revocation_basepoint_secret` and only the local party knows the `per_commitment_secret`. Neither can compute the full revocation private key alone:

```text
revocation_privkey = revocation_basepoint_secret * SHA256(revocation_basepoint || per_commitment_point)
                   + per_commitment_secret * SHA256(per_commitment_point || revocation_basepoint)
```

Both secret scalars are needed. The counterparty has `revocation_basepoint_secret` but not `per_commitment_secret`. The local party has `per_commitment_secret` but not `revocation_basepoint_secret`.

### After Revocation

When the local party revokes the old state, they reveal their `per_commitment_secret` to the counterparty. Now the counterparty has **both** components:

- Their own `revocation_basepoint_secret` (they always had this)
- The revealed `per_commitment_secret` (just received)

With both secrets, the counterparty can compute the full `revocation_privkey` and spend the revocation path if the old commitment is broadcast.

## The Revocation Private Key

The corresponding private key follows the same structure but with scalar arithmetic:

```text
revocation_privkey = revocation_basepoint_secret * SHA256(revocation_basepoint || per_commitment_point)
                   + per_commitment_secret * SHA256(per_commitment_point || revocation_basepoint)
```

All operations are modular arithmetic over the secp256k1 curve order:

```python
def derive_revocation_privkey(
    revocation_basepoint_secret: bytes,
    per_commitment_secret: bytes,
    revocation_basepoint: bytes,
    per_commitment_point: bytes
) -> bytes:
    CURVE_ORDER = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141

    hash1 = hashlib.sha256(revocation_basepoint + per_commitment_point).digest()
    hash2 = hashlib.sha256(per_commitment_point + revocation_basepoint).digest()

    term1 = (int.from_bytes(revocation_basepoint_secret, 'big')
             * int.from_bytes(hash1, 'big')) % CURVE_ORDER
    term2 = (int.from_bytes(per_commitment_secret, 'big')
             * int.from_bytes(hash2, 'big')) % CURVE_ORDER

    privkey = (term1 + term2) % CURVE_ORDER
    return privkey.to_bytes(32, 'big')
```

## How It Fits Into the Channel

Here is the timeline for revocation keys in a channel:

### Channel Open (Commitment 0)

1. Alice generates `per_commitment_secret_0` from her commitment seed.
2. Alice computes `per_commitment_point_0 = per_commitment_secret_0 * G`.
3. Alice sends `per_commitment_point_0` to Bob.
4. Bob uses `per_commitment_point_0` and his own `revocation_basepoint` to compute `revocation_pubkey_0`.
5. This revocation pubkey goes into Alice's commitment transaction #0.

### State Update (Commitment 0 -> 1)

1. Alice generates `per_commitment_secret_1` and `per_commitment_point_1`.
2. Alice sends `per_commitment_point_1` to Bob (for the new state).
3. Alice reveals `per_commitment_secret_0` to Bob (revoking the old state).
4. Bob can now derive `revocation_privkey_0` -- the key to penalize Alice if she broadcasts commitment #0.
5. Bob constructs new commitment transaction #1 using `per_commitment_point_1`.

### If Alice Cheats

If Alice broadcasts the revoked commitment #0:
1. Bob sees the old commitment on-chain.
2. Bob has `per_commitment_secret_0` (Alice revealed it) and his own `revocation_basepoint_secret`.
3. Bob computes `revocation_privkey_0`.
4. Bob sweeps Alice's output using the revocation key, claiming all funds.

The CSV delay on Alice's output gives Bob time to detect the cheating and respond.
