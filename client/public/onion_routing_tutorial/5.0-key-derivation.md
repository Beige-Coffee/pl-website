# Key Derivation

Chapter 3 ended with Alice (and each forwarder) holding a 32-byte shared secret per hop. That secret is enough material to seed the rest of what we're building, but we don't want to use it directly for any single cryptographic job. We're going to expand it into a small family of named keys, each with a specific purpose.

This is going to be a short chapter with a small but important coding exercise. By the end you'll have all the keys you need to actually start building the onion packet itself.

## Why one shared secret isn't enough

It's tempting to think: we already have 32 bytes of shared randomness, can't we just use it as the encryption key and call it a day? In theory, you can. In practice, reusing a single key for multiple cryptographic jobs is one of the easier ways to wreck a protocol.

A few specific reasons we want different keys for different jobs:

- **Encryption and authentication should be independent.** If the same key is used for both ChaCha20 (encrypting payload bytes) and HMAC (authenticating the same bytes), an attacker who recovers either keystream or HMAC values can sometimes reuse them to cross-attack the other primitive. AEAD constructions like ChaCha20-Poly1305 internally derive separate keys; we want the same separation here.
- **Forward and backward directions need different keys.** An onion packet flows forward from Alice to Dave; an error packet flows backward from a failing hop to Alice. Using the same HMAC key for both directions would let a forwarder forge fake errors that look authentic. Different keys = different domains.
- **The filler is a sender-side computation.** Alice generates filler bytes during packet construction (we'll see why in chapter 6). Forwarders never compute filler. Giving Alice's filler computation its own key keeps it separate from the keys that actually flow through the packet.
- **Compromise containment.** If one key leaks somehow (a bad random number generator, a side-channel attack, anything), we want the others to remain safe. Independent keys mean a leak in one doesn't cascade.

We need **five** distinct 32-byte keys per hop, each domain-separated from the others.

## The construction: HMAC-SHA256 with a named purpose

Lightning's BOLT 4 doesn't use HKDF directly here. Instead, it uses a deliberately simple construction:

```
generate_key(name, shared_secret) = HMAC-SHA256(key=name, msg=shared_secret)
```

Where `name` is a short ASCII label like `"rho"`, `"mu"`, `"um"`, `"pad"`, or `"ammag"`. The result is 32 bytes (the SHA256 output size). Five different names produce five independent-looking 32-byte keys from the same shared secret.

<hmac-recap></hmac-recap>

This works because HMAC-SHA256 has the property that small changes in either the key *or* the message input produce outputs that look statistically uncorrelated. Using the same `shared_secret` with five different name strings produces five keys that are, for all practical purposes, independent. If we needed more keys later, we could add more names. (The BOLT 4 spec defines exactly these five.)

If you want the deeper derivation of why HMAC has these properties, the [HKDF chapter](/noise-tutorial/crypto-primitives) of the Noise course walks through the formal construction. For our purposes here, we trust HMAC and focus on what we use the keys for.

## The five keys, and what they do

Take a look at the visual below. One shared secret in, five domain-separated 32-byte keys out, each with a specific role in the protocol:

<kdf-pipeline></kdf-pipeline>

Here's the rundown:

**`rho`** is the stream cipher key for the hop payload area. When Alice wraps a layer of the onion, she generates a 1300-byte ChaCha20 keystream from `rho_i` and XORs it onto the packet's hop-data section. When the forwarder peels the layer, it generates the same keystream and XORs it back off, revealing the inner packet. (We'll dig into this in chapter 7.)

**`mu`** is the HMAC key for the packet's authentication tag. Each layer of the onion includes a 32-byte HMAC computed over the next layer's contents using `mu_i`. The forwarder verifies this HMAC before doing anything else; if it fails, the packet has been tampered with and the forwarder rejects it.

**`um`** is the HMAC key for *error* packets. If hop `i` decides to fail the payment (couldn't find a route, channel is full, expired timelock, anything), it constructs an encrypted error message authenticated with `um_i`. Alice can verify the HMAC on the way back by recomputing it from `um_i`.

**`pad`** is used only by the sender, and only at the start of construction. It seeds a deterministic 1300-byte filler that pads out the packet to the right length on the very first wrapping step. (Why we need filler at all is a chapter-6 question; for now, just know `pad_i` is not transmitted and not used by forwarders.)

**`ammag`** is the stream cipher key for *error* packets, the backward-direction analog of `rho`. When a hop builds or forwards an error, it XORs an `ammag`-derived ChaCha20 keystream onto the error payload to encrypt it. Alice peels these layers off in order on the return path.

> **Fun Fact!** BOLT 4's choice of names: `rho` and `mu` come from typical cryptographic notation for stream cipher and MAC keys. `ammag` is `gamma` reversed, the backward-direction counterpart of an encryption key. `um` is `mu` reversed, the backward-direction MAC key. `pad` does what it says.

## Independence in practice

The reason this construction is safe is that recovering one key does not help an attacker recover the others.

Suppose an attacker somehow obtains `rho_i` (maybe they exploited a weakness in our ChaCha20 implementation). They want to use that to forge an HMAC, which means they need `mu_i`. They have HMAC-SHA256(`"rho"`, `ss_i`) and they want HMAC-SHA256(`"mu"`, `ss_i`). Without `ss_i`, the only way to derive `mu_i` from `rho_i` would be to invert HMAC-SHA256, which is what HMAC is built to prevent.

The same argument applies pairwise to all five keys. Each one is independent of the others because the only common input (`ss_i`) is hidden behind HMAC's one-way property.

<details>
<summary>Why HMAC and not a single hash like SHA256?</summary>

You could imagine writing this as `SHA256(name || shared_secret)`. It would even work in this specific case. The reason BOLT 4 uses HMAC is that HMAC has a clearer security argument when the inputs are attacker-controlled in the future. If a future BOLT extension introduced a new key whose name happened to start with bytes that look like a valid intermediate hash state, a length-extension attack on plain SHA256 could cause subtle problems. HMAC sidesteps this by mixing the key in with two different padding constants. In short: HMAC is the right tool when you're using a hash as a PRF, even when you don't currently have an attacker model that exploits the difference.

</details>

## Per-hop, per-payment

Worth flagging explicitly: every hop has its own set of five keys, derived from that hop's shared secret. Bob's `rho` is different from Charlie's `rho`, which is different from Dave's `rho`. And every payment has fresh shared secrets (because Alice uses a fresh session key per payment), so every payment has fresh keys all the way down.

For our 3-hop route, that's 3 hops × 5 keys = **15 distinct 32-byte keys per payment**. Take a look at the matrix below: every cell is one of those 15 keys, color-coded by purpose. Hover any cell to see exactly what that key does.
