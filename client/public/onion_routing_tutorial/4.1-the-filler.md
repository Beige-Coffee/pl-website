# The Filler

In the previous chapter, we identified three problems with a naive onion construction: shrinking packets, no layered encryption, and no authentication. The Sphinx protocol solves all three. But its fixed-size property introduces a subtle challenge.

## The Shifting Problem

When Alice constructs the onion, she works from the **innermost hop** (Dave) outward to the **outermost hop** (Bob). At each step, she:

1. **Shifts** the existing payload to the right to make room for the new hop's data
2. **Inserts** the new hop's payload at the front
3. **Encrypts** the entire 1,300-byte buffer

Here's the problem: when she shifts the payload right, bytes "fall off" the right end. The buffer is always exactly 1,300 bytes, so any rightward shift pushes some bytes past the boundary. Those bytes are lost.

When the **final hop** (Dave) later peels all the layers and decrypts, the trailing bytes of his 1,300-byte payload need to contain valid data. But those are exactly the positions where bytes were lost during construction. Without intervention, Dave would see random garbage at the end of his payload, and his HMAC verification would fail.

## What the Filler Does

The **filler** is a pre-computed byte sequence that Alice generates before constructing the onion. After encrypting the innermost layer (Dave's), she overwrites the trailing bytes of the payload with the filler. These filler bytes are specifically crafted so that after Dave decrypts all the layers above him, the trailing positions contain valid zeros (or whatever the protocol expects).

Think of it this way: the filler "compensates" for the bytes that fell off during the rightward shifts. It's a deterministic correction factor that ensures the math works out at the end.

<details>
  <summary>Why does the filler use XOR with the rho stream?</summary>

The filler is built by simulating the encryption that each intermediate hop will apply when peeling its layer. During peeling, each hop XOR-decrypts the payload with its rho cipher stream. The filler pre-applies these same XOR operations in the correct order so that the cumulative effect at the final hop produces the right trailing bytes. Since XOR is its own inverse (applying it twice cancels out), the filler essentially "pre-cancels" the encryption that intermediate hops will add, leaving the correct values after all layers are peeled.
</details>

## The Filler Algorithm

Alice generates the filler by iterating through each hop **except the last** (since the last hop's shift doesn't cause any data loss, as it's the first shift during construction). For each hop, she:

1. Generates the rho cipher stream for that hop (using the same `generate_cipher_stream` helper that the wrapping step will use)
2. Extends the filler by the number of bytes that this hop's shift pushes off the right end
3. XOR-encrypts the growing filler with the appropriate slice of the cipher stream

The "shift size" for each hop is `len(hop_payload) + 32`. The extra 32 bytes account for the per-hop HMAC that's inserted alongside each payload during wrapping.

The "appropriate offset" is `1300 - len(filler)`, because the filler corresponds to the trailing bytes of the 1,300-byte buffer.
