# Shared Secrets per Hop

We left chapter 2 with Alice holding three little hop payloads, one per forwarder, plus a final-destination payload for Dave. Each one is an unencrypted blob of TLV records. If she just sent Bob's payload to Bob in the clear, Bob would learn nothing he shouldn't, but a passive eavesdropper between Alice and Bob would. Worse, Alice has no way to even *deliver* Charlie's and Dave's payloads, since her packet has to pass through Bob to get there, and we don't want Bob reading them.

What Alice needs is a way to encrypt each hop's payload so that **only that hop** can decrypt it. Crucially, she has to do this without anyone's private keys (she has her own keys, but no one else's). The cryptographic trick that makes this work is the same one we used to bootstrap the Noise handshake: **ECDH**.

Buckle up.

## A quick refresher on ECDH

If you completed the Noise course, you already know ECDH. Take a look at the visual below for a two-second reminder.

<ecdh-recap></ecdh-recap>

The two-sentence version: given a private key `a` and a corresponding public key `A = a·G`, plus another party's keypair `(b, B = b·G)`, both parties can compute the same shared point `a·B = b·A = ab·G` from each other's public key plus their own private key. Hash that point with SHA256 and you've got a 32-byte shared secret only those two parties can derive. (For the full derivation, see the [ECDH chapter](/noise-tutorial/crypto-primitives) of the Noise course.)

That works because **scalar multiplication on the elliptic curve is commutative**: `a·(b·G) = b·(a·G) = ab·G`. Alice can compute the shared secret with her private key and Bob's public key, and Bob can compute the same value with his private key and Alice's public key. An eavesdropper has both public keys but neither private key, so they can't reach the same point without solving the elliptic curve discrete log problem (which, conveniently, is hard).

For Lightning, this means Alice can derive a shared secret with each of her three forwarders using only:

- Her own private key for the payment (which she generates).
- Each forwarder's node public key (which is published in gossip and known to anyone who's heard about that node).

And each forwarder can derive the same shared secret with their node private key plus a public key Alice sends them in the onion packet. **Same shared secret, different paths to it.**

<details>
<summary>Why doesn't Alice just use her real Lightning node key for ECDH?</summary>

She could, but it would wreck her privacy. Her node key is published in the gossip network and everyone knows it. If Alice used her node key to derive shared secrets for her payments, every forwarder would see her node key in the packet (because they need it to do ECDH on their side) and could trivially identify her as the sender of every payment she sent.

To stay anonymous, Alice generates a **fresh session key per payment**. The corresponding session public key looks like random bytes (because it isn't tied to any registered identity), and a different one is used for every payment. Forwarders can't link two of Alice's payments by their session keys alone, since each payment has its own.

</details>

## The naive solution: one ephemeral key per hop

So Alice needs an ephemeral keypair to do ECDH. The dumbest version of the construction would be: pick one keypair for each hop. Alice generates `(s_bob, S_bob)`, `(s_charlie, S_charlie)`, `(s_dave, S_dave)`, computes the shared secret with each hop's node key, and stuffs each `S_*` into the corresponding hop's slot in the packet.

That gives Alice three shared secrets, one per hop. Each hop sees its own `S_*`, computes ECDH against it, and recovers the matching shared secret. It would work.

But it has two problems. First, a packet that explicitly carries three different ephemeral public keys (one per hop) tells every forwarder that the route has at least three hops. Bob can count how many `S_*` slots are filled in and learn the route length, which we said in chapter 2 we don't want him to know. Second, it's wasteful: 33 bytes per hop adds up, and the packet size is precious.

There's a much better idea, and it's the heart of the **Sphinx** packet construction we'll spend the next several chapters building.

## The clever idea: a single ephemeral key, advanced by blinding

Here's the construction. Alice picks **one** ephemeral keypair, called the **session key**, and uses it as the starting point for a chain. Each hop in the route gets its own ephemeral pubkey, but they're all derived deterministically from that single session key plus information the chain accumulates as it walks the route.

Take a look at the diagram below to see how a single step of the chain works. We call it the **blinding step**.

<blinding-factor></blinding-factor>

The blinding factor `b_i = SHA256(E_i || ss_i)` is computed from public information, both the ephemeral pubkey at hop `i` and the shared secret with hop `i`. The next ephemeral key is `e_{i+1} = e_i · b_i` (or in pubkey form, `E_{i+1} = E_i · b_i`). Each hop's ephemeral key is the previous hop's, "blinded" by a factor derived from public information that the next hop can recompute.

Now look at the full chain across all three hops:

For Bob (hop 0):
- Ephemeral private key: `e₀ = session_key`
- Ephemeral public key: `E₀ = e₀ · G`
- Shared secret: `ss₀ = SHA256(e₀ · bob_pubkey)`
- Blinding factor: `b₀ = SHA256(E₀ ‖ ss₀)`

For Charlie (hop 1):
- Ephemeral private key: `e₁ = e₀ · b₀ (mod n)`
- Ephemeral public key: `E₁ = E₀ · b₀` (point multiplication, equivalent to `e₁ · G`)
- Shared secret: `ss₁ = SHA256(e₁ · charlie_pubkey)`
- Blinding factor: `b₁ = SHA256(E₁ ‖ ss₁)`

For Dave (hop 2):
- Ephemeral private key: `e₂ = e₁ · b₁ (mod n)`
- Ephemeral public key: `E₂ = E₁ · b₁`
- Shared secret: `ss₂ = SHA256(e₂ · dave_pubkey)`
- Blinding factor: `b₂ = SHA256(E₂ ‖ ss₂)`

<ecdh-chain></ecdh-chain>

Look at what this buys us. Alice computes `(E₀, ss₀, E₁, ss₁, E₂, ss₂)` in advance, working forward from her session key. She'll pack `E₀` into the start of the onion. **That's the only ephemeral key she sends.** The forwarders are going to derive the rest themselves.

## Why the forwarders can advance the chain

Bob receives the onion. The first 33 bytes contain `E₀`. Bob does:

1. `ss₀ = SHA256(bob_privkey · E₀)`. This is the same `ss₀` Alice computed (`bob_privkey · E₀ = bob_privkey · (e₀ · G) = e₀ · (bob_privkey · G) = e₀ · bob_pubkey`).
2. `b₀ = SHA256(E₀ ‖ ss₀)`. Same blinding factor Alice computed, derived from public information Bob now has.
3. `E₁ = E₀ · b₀`. Bob computes Charlie's ephemeral public key by point-multiplying `E₀` by `b₀`. This is the *same* `E₁` Alice computed, since `E₀ · b₀ = (e₀ · G) · b₀ = (e₀ · b₀) · G = e₁ · G`.
4. Bob writes `E₁` into the outgoing packet's ephemeral key slot and forwards it to Charlie.

When Charlie receives the onion, the ephemeral key slot now contains `E₁`. He runs the same routine with his own private key, derives `ss₁`, computes `b₁`, advances to `E₂`, and forwards to Dave. Dave does the same with his private key to derive `ss₂` (he doesn't need to advance the chain, since there's no next hop).

The shape of this is worth pausing on. Alice and the forwarders **each compute exactly the same sequence of values, just from different angles.** Alice has the private session key and walks the chain forward using public node keys. Each forwarder has its private node key and walks the chain forward starting from whatever ephemeral pubkey was in its inbound packet. Every value they need is either public or derivable from a private key one of them already has.

<details>
<summary>What happens to the ephemeral key when Bob forwards the packet?</summary>

Bob *replaces* the ephemeral key field in the outgoing packet with `E₁` (the next hop's ephemeral key). He does not include `E₀` anywhere in what he sends to Charlie. Charlie only ever sees `E₁`. So even though Alice and Bob both know `E₀`, Charlie can't see it, can't derive `ss₀`, and can't decrypt Bob's slice of the onion. Each hop only ever sees the ephemeral key intended for itself.

This is also why the chain has to advance with point multiplication rather than, say, "Alice puts a list of ephemeral keys in the packet." Each forwarder needs to compute the next ephemeral key on the fly so it can forward it; if they were all in the packet, you'd be back to the naive design that leaks route length.

</details>

## What Alice ends up with

After running the chain forward, Alice has three `(Eᵢ, ssᵢ)` pairs, one per hop. The `Eᵢ` values she'll never directly send except for `E₀`: the rest get computed by the forwarders themselves. The `ssᵢ` values are private to her and the corresponding hop. Alice will use them in chapter 4 to derive several distinct keys per hop: one for encrypting that hop's payload, one for an authentication tag, one for the filler we'll meet in chapter 6, and one for handling errors on the return path.

For now, the takeaway is just that **Alice has produced one shared secret per hop, without any prior interaction with the hops, using only public information about them and her own one-time session key.** That's the foundation the rest of the construction sits on.
