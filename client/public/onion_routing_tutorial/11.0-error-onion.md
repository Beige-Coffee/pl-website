# The Error Onion

So far we've assumed payments succeed. Real Lightning isn't always so kind. A hop might be offline, a channel might lack liquidity, a <g>CLTV</g> might be too tight, an invoice might have expired. When something goes wrong, the failing hop has to tell Alice. But it can't shout it across the network in plaintext, for the same privacy reasons we built the forward onion in the first place.

This chapter covers BOLT 4's solution: an **<g>error onion</g>**, an encrypted packet that wraps backward through the route, the mirror image of the forward onion's construction.

## How the error onion works

Every hop on the return path wraps the error with its own `ammag` keystream, layer by layer, just like the forward onion. The failing hop's `um` key produces an HMAC that authenticates the error contents. By the time Alice receives the error, she has multiple wrapped layers and uses her chain of shared secrets to peel them off in order.

Two privacy properties:

- **Authenticity:** Alice can tell which hop generated the error and that the error wasn't forged or modified by another hop along the return path (because only that hop knows its own `um` key).
- **Privacy:** Hops between the failing one and Alice can't read what the error says or even know where it originated (because each hop only knows its own `ammag` and `um`, not the ones for other hops).

## What the failing hop builds

Let's trace what Charlie does when he fails the payment.

1. Construct the **failure message**: a structured byte sequence with the failure code plus any associated data (like a `channel_update` for `UPDATE`-flagged failures).
2. **Length-prefix and pad it to a fixed size** so the on-wire length doesn't leak which failure it is. In our worked example the unencrypted payload comes out to 260 bytes:
   - `failure_len` (2 bytes): how long the real message is.
   - the `failure_message` itself (`failure_len` bytes).
   - `pad_len` (2 bytes): how many padding bytes follow.
   - that many zero bytes of padding.

   `failure_len + pad_len` must come to at least 256, and our example uses exactly 256 (the spec's classic recommendation), so the message and its padding stay a constant size no matter how short the real message is; the two length fields bring the total to 260. Failure messages bigger than 256 bytes (say, ones carrying a long `channel_update` plus extra TLV data) get padded to a bigger fixed total instead; the spec's own test vector pads to 1,024, a 1,060-byte packet. Whatever the total, a decoder reads the two length fields rather than assuming a size.
3. **Compute an HMAC** over the 260-byte payload with his `um` key: `error_hmac = HMAC-SHA256(um_charlie, payload)`.
4. **Assemble the error packet:** `error_hmac (32 bytes) || payload (260 bytes)`, 292 bytes in all for our example.
5. **Encrypt it** by XOR-ing with his `ammag` keystream, generated to the packet's full length: `wrapped = error_packet ⊕ ChaCha20(ammag_charlie, len(error_packet))`.
6. **Send it back** to Bob, his upstream hop, as the failure data on the <g>HTLC</g>.

Why the explicit length prefix instead of just trimming trailing zeros at decode time? Some failure messages contain binary data with internal zero bytes. For example, `temporary_channel_failure` carries a serialized `channel_update`, whose bytes can include zeros at any offset. A naive "strip trailing zeros" decoder would corrupt those messages. The u16 `failure_len` at the front tells the decoder exactly where the message ends.

## What the upstream hops do

Bob receives `wrapped` from Charlie. He doesn't know what's inside (he doesn't have Charlie's `ammag` key), and BOLT 4 says he doesn't need to. His only job is to wrap one more layer: `wrapped = wrapped ⊕ ChaCha20(ammag_bob, len(wrapped))`. He covers however many bytes arrived, then sends the new `wrapped` upstream to Alice on his own HTLC.

If there were more hops between Alice and Charlie (say, a 5-hop route), each one would wrap with its own `ammag` in turn. By the time the error reaches Alice, it's been XORed with every intermediate hop's keystream plus the originating Charlie's keystream.

This is symmetric in a useful way: Alice will peel the layers in **the same order** Charlie's wrap propagated them, hop by hop, until she finds the one whose HMAC verifies.

## What Alice does

When Alice receives the wrapped error, she has shared secrets <m>ss_AB</m>, <m>ss_AC</m>, <m>ss_AD</m> (one per hop) and can derive the corresponding `um` and `ammag` keys for each. She loops through her hops in order:

For each hop `i` in order (`i = 0, 1, 2, ..., N-1`), Alice does two things:

1. XOR the wrapped bytes with that hop's keystream, `ChaCha20(ammag_i, len(wrapped))`.
2. Check whether the leading 32 bytes are a valid HMAC over the rest under that hop's `um` key, i.e. whether `HMAC-SHA256(um_i, wrapped[32:])` equals `wrapped[:32]`. If it does, hop `i` is the one that failed: Alice reads the `failure_len` prefix at the front of the payload and pulls out exactly that many bytes as the failure message. If it doesn't, she moves on to the next hop's keys.

Why does this work? The wrapping order from the network's perspective is: Charlie XORs first, then Bob XORs, then... back to Alice. From Alice's perspective on the return, the *outermost* layer is the most recent hop's wrap (Bob's, in our 3-hop example), and the *innermost* is Charlie's (the failing hop). Alice peels in the order Bob → Charlie, which is `i = 0, 1, ...` since hop 0 is the first forwarder. After XOR-ing with hop <m>i</m>'s `ammag`, if hop <m>i</m> was the failing hop, the contents reveal a 32-byte HMAC that verifies under <m>um_i</m>. If not, Alice continues.

If she peels through every layer and never finds a valid HMAC, the error packet is malformed (or generated by a hop without a valid `um` key, which shouldn't happen). In practice this means a peer along the return path tampered with the error.

## A concrete walkthrough

For our route Alice → Bob → Charlie → Dave with Charlie failing:

1. Charlie builds: `error = HMAC(um_charlie, padded_msg) || padded_msg`. Then wraps: `wire = error ⊕ ammag_charlie_stream`. Sends `wire` back to Bob.
2. Bob wraps again: `wire = wire ⊕ ammag_bob_stream`. Sends to Alice.

Alice's peel:

3. **Iteration i=0 (Bob's keys):** `peeled = wire ⊕ ammag_bob_stream`. Check `HMAC(um_bob, peeled[32:]) == peeled[:32]`. **No**, because Bob isn't the failing hop, so his `um` doesn't authenticate Charlie's error. Continue.
4. **Iteration i=1 (Charlie's keys):** `peeled = peeled ⊕ ammag_charlie_stream`. Now `peeled` is exactly Charlie's original `error` packet. Check `HMAC(um_charlie, peeled[32:]) == peeled[:32]`. **Yes**, Charlie generated this error. Parse the length prefix at `peeled[32:34]` to find the message length, then read that many bytes starting at `peeled[34:]`.

The failing hop is identified by **whose `um` key verifies**, which Alice can check by trial-and-decrypt as she peels. This is faster than it sounds: there are at most N hops in the route, the keystream computations are cheap, and the HMAC check is one SHA256 computation.

<details>
<summary>Why does the failing hop need to wrap with its own ammag if it's already authenticated with um?</summary>

Two reasons. First, the HMAC alone reveals the error contents to upstream hops if the encryption layer is missing. Bob shouldn't be able to read "Charlie failed because of channel X", since that leaks information about Charlie's state. Second, without per-hop encryption, a malicious upstream hop could swap one valid error for another (any error from any failing hop), since the HMAC tag would still verify under the same key. The `ammag` wrap binds the error to the specific position in the route: only after peeling Bob's layer does the error contents become inspectable, and only then does the `um_charlie` HMAC verify.

</details>

<details>
<summary>What if a malicious forwarder injects a fake error?</summary>

A forwarder can write any bytes it likes onto the return HTLC. Alice will try to peel them: XOR with `ammag_0`, check `um_0` HMAC; if no match, XOR with `ammag_1`, check `um_1` HMAC; and so on. Since the forwarder doesn't have any of the `um` keys (those require knowing the corresponding shared secrets, which require knowing either the session key or the hop's node privkey), no manufactured error will ever produce a valid HMAC at any layer. Alice's algorithm will reach the end of the chain without finding a match, and she'll know the error was tampered with. She can disconnect the upstream peer or downgrade their reliability score.

</details>

## The exercises

One exercise this chapter: Alice's peel. You just saw the failing hop build its error packet in the walkthrough above, so the coding focus is the harder, more interesting half, Alice decrypting it.
