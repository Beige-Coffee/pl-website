# The Error Onion

So far we've spent a lot of time reviewing what happens when payments succeed. This is good, as nowadays (dare I say) most payments do succeed! However, sometimes they do not, so the protocol needs a way to send error messages back, allowing parties to quickly unwind the HTLCs. Additionally, error messages help provide the sender with more information as to why the payment failed. For example, a node might be offline, a channel might lack liquidity, a <g>CLTV</g> might be too tight, or an invoice might have expired. In any case, when something goes wrong, the failing hop should tell Alice why. And it should be able to tell Alice *privately*. 

In this chapter, we'll learn how a node can route an **<g>error onion</g>** backward through the route, privately informing a sender as to why a payment failed.

## How a failure travels back

So, things didn't quite work out, and the payment failed... the failing hop needs to get a message back to Alice. To do this, they'll send one of the following messages to their peer, which gets passed hop by hop toward Alice. 

- `update_fail_htlc` indicates that the payment failed, and it carries a `reason` field. **SPOILER ALERT: The `reason` is the error onion we're about to build!**
- `update_fail_malformed_htlc` is a special case. If a hop receives an onion so corrupted that it can't even decrypt it, then it has no way to retrieve the ephemeral public key and derive the shared secret, so it can't build a proper error onion. Instead, it returns a bare `failure_code` in this message, and its upstream neighbor turns that into a normal `update_fail_htlc` for the rest of the trip back.

> NOTE: The happy path is simpler. When a payment succeeds, the destination just sends an `update_fulfill_htlc` message back to the sender, hop by hop.

## Types of failure messages

As any self-help guru will tell you, whenever you fail, it's helpful to know *why*. Otherwise, how will you do better the next time? Similarly, in the Lightning Network, if your payment fails, you'll need to know why so you can make the necessary adjustments. To help with this self-improvement, BOLT 4 defines a two-byte **failure code**, where the top byte is a set of flags that tell Alice what *kind* of failure it was. Below are the four flags:

- `BADONION` (`0x8000`): The onion itself was unreadable. This is the corrupted-onion case from above, the one that comes back as `update_fail_malformed_htlc`.
- `PERM` (`0x4000`): The failure is permanent (as opposed to transient), so there's no point in Alice retrying this hop.
- `NODE` (`0x2000`): The whole node is the problem, not just one of its channels, so Alice should set aside all of that node's channels.
- `UPDATE` (`0x1000`): A channel forwarding parameter was violated, such as the amount, fee, or timelock failing to meet the hop's policy.

It's important to note that these flags can **combine** into a single code, so one failure can carry several of these facts at once (for example, `permanent_node_failure` is both `PERM` and `NODE`).

## How the error onion works

Let's begin with a high-level review. When a hop decides to fail a payment, it authenticates the **failure message** with its `um` key, encrypts it with its `ammag` keystream, and then hands it to its *upstream* peer. As each upstream hop receives the message, it adds one more `ammag` layer and passes it along, keeping the packet the same size the whole way back. Each additional layer makes the error look completely different on every channel it crosses, so it can't be traced as it travels back. If this sounds confusing, don't worry, we'll dig into it in more detail shortly!

Since Alice won't, at first, know which hop the error came from, she will need to find out by trial-and-decrypt. Specifically, she will do this by iteratively peeling a layer with each hop's `ammag` key, checking the result against their `um` HMAC to see if there is a match. If she finds a key that produces a valid HMAC, then she knows she has identified the sender of the failing onion, and she can decrypt the bytes and parse the failure message.

This protocol design provides two important properties:

- **Alice can trust the error she reads.** When a layer's HMAC verifies, she knows that error came from that exact hop and wasn't forged or altered, because only the failing hop can produce a valid `um` HMAC, and any tampering would break it. (A return-path hop can still corrupt the bytes so that *no* layer verifies; Alice will detect that, but pinning down exactly *who* tampered needs the attribution-data extension noted in chapter 6.)
- **Nobody else can read it.** The hops between the failing one and Alice can't read what the error says, or even tell where it originated. Each hop only knows its own `ammag` and `um` keys, not anyone else's.

<details>
<summary>Why re-encrypt something that's already encrypted?</summary>

In this case, it's not about hiding the contents, since the message is already encrypted! Instead, it's about making the error look different on every channel it crosses. If the error message were passed along unchanged, then the same bytes would cross every channel on the way back. This means an observer who sees the message more than once across the route could gain information that may help them stitch the route together. By re-wrapping it at each hop, we ensure the message looks different and cannot be correlated across hops on the network.

</details>

## What the failing hop builds

Now for the fun part! Let's dig into some details and see what Charlie does when he fails the payment.

#### Step 1. Construct the failure message
First, Charlie constructs the **failure message**, which contains the failure code plus any associated data.

#### Step 2. Package it to a fixed size
Next, he packages the message so the error's size can't leak which failure it is. Concretely, he writes four pieces, one after another (260 bytes total in our example):

- `failure_len` (2 bytes): how long the real message is.
- the `failure_message` itself (`failure_len` bytes).
- `pad_len` (2 bytes): how many padding bytes follow.
- the padding itself (`pad_len` bytes). BOLT 4 only constrains the *length* of the pad, not its contents, but it's conventional to fill it with zeros.

How much padding? Per BOLT 4, `failure_len + pad_len` must come to at least 256. That way, the message plus its padding stays a constant size no matter how short the real message is, and the two length fields bring the total to 260.

<python-snippet id="error-package"></python-snippet>

#### Step 3. Authenticate with `um`
Then, Charlie computes an HMAC over the 260-byte payload with his `um` key: `error_hmac = HMAC-SHA256(um_charlie, payload)`.

#### Step 4. Prepend the HMAC
After that, he prepends the HMAC to the payload: `error_hmac (32 bytes) || payload (260 bytes)`. That's the full error packet, 292 bytes in all for our example.

<python-snippet id="error-hmac"></python-snippet>

#### Step 5. Wrap with `ammag`
Now, he encrypts the whole packet by XOR-ing it with his `ammag` keystream: `wrapped = error_packet ⊕ ChaCha20(ammag_charlie, len(error_packet))`.

<python-snippet id="error-wrap"></python-snippet>

#### Step 6. Send it upstream
Finally, Charlie sends `wrapped` to Bob, his upstream hop. It rides in the `reason` field of an `update_fail_htlc` message, the same envelope we met back in *How a failure travels back* at the start of the chapter.

## What the upstream hops do

Now that we've seen what Charlie does, let's review the error onion from Bob's perspective. When Bob receives the error onion, called `wrapped` in our example, from Charlie, he doesn't know what's inside (he doesn't have Charlie's `ammag` key), but that's okay! His only job is to wrap one more layer ( `wrapped = wrapped ⊕ ChaCha20(ammag_bob, len(wrapped))`) and send the new `wrapped` upstream to Alice.

<python-snippet id="error-rewrap"></python-snippet>

If there were more hops between Alice and Charlie (say, a 5-hop route), each one would wrap with its own `ammag` in turn. By the time the error reaches Alice, it's been XORed with every intermediate hop's keystream plus the originating Charlie's keystream.

## What Alice does

When Alice receives the wrapped error, she has shared secrets <m>ss_AB</m>, <m>ss_AC</m>, <m>ss_AD</m> (one per hop) and can derive the corresponding `um` and `ammag` keys for each. Armed with this cryptographic material, she loops through each hop, starting with the most recent (in this case, Bob).

For each hop `i` in order (`i = 0, 1, 2, ..., N-1`), Alice does two things:

1. XOR the wrapped bytes with that hop's keystream, `ChaCha20(ammag_i, len(wrapped))`.
2. Check whether the leading 32 bytes are a valid HMAC over the rest under that hop's `um` key. In other words, whether `HMAC-SHA256(um_i, wrapped[32:])` equals `wrapped[:32]`. If it does, then hop `i` is the one that failed! From here, Alice will read the `failure_len` prefix at the front of the payload and pull out exactly that many bytes as the failure message.

<python-snippet id="error-trial-peel"></python-snippet>

Why does this work? From Alice's perspective, the *outermost* layer is the most recent hop's wrap (Bob's, in our 3-hop example), and the *innermost* is Charlie's (the failing hop). Alice peels in the order Bob → Charlie, which is `i = 0, 1, ...` since hop 0 is the first forwarder. After XOR-ing with hop <m>i</m>'s `ammag`, if hop <m>i</m> was the failing hop, the contents reveal a 32-byte HMAC that verifies under <m>um_i</m>. If not, Alice will continue.
