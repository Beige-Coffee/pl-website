# Wrapping Layer by Layer

We have all the ingredients: shared secrets, per-hop keys, ephemeral key chain, hop payloads, and filler bytes. It's time to build the onion.

## The Core Algorithm

Alice constructs the onion by wrapping layers from the **innermost hop (Dave) outward to the outermost hop (Bob)**. This might feel backward, but it's the only way to create nested encryption. The outermost layer (Bob's) must wrap everything inside it, so it has to be applied last.

<checkpoint id="innermost-hop-first"></checkpoint>

Here's the wrapping algorithm. Alice starts with a 1,300-byte buffer of zeros and an all-zero 32-byte HMAC (signaling "this is the last hop"). Then, for each hop from Dave to Bob:

**Step 1: Make room.** Shift the existing buffer to the right by `len(hop_payload) + 32` bytes. This pushes existing data rightward (bytes fall off the end) and opens up space at the front.

**Step 2: Insert the payload and HMAC.** Write the current hop's TLV payload into the first bytes, followed by the current HMAC (32 bytes). The HMAC placed here will be used by the **next** hop downstream to verify the packet after it peels one layer.

**Step 3: Encrypt.** XOR the entire 1,300-byte buffer with a pseudo-random stream generated from this hop's **rho key**. This is the core of Sphinx's layered encryption. Each layer of XOR encryption wraps all the inner layers.

**Step 4: Apply filler (innermost hop only).** When processing the innermost hop (the first iteration), overwrite the trailing bytes of the buffer with the pre-computed filler. This corrects the "lost bytes" from the rightward shift.

**Step 5: Compute the new HMAC.** Compute `HMAC-SHA256(mu_key, encrypted_payload || associated_data)`, where the associated data is the payment hash. This HMAC will authenticate the packet for the hop that processes this layer.

<details>
  <summary>Why does the HMAC include the payment hash as associated data?</summary>

The payment hash binds the onion packet to a specific payment. Without it, an attacker could take an onion packet from one payment and attach it to a different payment's HTLC. The HMAC would still verify (since the encrypted payload hasn't changed), but the packet would be processed in the wrong payment context. Including the payment hash as associated data ensures the HMAC is only valid for the specific payment this onion was constructed for.
</details>

<details>
  <summary>What happens to the current_hmac between iterations?</summary>

The HMAC computed at the end of each iteration serves a dual purpose. First, it becomes the HMAC that's inserted into the payload during the NEXT iteration's "insert" step. When the next outer hop peels its layer, this HMAC will be exposed as the inner hop's authentication tag. Second, at the final iteration (Bob), the HMAC becomes the outermost HMAC, which is included in the packet header. Bob verifies this HMAC before processing anything.
</details>

## Putting It All Together

The full `construct_packet` method orchestrates the entire process:

1. Ensure shared secrets, hop keys, and ephemeral keys are computed
2. Build TLV payloads for all hops
3. Generate the filler
4. Initialize the 1,300-byte buffer and zero HMAC
5. Wrap layers from innermost (Dave) to outermost (Bob)
6. Assemble the final packet: `version + ephemeral_pubkey + payload + hmac`
