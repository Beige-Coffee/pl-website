# Funding Script

## The Foundation of a Channel

The funding script is the Bitcoin script that locks funds into a Lightning channel. It is a **2-of-2 multisig** script, meaning both parties must sign to spend the output. This is the foundation upon which the entire channel is built -- every commitment transaction, every HTLC, and every close operation ultimately spends this single output.

## Why 2-of-2 Multisig?

A 2-of-2 multisig ensures that **neither party can unilaterally move the funds**. This is essential for trustless off-chain updates:

- If Alice could spend alone, she could steal Bob's balance at any time.
- If Bob could spend alone, he could steal Alice's balance.
- With 2-of-2, both must agree on every state change by providing their signatures.

This constraint is what makes commitment transactions meaningful. Each commitment transaction is pre-signed by one party and held by the other, ready to be broadcast if needed. But the funding output can only be spent with both signatures, so both parties must cooperate to create valid commitment transactions.

## The Script Structure

BOLT 3 specifies the funding script as:

```text
OP_2 <pubkey1> <pubkey2> OP_2 OP_CHECKMULTISIG
```

This is a standard Bitcoin multisig script. Let us trace through how it executes:

1. `OP_2` pushes the number 2 onto the stack (the number of required signatures).
2. `<pubkey1>` pushes the first public key (33 bytes, compressed).
3. `<pubkey2>` pushes the second public key (33 bytes, compressed).
4. `OP_2` pushes the number 2 (the total number of public keys).
5. `OP_CHECKMULTISIG` verifies that 2 valid signatures are provided for the 2 public keys.

To spend this output, the witness must provide two valid signatures (plus an empty element due to the `OP_CHECKMULTISIG` bug, which we will discuss later).

## Lexicographic Pubkey Sorting

Here is a critical detail: BOLT 3 requires that the two public keys in the funding script be **sorted lexicographically** (i.e., by their byte values, treating them as big-endian numbers).

```python
def make_funding_script(pubkey_a: bytes, pubkey_b: bytes) -> bytes:
    # Sort pubkeys lexicographically
    if pubkey_a > pubkey_b:
        pubkey_a, pubkey_b = pubkey_b, pubkey_a

    # Build the script: OP_2 <key1> <key2> OP_2 OP_CHECKMULTISIG
    script = bytes([
        0x52,               # OP_2
        0x21,               # Push 33 bytes
    ]) + pubkey_a + bytes([
        0x21,               # Push 33 bytes
    ]) + pubkey_b + bytes([
        0x52,               # OP_2
        0xae,               # OP_CHECKMULTISIG
    ])
    return script
```

### Why Sort?

Both Alice and Bob need to independently construct the exact same funding script. The script's SHA256 hash determines the P2WSH address where the funds are sent. If Alice puts her key first and Bob puts his key first, they would compute different script hashes and disagree on the funding address.

Lexicographic sorting is a simple, deterministic rule that both parties can follow independently without any coordination. No matter who opens the channel or what order the keys were exchanged in, the result is always the same script.

## Script Opcodes Reference

Here is a quick reference for the opcodes used in the funding script:

| Opcode | Hex | Description |
|--------|-----|-------------|
| `OP_2` | `0x52` | Pushes the number 2 onto the stack |
| `OP_CHECKMULTISIG` | `0xae` | Verifies M-of-N signatures against N public keys |

And the push data opcode:

| Byte | Meaning |
|------|---------|
| `0x21` | Push the next 33 bytes onto the stack |

The `0x21` is not an opcode per se -- it is a **push data** instruction. In Bitcoin Script, byte values from `0x01` to `0x4b` mean "push the next N bytes." Since a compressed public key is 33 bytes (0x21 in hex), we use `0x21` to push each key.

## From Script to Address

The funding script is not used directly as an output script. Instead, it is wrapped in a **P2WSH** (Pay-to-Witness-Script-Hash) output. The P2WSH output contains only the SHA256 hash of the script:

```python
import hashlib

def script_to_p2wsh(script: bytes) -> bytes:
    """Create P2WSH output script from witness script."""
    script_hash = hashlib.sha256(script).digest()
    # OP_0 <32-byte-hash>
    return bytes([0x00, 0x20]) + script_hash
```

The resulting P2WSH output script is 34 bytes:
- `0x00` -- SegWit version 0
- `0x20` -- Push 32 bytes
- The 32-byte SHA256 hash of the funding script

When spending this output, the full funding script is revealed in the witness data. This is more efficient on-chain because the output is small (34 bytes), and the full script only appears when the channel is closed.

## Putting It All Together

The complete flow for creating a funding script is:

1. Both parties exchange their `funding_pubkey` during channel open.
2. Sort the two public keys lexicographically.
3. Construct the multisig script: `OP_2 <sorted_key1> <sorted_key2> OP_2 OP_CHECKMULTISIG`.
4. Hash the script with SHA256 to get the P2WSH output.
5. Use this P2WSH output as the destination for the funding transaction.
