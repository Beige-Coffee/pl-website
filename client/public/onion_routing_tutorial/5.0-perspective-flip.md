# The Receiver's Perspective

For eight chapters, you've been Alice. You computed shared secrets, derived per-hop keys, built the ephemeral key chain, encoded TLV payloads, generated filler bytes, and wrapped layer after layer of encryption into a perfect 1,366-byte onion packet. You know every detail of how the packet was constructed because you built it.

Now forget all of that.

## You Are Bob

You're a Lightning node. You have a private key, a set of payment channels, and a connection to your peers. A few moments ago, Carol opened a channel with you. Alice opened one too. You've been routing payments for weeks.

An `update_add_htlc` message just arrived from Alice. It contains:

- An **onion packet**: 1,366 bytes of opaque data
- A **payment hash**: 32 bytes identifying this payment
- An **amount**: how many millisatoshis Alice is offering you
- A **CLTV expiry**: the timelock on the incoming HTLC

That's it. You have no idea who built this packet, where it's ultimately going, or how many hops remain in the route. You don't know if Alice is the original sender or just another forwarding node. You don't know if the next hop is the final destination or one of twenty intermediaries. All you have is 1,366 bytes and a decision to make.

## The Packet from Bob's Side

Let's look at what Bob actually sees:

The packet is 1,366 bytes: a 1-byte version, 33-byte ephemeral public key, 1,300-byte encrypted payload, and 32-byte HMAC. Bob sees the same structure Alice created, but from his side it's all opaque data.

Bob sees the same structure Alice created. But from Bob's perspective, the ephemeral public key is just a point on the curve from an unknown sender. The 1,300-byte payload is an encrypted blob he can't read yet. The HMAC is a 32-byte tag he needs to verify before doing anything else.

The only tool Bob has is his **private key**. With it, he can perform ECDH against the ephemeral public key to derive a shared secret. From that shared secret, he can derive the same per-hop keys that Alice derived for him (rho, mu, um, pad, ammag). Then, and only then, can he verify the HMAC and decrypt his layer.

## Introducing SphinxPacketProcessor

In the next two chapters, we'll build the receiver's counterpart to `SphinxPacketBuilder`. The **`SphinxPacketProcessor`** class takes a node's private key and processes incoming onion packets. Its core operation, `peel_layer`, is the mirror image of the builder's `wrap_layer`:

| Builder (Alice) | Processor (Bob) |
|---|---|
| Compute shared secret via ECDH | Compute shared secret via ECDH |
| Derive per-hop keys | Derive per-hop keys |
| Wrap: shift right, insert, encrypt, HMAC | Peel: verify HMAC, decrypt, extract, shift left |
| Blind ephemeral key forward | Blind ephemeral key forward |

The symmetry is elegant. Alice uses the session private key and the hop's public key. Bob uses his private key and the ephemeral public key. Both arrive at the same shared secret, the same keys, and the same cipher stream. What Alice encrypted, Bob can decrypt. What Alice authenticated, Bob can verify.
