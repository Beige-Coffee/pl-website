# Commitment Secrets & Per-Commitment Points

## The Problem of Secret Storage

Every time a channel state is updated, the old state is revoked by revealing a per-commitment secret. Over the lifetime of a channel, there could be millions of state updates. If Bob had to store every single per-commitment secret that Alice revealed, the storage requirements would grow linearly without bound.

BOLT 3 solves this with an elegant algorithm that allows compact storage of all revealed secrets using only **O(log n) space** instead of O(n). This algorithm is based on a structure called a **shachain**.

## Per-Commitment Secret Generation

Each per-commitment secret is derived deterministically from the **commitment seed** (key family 5). The algorithm uses a combination of bit manipulation and SHA256 hashing.

### The Algorithm

Given a commitment seed and a commitment number `i` (counting from a maximum value down to 0), the per-commitment secret is generated as follows:

1. Start with the seed (32 bytes).
2. For each bit position `b` from 47 down to 0:
   - If bit `b` of the commitment number is **not set** in `i`:
     - Flip bit `b` of byte position `b / 8` in the current value.
     - SHA256 hash the entire 32-byte value.
3. The result is the per-commitment secret for commitment number `i`.

Here is the algorithm in Python:

```python
import hashlib

def derive_commitment_secret(seed: bytes, index: int) -> bytes:
    """Derive per-commitment secret for a given index.

    The index counts DOWN from 2^48 - 1 to 0.
    Index 2^48 - 1 is the FIRST commitment (commitment #0).
    """
    secret = bytearray(seed)

    for bit in range(47, -1, -1):
        if not (index & (1 << bit)):
            # Flip the bit
            byte_index = bit // 8
            bit_within_byte = bit % 8
            secret[byte_index] ^= (1 << bit_within_byte)
            # Hash
            secret = bytearray(hashlib.sha256(bytes(secret)).digest())

    return bytes(secret)
```

### Understanding the Index

The index scheme might seem counterintuitive at first:

- Commitment **#0** (the first commitment) uses index `281474976710655` (which is `2^48 - 1`).
- Commitment **#1** uses index `281474976710654` (which is `2^48 - 2`).
- The index **decreases** as commitment numbers increase.

This descending order is critical for the compact storage scheme. It allows the receiver to store secrets in a tree structure where later secrets (with lower indices) can derive earlier secrets (with higher indices) through the same bit-flipping and hashing process.

## Why This Specific Algorithm?

The bit-flipping plus hashing algorithm has a special property: given a secret at index `i`, you can derive the secret at any index `j > i` (corresponding to an earlier commitment). This is because:

- If `j` has a bit set that `i` does not, you can flip that bit and hash to get closer to `j`.
- The derivation is one-directional: you can go from lower indices to higher indices but not the reverse.

This means:

- **Alice** can generate secrets for indices going down (newest to oldest) by starting from the seed.
- **Bob** can store just O(log n) secrets and derive any previously revealed secret from them.

In practice, Bob only needs to store at most 48 secrets (one for each bit position) to be able to reconstruct any of the up to 2^48 previously revealed secrets. This is the shachain data structure.

## Per-Commitment Points

A **per-commitment point** is simply the elliptic curve public key corresponding to a per-commitment secret:

```python
def commitment_point_from_secret(secret: bytes) -> bytes:
    """Compute the per-commitment point (public key) from a per-commitment secret."""
    # The per-commitment point is secret * G (the generator point)
    return privkey_to_pubkey(secret)
```

The per-commitment point is shared with the counterparty **before** the corresponding commitment transaction is created. The per-commitment secret is revealed **after** the state is revoked.

### The Flow

For commitment number `n`:

1. The committing party derives `per_commitment_secret_n` from their commitment seed.
2. They compute `per_commitment_point_n = per_commitment_secret_n * G`.
3. They share `per_commitment_point_n` with the counterparty.
4. Both parties use this point to derive per-commitment keys for the commitment transaction.
5. When state `n` is revoked (state `n+1` is created), the secret `per_commitment_secret_n` is revealed.

## Per-Commitment Points in Key Derivation

Per-commitment points are used to derive several keys in each commitment transaction:

| Key | Derivation |
|-----|-----------|
| `revocation_pubkey` | From `revocation_basepoint` and `per_commitment_point` |
| `local_delayed_pubkey` | From `delayed_payment_basepoint` and `per_commitment_point` |
| `local_htlc_pubkey` | From `htlc_basepoint` and `per_commitment_point` |
| `remote_htlc_pubkey` | From remote `htlc_basepoint` and `per_commitment_point` |

We will cover the exact derivation formulas in the next chapter on per-commitment key derivation.

## The Security Model

The per-commitment secret system creates a clean security boundary:

- **Before revocation**: Only the committing party knows the secret. The per-commitment point is public, but the secret cannot be derived from it (this is the discrete logarithm problem).
- **After revocation**: The secret is revealed, giving the counterparty the ability to derive revocation keys. This is the point of no return -- once the secret is shared, the old state can be penalized.
- **Storage efficiency**: The shachain structure means a node can handle millions of state updates without running out of storage for revocation data.
