Now that we've built our intuition for how payment information moves through the network, it's time to dig in deeper and name the encryption framework that we've been alluding to: **onion routing**.

Onion routing is a cryptographic framework for sending packets of information privately across a multi-hop network. It derives its name from the fact that the packets are, quite literally, wrapped in layers of encryption such that each forwarder "peels" off the layer that is addressed to them, finds a smaller onion underneath, and passes that on to the next recipient.

The specific construction Lightning uses is called **Sphinx**, defined in BOLT 4. It's a packet format from a 2009 academic paper by Danezis and Goldberg, originally built for anonymous mix networks. Lightning adapted it for payment routing because it gives the privacy properties we want: forwarders learn only what they need, and (spoiler alert: as we'll see in the next chapter) the packet's size doesn't reveal how long the route is.

What we built up to in chapter 4 was the foundation Sphinx needs: each hop has its own 32-byte shared secret with Alice, derived deterministically from a single session key. This chapter takes the next step: turning each shared secret into the small family of keys the onion construction actually uses.

## Inside the onion packet

Up until now, we've visualized the onion as three stacked encrypted slices, one per hop. As you can see above, the onion packet has three sections, each with a specific purpose:

- **`HEADER`** holds the version byte (currently `0x00`) and the *ephemeral pubkey* for the current hop. The version byte exists so future BOLT updates can introduce a new packet format without breaking older nodes: any forwarder that receives a version it doesn't recognize can simply reject the packet. The ephemeral pubkey is what the forwarder combines with their own node private key (via ECDH) to derive their shared secret with Alice. For example, when Bob receives the packet, he uses the ephemeral pubkey from the header to derive <m>ss_AB</m>, and from that secret he derives the keys he needs to peel his layer. Without the header, no shared secret can be derived, and the forwarder can't decrypt anything in the packet.
- **`PAYLOAD AREA`** carries the per-hop instructions, wrapped in one layer of encryption per hop in the route. Each forwarder strips one layer off the entire buffer, exposing their slice at the front while the inner layers still cover the rest.
- **`HMAC TAG`** is a 32-byte authentication tag covering the payload area. HMAC (short for *Hash-based Message Authentication Code*) is a cryptographic primitive that combines a secret key with the message bytes to produce a fixed-size tag, which only someone holding the key can generate or verify. Before any decryption, each hop computes its own tag over the payload using its mu key and checks it against the one in the packet. If they don't match, the packet was either altered or built wrong, and the receiving node rejects it immediately. (For a deeper dive into HMAC and how it's constructed, see the [cryptographic primitives chapter](/noise-tutorial/crypto-primitives) of the Noise Protocol course.)

## Cryptographic operations across the roundtrip

As the packet flows through the route, it has to support a handful of distinct cryptographic operations. So far, we've been focusing on encrypting the payload, but there are several others. See the list below:

- **Encrypt the forward payload** with a stream cipher (in this case, ChaCha20) so that only the intended hop can read the payment instructions. As we learned earlier, without this encryption, every hop would be able to see Alice's full route in plaintext.
- **Authenticate the forward layer** with a MAC (HMAC-SHA256) so the forwarder can detect tampering *before* processing. Without it, an attacker could flip bytes mid-flight and force the forwarder to act on wrong information.
- **Encrypt return-path errors** with a stream cipher so only Alice can read the failure reason. Without it, intermediate hops on the return path would learn what went wrong and where, which can be a serious privacy leak.
- **Authenticate return-path errors** with a MAC so Alice can verify the error actually came from a specific hop on her route. Without it, anyone could inject fake errors and Alice would accept them as real.
- **Initialize the empty buffer** with a stream cipher (sender-only) so unused space at the end of the packet doesn't leak path length. We'll come back to this one shortly.

Now, here's the kicker... **each one of these operations needs its own dedicated key.** If we reused one key across multiple operations, we'd break the security guarantees these cryptographic tools rely on. Below are a few specific reasons:

- **Encryption and authentication are different jobs.** ChaCha20 (the stream cipher) and HMAC-SHA256 (the MAC) each come with their own security guarantees, but those guarantees only hold when each key is used for *one* job. If we used the same key for both, those guarantees would no longer apply.

- **Forward and backward directions can't share a key.** [BOLT 4](https://github.com/lightning/bolts/blob/master/04-onion-routing.md#pseudo-random-byte-stream) uses ChaCha20 with the same nonce (zero) for both forward and backward encryption. The spec itself notes that *"the use of a fixed nonce is safe, since the keys are never reused."* As we learned in the Noise Protocol course, reusing the same nonce with the same key is a very bad idea, as an attacker could XOR the forward and backward payload ciphertexts together to recover the original plaintext. The same logic applies to MACs: if [forward](https://github.com/lightning/bolts/blob/master/04-onion-routing.md#key-generation) and [backward](https://github.com/lightning/bolts/blob/master/04-onion-routing.md#returning-errors) authentication shared a key, a forwarder who legitimately verified the forward MAC during routing could turn around and forge a fake error packet that Alice would accept as real.

So, using *just* one shared secret won't be sufficient for our purposes. Instead, we'll need to derive **five distinct 32-byte keys**. ## Deriving the five keys

Now that we've seen each cryptographic operations and learned why each needs its own key, let's review the specification, defined in [BOLT 4's Key Generation section](https://github.com/lightning/bolts/blob/master/04-onion-routing.md#key-generation), for how to derive them. Below is the formula, provided in the BOLT.

```
generate_key(name, secret) = HMAC-SHA256(key=name, msg=secret)
```

In the formula, `name` is a short ASCII label used to derive unique keys for each of the purposes we reviewed above. As we'll soon see, four of these keys are used to derive keys that are unique *to each hop*, using that hop's shared secret <m>ss_i</m>. The fifth name (`pad`) is combined with Alice's session key to derive a key that is unique to this specific payment.

<kdf-pipeline></kdf-pipeline>

The five keys, by name and role:

- **`rho`**: stream cipher key for the forward payload. Drives the ChaCha20 keystream that XORs each hop's slice into ciphertext (and back out when the hop peels its layer).
- **`mu`**: HMAC key for the forward authentication tag. Forwarders verify this *first*, before any decryption.
- **`um`**: HMAC key for return-path errors. The failing hop signs the error packet with `um`; Alice verifies it on the way back.
- **`ammag`**: stream cipher key for return-path errors. Wraps the error payload on its trip back through the route.
- **`pad`**: buffer-init key. Derived from Alice's session key (not a per-hop secret). Pre-fills the empty payload area with random-looking bytes before any onion layers are applied, so the trailing portion of the packet doesn't leak path length.

Every hop has its own `rho`, `mu`, `um`, and `ammag` (Bob's `rho` is different from Charlie's, which is different from Dave's), and every payment has a fresh session key driving fresh shared secrets driving fresh keys all the way down. Nothing is reused across payments.

> **Fun Fact!** The names come from cryptographic convention plus a few wordplay-y choices. `rho` and `mu` match the Sphinx paper's stream-cipher (`h_ρ`) and MAC (`h_μ`) hash oracles. `ammag` is `gamma` reversed, the return-direction counterpart of an encryption key. `um` is `mu` reversed, the return-direction MAC key. `pad` does what it says.

<summary>What about attribution data: isn't there a sixth key now?</summary>

In September 2025, BOLT 4 added an optional **attribution data** feature for return-path errors, along with a fifth per-hop key called `ammagext` (derived the same way: <m>ammagext_i</m> = HMAC-SHA256("ammagext", <m>ss_i</m>)). Attribution data lets Alice identify *which* hop in the route failed even when intermediate hops misbehave.

The construction is identical to the other four per-hop keys, so once you have `rho`, `mu`, `um`, and `ammag` down, adding `ammagext` is a one-line extension. We're scoping this chapter to the original four because attribution data is still rolling out across implementations and the core privacy story doesn't depend on it.
