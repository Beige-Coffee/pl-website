# The Error Onion

So far we've assumed payments succeed. Real Lightning isn't always so kind. A hop might be offline, a channel might lack liquidity, a CLTV might be too tight, an invoice might have expired. When something goes wrong, the failing hop has to tell Alice. But it can't shout it across the network in plaintext, for the same privacy reasons we built the forward onion in the first place.

This chapter covers BOLT 4's solution: an **error onion**, an encrypted packet that wraps backward through the route, mirror-image of the forward onion's construction.

## Why errors need their own onion

When a forward onion succeeds, the preimage propagates from Dave back through Carol and Bob to Alice as part of the HTLC settlement on each channel. That return path doesn't need any extra cryptography because revealing the preimage already proves the payment was claimed.

Errors are different. There's no preimage to return. There's a failure code (e.g., `temporary_channel_failure`, `incorrect_or_unknown_payment_details`, `expiry_too_soon`), and the failing hop wants Alice to receive that code with two properties:

- **Authenticity:** Alice has to be able to tell which hop generated the error and that the error wasn't forged or modified by another hop along the return path.
- **Privacy:** Hops between the failing one and Alice shouldn't learn what the error says or even where in the route it originated.

BOLT 4's solution is: every hop on the return path wraps the error with its own `ammag` keystream, layer by layer, just like the forward onion. The failing hop's `um` key produces an HMAC that authenticates the error contents. By the time Alice receives the error, she has multiple wrapped layers and uses her chain of shared secrets to peel them off.

## What the failing hop builds

Suppose Carol decides to fail the payment with `temporary_channel_failure`. She:

1. Constructs the **failure message**: a structured byte sequence specifying the failure code plus any associated data (such as a `channel_update` for `UPDATE`-flagged failures).
2. **Pads the message** to a fixed size (256 bytes in BOLT 4) so the length doesn't leak any information about what kind of failure it is.
3. **Computes an HMAC** over the padded message using her `um` key:
   ```
   error_hmac = HMAC-SHA256(um_carol, padded_failure_message)
   ```
4. **Assembles the error packet:** `error_hmac (32) || padded_failure_message (256)` = 288 bytes total.
5. **Encrypts it** by XOR-ing with her `ammag` keystream:
   ```
   wrapped = error_packet ⊕ ChaCha20(ammag_carol, 288)
   ```
6. **Sends it back** to Bob (the upstream hop) as the failure data on the HTLC.

## What the upstream hops do

Bob receives `wrapped` from Carol. He doesn't know what's inside (he doesn't have Carol's `ammag` key) and BOLT 4 specifies he doesn't need to know. His only job is to wrap one more layer:

```
wrapped = wrapped ⊕ ChaCha20(ammag_bob, 288)
```

He then sends the new `wrapped` to Alice on his upstream HTLC.

If there were more hops between Alice and Carol (say, a 5-hop route), each one would wrap with their own `ammag` in turn. By the time it reaches Alice, the error has been XORed with every intermediate hop's keystream plus the originating Carol's keystream.

This is symmetric in a useful way: Alice will peel the layers in **the same order** she'd construct them on the forward path, hop by hop, until she finds the one whose HMAC verifies.

## What Alice does

When Alice receives the wrapped error, she has shared secrets `ss_0, ss_1, ss_2` (one per hop) and can derive the corresponding `um` and `ammag` keys for each. She does the following:

For `i = 0, 1, 2, ..., N-1`:
1. XOR the wrapped bytes with `ChaCha20(ammag_i, 288)`.
2. Check whether the leading 32 bytes are a valid HMAC over the trailing 256 bytes using `um_i`:
   ```
   if HMAC-SHA256(um_i, wrapped[32:]) == wrapped[:32]:
       # hop i is the failing hop
       parse_failure_message(wrapped[32:])
       return
   ```
3. If the HMAC didn't verify, continue to the next hop's keys.

Why does this work? The wrapping order from Carol's perspective is: Carol XORs first, then Bob XORs, then ... back to Alice. From Alice's perspective on the return, the *outermost* layer is the most recent hop's wrap (Bob's, in our 3-hop example), and the *innermost* is Carol's (the failing hop). Alice peels in the order `bob → carol`, which is `i = 0, 1, ...` since hop 0 is the first forwarder. After XOR-ing with hop `i`'s ammag, if hop `i` was the failing hop, the contents reveal a 32-byte HMAC that verifies under `um_i`. If not, Alice continues.

If she peels through every layer and never finds a valid HMAC, the error packet is malformed (or generated by a hop without a valid `um` key, which shouldn't happen). In practice this means a peer along the return path tampered with the error.

## A concrete walkthrough

For our route Bob → Carol → Dave with Carol failing:

1. Carol builds: `error = HMAC(um_carol, padded_msg) || padded_msg`. Then wraps: `wire = error ⊕ ammag_carol_stream`. Sends `wire` back to Bob.
2. Bob wraps again: `wire = wire ⊕ ammag_bob_stream`. Sends to Alice.

Alice's peel:

3. **Iteration i=0 (Bob's keys):** `peeled = wire ⊕ ammag_bob_stream`. Check `HMAC(um_bob, peeled[32:]) == peeled[:32]`. **No**, because Bob isn't the failing hop, so his `um` doesn't authenticate Carol's error. Continue.
4. **Iteration i=1 (Carol's keys):** `peeled = peeled ⊕ ammag_carol_stream`. Now `peeled` is exactly Carol's original `error` packet. Check `HMAC(um_carol, peeled[32:]) == peeled[:32]`. **Yes**, Carol generated this error. Parse `peeled[32:]` as the failure message.

The failing hop is identified by **whose `um` key verifies**, which Alice can check by trial-and-decrypt as she peels. This is faster than it sounds: there are at most `N` hops in the route, the keystream computations are cheap, and the HMAC check is one SHA256 computation.

<details>
<summary>Why does the failing hop need to wrap with its own ammag if it's already authenticated with um?</summary>

Two reasons. First, the HMAC alone reveals to upstream hops what the error contents are if the encryption layer is missing. Bob shouldn't be able to read "Carol failed because of channel X", since that leaks information about Carol's state. Second, without per-hop encryption, a malicious upstream hop could swap one valid error for another (any error from any failing hop), since the HMAC tag would still verify. The ammag wrap binds the error to the specific position in the route: only after peeling Bob's layer does the error contents become inspectable, and only then does the `um_carol` HMAC verify.

</details>

<details>
<summary>What if a malicious forwarder injects a fake error?</summary>

A forwarder can write any bytes it likes onto the return HTLC. Alice will try to peel them: XOR with `ammag_0`, check `um_0` HMAC; if no match, XOR with `ammag_1`, check `um_1` HMAC; and so on. Since the forwarder doesn't have any of the `um` keys (those require knowing the corresponding shared secrets, which require knowing either the session key or the hop's node privkey), no manufactured error will ever produce a valid HMAC at any layer. Alice's algorithm will reach the end of the chain without finding a match, and she'll know the error was tampered with. She can disconnect the upstream peer or downgrade their reliability score.

</details>
