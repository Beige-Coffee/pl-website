# Forwarding & Validation

Bob can now peel a layer and produce the packet for the next hop. But peeling is only part of his job. Before he forwards anything, Bob needs to verify the packet is authentic, parse his <g>TLV</g> instructions, sanity-check the forwarding parameters, and decide whether to proceed or fail. This chapter covers that full loop.

By the end, you'll have written two exercises that together close the forwarder side: a small focused `verify_hmac` for the integrity check, and the orchestrator `process` that ties verification, peeling, parsing, and branching together.

## A first attempt: just decrypt and read

Suppose Bob receives an onion and his first instinct is: derive his keys, run the chapter-9 peel, parse the TLV, and decide what to do. Skip integrity checks for now and go fast.

What goes wrong with this design? Two kinds of trouble.

First, **untrusted bytes hit the parser before any check**. An attacker who can modify a packet between Alice and Bob (a man-in-the-middle, or a misbehaving upstream peer) can feed Bob whatever bytes they want. Bob's TLV parser is now processing adversarial input. If there's any bug in the parser (an off-by-one on a bigsize length, an unbounded read, anything), the attacker can exploit it. Reducing the surface that sees adversarial data is a defensive win.

Second, **Bob spends CPU on bogus packets**. Generating 2,600 bytes of ChaCha20 keystream for a peel isn't free. If we run the peel before checking integrity, we burn cycles on every tampered packet a bad peer sends our way, which is a useful denial-of-service tool for the bad peer.

So decrypt-first is the wrong order. Both problems go away if Bob verifies the <g>HMAC</g> *before* doing anything else.

## The right order: verify, then peel

The full processing order is:

1. **Length and version checks.** If the packet isn't 1,366 bytes or the version byte isn't `0x00`, reject immediately. Cheap, defensive.
2. **HMAC verification.** Recompute `HMAC-SHA256(mu_i, hop_payloads || associated_data)` and compare against the packet's `hmac` field. Constant-time comparison if you can. If it fails, reject with `invalid_onion_hmac`. Don't proceed.
3. **Peel.** This is the chapter-9 algorithm: derive `rho`, decrypt with extended keystream, extract the hop payload. Only after HMAC verification passes.
4. **Parse the TLV payload.** Determine if Bob is a forwarder (`short_channel_id` present) or the destination (`payment_data` present).
5. **Sanity-check fees and timelocks.** If forwarding, make sure the incoming <g>HTLC</g>'s amount and <g>CLTV</g> are consistent with what the TLV asks Bob to forward.
6. **Forward or settle.** If everything checks out, hand the outgoing HTLC to the appropriate channel. If not, fail with the appropriate error code.

## What `associated_data` is, and why it's in the HMAC

The `associated_data` is a 32-byte value bound into every hop's HMAC at construction time. For HTLC payments, it's the `payment_hash` (sometimes written `H(preimage)`). Bob already knows this value: when his upstream peer forwards an HTLC to him, the HTLC carries the `payment_hash` on it. The same `payment_hash` appears on the outbound HTLC Bob sends to Charlie, and so on, all the way to the destination.

When Alice built the onion (chapter 8), she computed every hop's HMAC over `hop_payloads || payment_hash`. When Bob verifies, he uses the *same* `payment_hash` from the HTLC sitting in front of him. If they match, the HMAC verifies and Bob proceeds.

Why is this in the HMAC? It binds the onion to a specific payment. Without it, an attacker who intercepted an in-flight onion could detach it from the original HTLC and re-attach it to a different one, getting forwarders to apply the routing instructions to the wrong payment. With `associated_data` in the HMAC, the onion is locked to one specific HTLC: any attempt to re-attach produces an HMAC failure at the first hop.

## Why the HMAC is over the encrypted bytes

Step 2 verifies the HMAC against `hop_payloads` *as they appear on the wire*, before any decryption. This is the encrypt-then-MAC pattern from the [Noise course's primitives chapter](/noise-tutorial/crypto-primitives). Two reasons it's the right design:

- **No untrusted data hits the parser.** If an attacker tampered with even one byte of `hop_payloads`, the HMAC fails and we reject before we run our TLV parser. The parser is a piece of code that has to handle adversarial inputs; reducing the surface it sees is a defensive win.
- **Clear failure mode.** A peer who modifies a packet between us and the upstream hop produces an HMAC failure. We can rate-limit or disconnect the peer cleanly. If we caught the failure later (e.g., after parsing), the diagnostic would be murkier.

## Parsing the TLV payload

The peel step gives us back `payload_bytes`, a chunk of bigsize-prefixed TLV records. Inside it are four kinds of fields:

- Type 2: `amt_to_forward` (msat).
- Type 4: `outgoing_cltv_value` (block height).
- Type 6: `short_channel_id` (intermediate hops only).
- Type 8: `payment_data` (final hop only): `payment_secret` + `total_msat`.

Bob walks the TLV records in order (TLV requires monotonically increasing types) and extracts whatever is present. If `short_channel_id` is there, Bob is forwarding. If `payment_data` is there but `short_channel_id` is not, Bob is the destination. Both being present is malformed; neither being present is also malformed.

For our exercises, we'll provide a small `parse_tlv_records(payload_bytes) → dict` parser in the starter code alongside the other crypto primitives (just like <g>ECDH</g>, HKDF, and ChaCha20). Walking bigsize-prefixed TLV bytes is BOLT 1 plumbing rather than an onion-routing idea, so we focus on the cryptographic decisions in the exercise itself rather than the bigsize loop.

## Fee and CLTV sanity checks

Bob's incoming HTLC carries an amount and a CLTV. The TLV in his payload says how much to forward and what CLTV to use on the outgoing HTLC. Bob has to confirm:

- **Incoming amount is enough.** The incoming amount must cover `amt_to_forward` plus Bob's own fee. If Alice computed everything correctly back in chapter 2's backward math, this will hold. If not (or if a previous hop forwarded less than instructed), Bob fails with `amount_below_minimum` or `fee_insufficient`.
- **Incoming CLTV gives enough cushion.** The incoming CLTV must be greater than `outgoing_cltv_value` by at least Bob's published CLTV delta. If not, Bob fails with `incorrect_cltv_expiry` or `expiry_too_soon`.
- **Outgoing CLTV is in the future.** Even if the delta math works, the outgoing CLTV has to be a block height that hasn't passed yet, ideally far enough in the future that downstream hops have time to settle.

If all checks pass, Bob hands the outgoing HTLC to the channel identified by `short_channel_id`, with amount `amt_to_forward` and timelock `outgoing_cltv_value`.

For the destination (`payment_data` present), the validation is different: confirm the `payment_secret` matches the invoice and the total amount is what the receiver asked for. Invoice validation is a different protocol layer; we won't dive into it here.

<details>
<summary>What if Bob's downstream channel is offline or has insufficient liquidity?</summary>

Bob fails the HTLC with `temporary_channel_failure`. The <g>error onion</g> (chapter 11) returns to Alice, who tries a different route on retry. This is one of the most common failure modes on real Lightning networks: pathfinding can't always know with certainty whether a particular channel has the liquidity in the right direction at the moment of payment, so retries are normal.

`temporary_channel_failure` is signaled by including the `UPDATE` flag (0x1000) and a `channel_update` message in the failure data, telling Alice the current state of the channel so her pathfinder can avoid stale information.

## What `process` returns

We'll model the result with three cases:

- **`ForwardInstruction(next_packet, short_channel_id, amt_to_forward, outgoing_cltv, ss)`**: Bob should send `next_packet` to the channel identified by `short_channel_id`, with the given amount and CLTV. The `ss` (shared secret) is kept for use on the error path if something downstream goes wrong.
- **`FinalDelivery(amt_to_forward, outgoing_cltv, payment_data, ss)`**: Bob is the destination. He compares `payment_data` against his pending invoice and either settles (revealing the <g>preimage</g>) or fails the HTLC.
- **`Rejection(code, ss)`**: Something's wrong. Build an error onion using the appropriate failure code and `um` key. (We'll do this in chapter 11.)

The `ss` propagates through every result type because the error path always needs it: either to wrap an error from this hop, or (for forwarders) to unwrap an inbound error from downstream and re-wrap it for the upstream peer.

## The exercises

Two exercises this chapter, each focused on a single idea.

The first, `verify_hmac`, is the defensive integrity check on its own. Compute HMAC-SHA256 over the inbound `hop_payloads || associated_data` with the forwarder's `mu` key, compare against the packet's `hmac` field, return True or False. Small, focused, and the foundation of every safe forwarder loop.

The second, `process`, ties everything together. Length and version checks, then `verify_hmac` (your function from above), then `peel_layer` (your function from chapter 9), then `parse_tlv_records` (the parser we provide), then the branching decision based on the TLV fields and the next-hop HMAC. The exercise tests the full forwarder flow with both the forwarding path and the destination path, plus malformed and tampered-packet tests that should produce a `Rejection`.
