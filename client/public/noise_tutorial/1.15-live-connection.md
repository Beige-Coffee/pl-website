### Is There Actually a Server?

Good question. Let's find out.

Below is a simple probe that opens a raw WebSocket connection to the course server and sends a plaintext "hello." There is no encryption and no handshake - it's just 5 bytes of ASCII over the wire.

The server is expecting a 50-byte Noise Protocol Act 1 message (1-byte version + 33-byte ephemeral public key + 16-byte MAC). When it gets plain English instead, it has no idea what to do, so it will reject the connection immediately.

### Putting Your Code to Work

The panel below will take all of the functions you've written throughout this course and run them against Programming Lightning's server. Under the hood, your Python code will open a real WebSocket connection, perform the XK handshake, and establish an encrypted channel, just as two Lightning nodes would.

Here's the flow:

1. **Pre-flight check**: When you click "Begin Handshake", the Lab will first verify that all 13 of your exercise solutions produce correct outputs. If they don't, you'll see a link to jump back and fix it.

2. **Handshake**: Once pre-flight checks pass, you'll watch each Act execute in real time. The color-coded bars on the arrows break down the raw bytes: version byte, ephemeral key, encrypted payload, MAC tag. Click any Act's bar to inspect individual bytes in the grid below.

3. **Encrypted channel**: After the handshake completes, a chat interface opens right inside the same component. The BOLT message buttons let you send real Lightning protocol messages. Each one gets encrypted by your `encrypt_message()`, sent over the wire, and (for request/response messages like `ping`) decrypted by the server which sends an encrypted reply. Click any chat bubble to inspect the raw encrypted bytes.

### Real Lightning Messages

The quick command buttons below the chat send real **Lightning BOLT protocol messages**. There are two patterns to notice:

**Request/response:**

- **`ping`** (BOLT 1, type 18): A keepalive message. The server responds with `pong`. This is the simplest round trip, and it proves your encryption and decryption are working end-to-end.

**Gossip broadcasts (fire-and-forget):**

These are one-way messages. The server receives them but does not reply. That's how real Lightning gossip works: nodes broadcast announcements to their peers, and peers silently forward them through the network.

- **`node_announcement`** (BOLT 7, type 257): Advertises a node's identity, alias, RGB color, supported features, and network addresses (IPv4, IPv6, Tor v3, DNS). Nodes not associated with a known channel are ignored to prevent denial-of-service attacks.
- **`channel_announcement`** (BOLT 7, type 256): Announces a new payment channel between two nodes. Contains the short channel ID, both nodes' public keys, both funding keys, and a chain hash identifying the network.
- **`channel_update`** (BOLT 7, type 258): Updates routing parameters for a channel: the fee rate, minimum HTLC amount, timelock delta, and whether the channel is currently disabled.

Click any sent gossip bubble to inspect the encrypted bytes on the wire.

### Watching Key Rotation

Between the handshake acts and the chat, the **Cipher State** inspector shows the live internal state of your `CipherState` objects: the current encryption key, the chaining key, and the nonce counter. These are the exact `self.key`, `self.chaining_key`, and `self.nonce` attributes from the class you wrote.

Every time you send or receive a message, the nonce increments by 2 (one for the length prefix, one for the body). **Watch the progress bar fill toward 1,000**.

When the nonce hits 1,000, your `_maybe_rotate()` method fires. It calls `hkdf_two_keys(self.chaining_key, self.key)` to derive a completely new encryption key, then resets the nonce to zero. The server does the same thing on its end. **The panel will flash gold and the key value will change before your eyes**.

If you don't want to send 500 messages manually (totally understandable!), you can use the **Burst** control below the chat. It auto-calculates how many pings are needed to reach the next rotation (500 messages = 1,000 nonces). When it rolls over, that's your code performing a live key rotation on a real encrypted connection!

<details>
  <summary>Why does key rotation happen every 1,000 nonces instead of, say, every message?</summary>

According to <a href="https://github.com/lightning/bolts/blob/master/08-transport.md#lightning-message-key-rotation" target="_blank">BOLT 8</a>, the purpose of key rotation is backwards secrecy: "Changing keys regularly and forgetting previous keys is useful to prevent the decryption of old messages, in the case of later key leakage." Rotating more often would limit the window of exposure if a key leaks, but each rotation requires an HKDF derivation. 1,000 nonces (500 messages) strikes a practical balance, keeping the exposure window small without adding noticeable overhead to normal traffic.

</details>

<details>
  <summary>What if the server and I get out of sync on key rotation?</summary>

They can't, as long as both sides follow the protocol correctly. Both your `CipherState` and the server's `CipherState` increment nonces identically for each encrypt/decrypt operation. Since they derive the same keys from the same chaining key, they always rotate at the same point. If something goes wrong (a dropped message, a decryption failure), the connection breaks immediately. There's no way to silently fall out of sync.

</details>
