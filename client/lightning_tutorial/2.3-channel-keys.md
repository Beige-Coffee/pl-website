# Channel Key Manager

Now that we understand BIP32 hierarchical derivation, let's put it to use! In this section, we'll build a **`ChannelKeyManager`** class that generates all of the keys and cryptographic material we'll need during the rest of this course.

## The BIP32 Library

The `bip32` library implements the full BIP32 specification, handling all the low-level HMAC-SHA512 child key derivation for us. Under the hood, it uses `libsecp256k1` (via `coincurve`) for elliptic curve operations, the same library used by Bitcoin Core!

<details>
  <summary>Click to learn about the BIP32 library</summary>

The `BIP32` class is the main entry point. You create an instance from a seed, and then derive keys at any path:

1. **`BIP32.from_seed(seed)`**: Creates an HD wallet from a 32-byte seed. Internally, this computes the master key and chain code using HMAC-SHA512 with the key `"Bitcoin seed"` (as specified in the BIP32 standard).

2. **`get_privkey_from_path(path)`**: Derives a 32-byte private key at the given BIP32 path. The path uses `h` to indicate hardened derivation (e.g., `"m/1017h/0h/0h/0/0"`).

3. **`get_pubkey_from_path(path)`**: Derives a 33-byte compressed public key at the given path.

</details>

The `BIP32` class is already imported for you in the provided code. To create a wallet and derive keys, you simply call `BIP32.from_seed(seed)` and then use `get_privkey_from_path()` with the correct derivation path string.

## Channel Keys

Our `ChannelKeyManager` will derive the following cryptographic material from a single seed. Each key is derived at a specific BIP32 path following the scheme we learned in the previous section. Most of these are private keys called **basepoint secrets**, which will be combined with other cryptographic material to create new private keys for each Lightning transaction.

- **Funding Secret**: Secret Key
- **Revocation Basepoint Secret**: Secret Key
- **Payment Basepoint Secret**: Secret Key
- **Delayed Payment Secret**: Secret Key
- **HTLC Basepoint Secret**: Secret Key
- **Commitment Seed**: A 256-bit scalar used to generate a series of secrets **for each Lightning state**.

> **Note:** All of the above secrets are defined in [BOLT #3: Bitcoin Transaction and Script Formats](https://github.com/lightning/bolts/blob/master/03-transactions.md) *except* for the "funding" private key. We included it in our `KeyFamily`, but it doesn't have to be. For example, LDK (a popular API-driven Lightning implementation) allows developers to bring their own on-chain wallet and, therefore, their own "funding" keys. This will become more clear when we open our Lightning channel.

For 5 of the 6 secrets, we'll also compute the corresponding compressed public key (called a **basepoint**) using the `privkey_to_pubkey` helper. The 6th secret is the **commitment seed**, which has no public key. These public keys are not embedded directly within transaction outputs. Instead, they're combined with other cryptographic material to derive the actual public keys used in each output. Note that Alice, Bob, and every node on the Lightning network will have their own set of these keys. In other words, each node will have their own version of the wallet we're building right now!

## Coding Exercise

To complete this exercise, implement the `__init__` constructor of the `ChannelKeyManager` class in the code editor below. The constructor takes a `seed` (bytes) and a `channel_index` (which defaults to 0), representing a unique index for each Lightning channel that we open.

Inside the constructor, create a BIP32 wallet from the seed and derive 6 channel secrets at the paths `m/1017h/0h/{family}h/0/{channel_index}`, where each **key family** corresponds to a different type of Lightning key. For 5 of the 6 secrets, compute the corresponding compressed public key using `privkey_to_pubkey`. The result is a `ChannelKeyManager` instance with all of these values stored as instance attributes (e.g., `self.funding_key`, `self.funding_pubkey`, `self.revocation_basepoint_secret`, `self.revocation_basepoint`, etc.).
