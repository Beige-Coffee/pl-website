# HTLC Fees & Dust

## The Cost of HTLCs

Each HTLC in a commitment transaction adds complexity and cost. The commitment transaction grows larger (more outputs), and each HTLC requires a second-stage transaction (HTLC-timeout or HTLC-success) to resolve. Understanding these costs is essential for implementing correct fee calculations as specified in BOLT 3.

## Commitment Transaction Weight

The commitment transaction fee is calculated from its **weight** in weight units. BOLT 3 defines the weight calculation as:

### Base Weight

The base commitment transaction weight (with no HTLCs) is **724 weight units**. This includes:

```text
Transaction overhead:
  version:              4 bytes * 4  =   16 wu
  segwit marker+flag:   2 bytes * 1  =    2 wu
  input count:          1 byte  * 4  =    4 wu
  input (funding):      41 bytes * 4 =  164 wu
  output count:         1 byte  * 4  =    4 wu
  locktime:             4 bytes * 4  =   16 wu
  witness header:       variable     =    2 wu

Outputs (to_local + to_remote):
  to_local (P2WSH):    43 bytes * 4  =  172 wu
  to_remote (P2WPKH):  31 bytes * 4  =  124 wu

Witness:
  funding witness:                    =  220 wu

Total:                                   724 wu
```

Note: SegWit transactions count witness data at 1/4 the weight of non-witness data. Non-witness bytes count as 4 weight units each, while witness bytes count as 1 weight unit each.

### Per-HTLC Weight

Each HTLC output adds **172 weight units** to the commitment transaction:

```text
HTLC output:  43 bytes * 4 = 172 wu
```

This is the P2WSH output that wraps the HTLC script (32-byte script hash plus the standard P2WSH overhead).

## Second-Stage Transaction Weights

The second-stage transactions (HTLC-timeout and HTLC-success) also have defined weights:

### HTLC-Timeout Weight: 663 weight units

```text
Transaction overhead + input + output:
  Non-witness:    ~113 bytes * 4 = 452 wu
  Witness data:                  = 211 wu
  Total:                           663 wu
```

### HTLC-Success Weight: 703 weight units

```text
Transaction overhead + input + output:
  Non-witness:    ~113 bytes * 4 = 452 wu
  Witness data:                  = 251 wu
  Total:                           703 wu
```

The HTLC-success transaction is 40 weight units heavier than HTLC-timeout because its witness includes the 32-byte payment preimage, while the timeout witness has an empty preimage field.

## Fee Calculation

The commitment transaction fee is:

```python
def commitment_fee(feerate: int, num_untrimmed_htlcs: int) -> int:
    """Calculate the commitment transaction fee in satoshis.

    feerate: satoshis per 1000 weight units
    num_untrimmed_htlcs: number of HTLC outputs that are NOT trimmed (above dust)
    """
    weight = 724 + 172 * num_untrimmed_htlcs
    return feerate * weight // 1000
```

The feerate is expressed in **satoshis per 1000 weight units** (sometimes written as sat/kw). This is negotiated between the channel parties and can be updated during the channel's lifetime.

## Dust and Trimming

### What Is Dust?

A dust output is one whose value is too small to be economically spent. Creating dust outputs is wasteful: they bloat the UTXO set and can never be profitably spent. Bitcoin Core nodes refuse to relay transactions with dust outputs.

### HTLC Trimming

BOLT 3 defines a more nuanced dust calculation for HTLCs. An HTLC is **trimmed** (excluded from the commitment transaction outputs) if its value is less than the cost to claim it via the second-stage transaction plus the dust limit:

```python
def is_htlc_trimmed(
    htlc_type: str,  # "offered" or "received"
    amount_msat: int,
    feerate: int,
    dust_limit_sat: int
) -> bool:
    """Determine if an HTLC should be trimmed from the commitment transaction."""
    amount_sat = amount_msat // 1000

    if htlc_type == "offered":
        # HTLC-timeout weight is 663
        htlc_fee = feerate * 663 // 1000
    else:
        # HTLC-success weight is 703
        htlc_fee = feerate * 703 // 1000

    return amount_sat < dust_limit_sat + htlc_fee
```

### What Happens to Trimmed HTLCs?

Trimmed HTLCs are not forgotten -- they are still tracked in the channel state. They simply are not included as outputs in the commitment transaction. Their value is effectively added to the transaction fee.

This means that if a commitment transaction is broadcast while trimmed HTLCs are pending, those HTLC amounts go to the miner. Both parties accept this risk because the amounts are, by definition, too small to be worth claiming through a second-stage transaction.

### Example

Suppose:
- Feerate: 15,000 sat/kw (15 sat per weight unit)
- Dust limit: 546 sat
- HTLC amount: 5,000 sat

For an offered HTLC:
```text
HTLC-timeout fee = 15000 * 663 / 1000 = 9,945 sat
Trim threshold = 546 + 9,945 = 10,491 sat
5,000 < 10,491 -> TRIMMED
```

For a received HTLC:
```text
HTLC-success fee = 15000 * 703 / 1000 = 10,545 sat
Trim threshold = 546 + 10,545 = 11,091 sat
5,000 < 11,091 -> TRIMMED
```

At this feerate, HTLCs below roughly 10,500-11,100 satoshis would be trimmed.

## P2WSH Wrapping

Both HTLC outputs and the to_local output are wrapped in **P2WSH** (Pay-to-Witness-Script-Hash). The reasons for this choice:

1. **Privacy**: The complex spending conditions (revocation paths, timelocks, preimage checks) are hidden until the output is spent. On-chain observers see only a 32-byte hash.

2. **Fee efficiency**: Witness data gets a 75% discount in weight calculations (1 weight unit per byte instead of 4). Since the HTLC scripts are quite large, this discount significantly reduces fees.

3. **No size limit issues**: P2WSH supports witness scripts up to 10,000 bytes, far more than the 520-byte limit of P2SH. While HTLC scripts are not that large, the headroom ensures compatibility.

## Putting It All Together

The complete fee and dust calculation for a commitment transaction:

1. **Start with the channel state**: local balance, remote balance, and a list of pending HTLCs.
2. **Determine trimmed HTLCs**: For each HTLC, calculate if it should be trimmed based on its type, amount, the current feerate, and the dust limit.
3. **Count untrimmed HTLCs**: These will become outputs in the commitment transaction.
4. **Calculate the commitment fee**: Base weight (724) plus per-HTLC weight (172) for each untrimmed HTLC, multiplied by the feerate.
5. **Deduct the fee**: Subtract the fee from the funder's output.
6. **Add trimmed HTLC values to fee**: The amounts of trimmed HTLCs are implicitly added to the transaction fee (the total inputs minus total outputs equals the fee).
7. **Check dust**: If either the to_local or to_remote output is below the dust limit after fee deduction, omit it.
8. **Sort outputs**: Apply BIP69-style sorting to all remaining outputs.

This process is deterministic: both parties can independently calculate the exact same commitment transaction, which is essential for the protocol to work.
