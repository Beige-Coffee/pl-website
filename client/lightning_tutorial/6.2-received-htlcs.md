# Received HTLCs

## The Other Side of the HTLC

A **received HTLC** is the mirror image of an offered HTLC. From the commitment transaction holder's perspective, a received HTLC represents an incoming payment -- funds that the holder can claim by revealing the payment preimage.

If Alice offered an HTLC to Bob, then in Bob's commitment transaction, that HTLC appears as a **received** HTLC. In Alice's commitment transaction, the same HTLC appears as an **offered** HTLC. The scripts are different because the timeout and claim paths are swapped.

## The Received HTLC Script

BOLT 3 specifies the received HTLC witness script as:

```text
# To remote node with revocation key
OP_DUP OP_HASH160 <RIPEMD160(SHA256(revocationpubkey))> OP_EQUAL
OP_IF
    OP_CHECKSIG
OP_ELSE
    <remote_htlcpubkey>
    OP_SWAP OP_SIZE 32 OP_EQUAL
    OP_IF
        # Exactly 32 bytes: claim path (has preimage)
        OP_HASH160 <RIPEMD160(payment_hash)> OP_EQUALVERIFY
        2 OP_SWAP <local_htlcpubkey> 2 OP_CHECKMULTISIG
    OP_ELSE
        # Below 32 bytes: timeout path (no preimage)
        OP_DROP <cltv_expiry> OP_CHECKLOCKTIMEVERIFY OP_DROP
        2 OP_SWAP <local_htlcpubkey> 2 OP_CHECKMULTISIG
    OP_ENDIF
OP_ENDIF
```

### Key Difference from Offered HTLCs

Compare the structure carefully with the offered HTLC script. The main differences are:

1. **The OP_IF/OP_ELSE branches are swapped**: In the offered script, `OP_NOTIF` (less than 32 bytes) is the timeout path and `OP_ELSE` (exactly 32 bytes) is the preimage path. In the received script, `OP_IF` (exactly 32 bytes) is the preimage path and `OP_ELSE` (less than 32 bytes) is the timeout path.

2. **CLTV is in the timeout branch**: The received HTLC script includes `OP_CHECKLOCKTIMEVERIFY` directly in the timeout branch. For offered HTLCs, the CLTV enforcement is in the HTLC-timeout transaction's locktime instead.

This difference exists because of who initiates each path:

- For **offered** HTLCs: The local party (broadcaster) reclaims via timeout. The HTLC-timeout second-stage transaction enforces the CLTV.
- For **received** HTLCs: The remote party reclaims via timeout. Since the remote party does not go through a second-stage transaction on the local party's commitment, the CLTV is enforced directly in the script.

## Spending Paths

### 1. Revocation Path

Same as offered HTLCs -- if the commitment was revoked, the counterparty sweeps with the revocation key:

```text
<revocation_sig> <revocationpubkey>
```

### 2. Preimage Path (Local Claims via HTLC-Success)

The local party (who received the HTLC) claims the funds by providing the preimage through an **HTLC-success** transaction:

```text
0 <remotehtlcsig> <localhtlcsig> <payment_preimage> <received_htlc_script>
```

The 32-byte preimage selects the preimage branch. The script verifies the hash and checks the 2-of-2 multisig.

### 3. Timeout Path (Remote Reclaims)

If the HTLC expires, the remote party can reclaim their funds. Unlike the offered HTLC timeout, the received HTLC timeout does not use a second-stage transaction on this commitment. However, on the remote party's own commitment transaction (where this HTLC appears as offered), they would use an HTLC-timeout transaction.

For the timeout spend on this commitment:

```text
0 <remotehtlcsig> <localhtlcsig> <> <received_htlc_script>
```

The empty value selects the timeout branch. The script checks `OP_CHECKLOCKTIMEVERIFY` to ensure the HTLC has expired.

## The HTLC-Success Transaction

The **HTLC-success** transaction is the second-stage transaction that the local party uses to claim a received HTLC after obtaining the preimage. It mirrors the HTLC-timeout transaction but with key differences:

| Field | HTLC-Timeout | HTLC-Success |
|-------|-------------|--------------|
| Input | Offered HTLC output | Received HTLC output |
| Locktime | `cltv_expiry` | `0` |
| Witness preimage | Empty `<>` | 32-byte `<preimage>` |
| Purpose | Reclaim expired funds | Claim with preimage |

### HTLC-Success Transaction Structure

```python
def build_htlc_success_tx(
    commitment_txid: bytes,
    htlc_output_index: int,
    amount_sat: int,
    revocation_pubkey: bytes,
    to_self_delay: int,
    local_delayed_pubkey: bytes,
    fee_sat: int
):
    """Build an HTLC-success transaction."""
    # Output: same revocable delay script as to_local
    output_script = to_local_script(revocation_pubkey, to_self_delay, local_delayed_pubkey)
    output_amount = amount_sat - fee_sat

    return {
        'version': 2,
        'inputs': [(commitment_txid, htlc_output_index, 0x00000000)],
        'outputs': [(output_amount, script_to_p2wsh(output_script))],
        'locktime': 0,  # No timelock -- can broadcast immediately with preimage
    }
```

The critical difference: the locktime is `0`. The HTLC-success transaction can be broadcast as soon as the commitment transaction is confirmed and the preimage is known. There is no waiting for a CLTV expiry because the local party is claiming the HTLC, not timing it out.

### HTLC-Success Output

Like the HTLC-timeout, the HTLC-success output uses the revocable delay script:

```text
OP_IF
    <revocationpubkey>
OP_ELSE
    <to_self_delay>
    OP_CHECKSEQUENCEVERIFY
    OP_DROP
    <local_delayedpubkey>
OP_ENDIF
OP_CHECKSIG
```

This ensures the revocation mechanism extends to HTLC claims as well. Even after the local party claims an HTLC with the preimage, there is still a CSV delay before they can spend the funds, allowing the counterparty to use the revocation key if the commitment was revoked.

### HTLC-Success Witness

The witness for the HTLC-success transaction input:

```text
0 <remotehtlcsig> <localhtlcsig> <payment_preimage> <received_htlc_script>
```

The 32-byte payment preimage is the key element that distinguishes this from the timeout path. The script verifies `HASH160(preimage) == payment_hash_ripemd160`.

## The Complete HTLC Picture

Let us summarize how offered and received HTLCs work together:

### Alice pays Bob (Alice offered, Bob received)

**On Alice's commitment** (offered HTLC):
- Bob claims with preimage (via HTLC-success on his own commitment, or directly with revocation)
- Alice reclaims after timeout (via HTLC-timeout transaction with locktime = cltv_expiry)

**On Bob's commitment** (received HTLC):
- Bob claims with preimage (via HTLC-success transaction with locktime = 0)
- Alice reclaims after timeout (directly in script with OP_CHECKLOCKTIMEVERIFY)

### Both Commitments Are Consistent

Although the scripts differ between offered and received versions, the economic outcome is the same: the person with the preimage gets paid, and if the HTLC expires, the funds return to the offerer.
