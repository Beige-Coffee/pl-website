# Inspecting the HTLC Timeout Transaction

Let's take a closer look at the **HTLC Timeout Transaction**, the second-stage transaction that allows the HTLC offerer (Alice) to reclaim her funds after the HTLC expires. As we learned earlier, this transaction is necessary because we can't combine an absolute timelock (CLTV for the HTLC expiry) and a relative timelock (CSV for the `to_self_delay`) in the same output script without creating problems for the payment route's timing.

Let's walk through the decoded output and examine each part of this transaction.

## Locktime

```
"locktime": 502,
```

The **locktime** is one of the most important fields in the HTLC Timeout Transaction. Unlike the commitment transaction (where the locktime encodes the obscured commitment number), here the locktime serves its traditional purpose: it represents the **`cltv_expiry`**, which is the block height at which the HTLC expires.

In this example, the locktime is 502 (hex `0x01f6`), meaning this transaction cannot be included in any block before block height 502. This is how the absolute timelock is enforced. Alice cannot reclaim her HTLC funds until this block height has passed.

<details>
  <summary>Why is the locktime so important for the HTLC Timeout Transaction?</summary>

The locktime enforces the `OP_CHECKLOCKTIMEVERIFY` (CLTV) expiry that both parties agreed upon when the HTLC was created. Without this locktime, Alice could potentially reclaim her HTLC funds before the agreed-upon expiry, which would be disastrous for the payment route.

Consider our earlier example: Alice routes a payment through Bob to Dianne. Bob forwards the HTLC to Dianne with an expiry at block 180, and Alice's HTLC with Bob expires at block 200. If Alice could expire her HTLC early (say, at block 170), Bob would lose the funds he forwarded to Dianne. The locktime on the HTLC Timeout Transaction prevents this scenario entirely, as Alice physically cannot broadcast this transaction until the agreed-upon block height.

By requiring the locktime to match the `cltv_expiry`, and by requiring Bob's signature on the HTLC Timeout Transaction (via the 2-of-2 multisig), both parties can be confident that the HTLC expiry terms will be honored.

</details>

## Input

```
"vin": [
  {
    "txid": "2b887d4c1c59cd600144a1e2f971d168437db453f841f2fefb2c164f28ff84ab",
    "vout": 1,
    "scriptSig": {
      "asm": "",
      "hex": ""
    },
    "sequence": 0
  }
]
```

The HTLC Timeout Transaction has a single **input** that spends the HTLC output from the commitment transaction. Let's break this down:

- **txid**: This is the transaction ID of the commitment transaction that contains the HTLC output we're spending from.
- **vout**: The output index of the HTLC output within the commitment transaction. In this case, it's index 1, which makes sense since outputs are sorted by value and the HTLC output is typically not the smallest or largest.
- **scriptSig**: Empty, as expected for a SegWit transaction. The witness data (signatures and script) will be provided in the witness field when the transaction is finalized.
- **sequence**: Set to `0`. This is notable because, unlike the commitment transaction input (where the sequence encodes part of the obscured commitment number), the HTLC Timeout Transaction uses a plain sequence value of 0. This value does not enable any relative timelocks on the input itself.

You'll also notice that the **witness** is empty (or missing from the decoded output). That's because this is an unsigned transaction. When we finalize the HTLC Timeout Transaction using the `finalize_htlc_timeout` function, we'll add the witness data, which includes the dummy `OP_0` element, Bob's pre-signed HTLC signature, Alice's HTLC signature, an empty byte string (to select the timeout path), and the full offered HTLC witness script.

## Output

```
"vout": [
  {
    "value": 0.00005000,
    "n": 0,
    "scriptPubKey": {
      "asm": "0 4adb4e2f00643db396dd120d4e7dc17625f5f2c11a40d857accc862d6b7dd80e",
      "type": "witness_v0_scripthash"
    }
  }
]
```

The HTLC Timeout Transaction has a **single output**, and it's a P2WSH output. This output uses the same **`to_local` script** structure that we see in the commitment transaction's `to_local` output. It contains two spending paths:

1. **Revocation Path**: Bob can immediately claim the funds if he has the revocation private key for this commitment state. This protects Bob in case Alice broadcasts an old commitment transaction and its associated HTLC Timeout Transaction after they've moved to a new channel state.
2. **Delayed Path**: Alice can claim the funds after waiting `to_self_delay` blocks. This is the CSV-enforced relative timelock that gives Bob enough time to check whether Alice is cheating.

Notice that the output value (0.00005000 BTC, or 5,000 satoshis) is less than the original HTLC amount. The difference is the transaction fee, which is deducted from the HTLC amount. As we learned earlier, for channels that don't support anchor outputs or zero-fee commitments, the fee must be included upfront when the transaction is created.

## Putting It All Together

Let's recap the full lifecycle of an HTLC timeout:

1. Alice and Bob add an HTLC to their commitment transactions, and they exchange signatures for the HTLC Timeout Transaction at the same time.
2. If the HTLC is not resolved off-chain (either fulfilled or failed) before the `cltv_expiry` block height, Alice can broadcast the commitment transaction.
3. Once the commitment transaction is confirmed and the `cltv_expiry` block height is reached, Alice can broadcast the HTLC Timeout Transaction.
4. After the HTLC Timeout Transaction is confirmed, Alice must wait an additional `to_self_delay` blocks before she can spend the output using the delayed path.

This two-stage process cleanly separates the absolute timelock (enforced by the HTLC Timeout Transaction's locktime) from the relative timelock (enforced by CSV in the `to_local` output script), solving the dilemma we discussed earlier.
