# Onion Routing 101

In the last chapter we worked out how Alice can derive a shared secret with each hop in the payment path. Before we put those secrets to work, we need to step back and review the protocol that uses them: onion routing.

Lightning uses the **Sphinx** onion-routing construction. Sphinx is a refinement of earlier onion-routing schemes that fixes two problems they had: packet sizes leaking route length, and headers growing with the route. Sphinx fixes the first by making every packet exactly 1,366 bytes (we'll dig into why in chapter 7), and the second with a compact header that's possible because every per-hop key derives from a single ephemeral Diffie-Hellman operation, exactly the construction you worked through in chapter 4.

The rest of this chapter walks through how Sphinx works at a high level. The goal is intuition: enough to confidently dig into the details and implement the protocol yourself in later chapters.

Each hop in the route gets its own **hop payload** inside the onion: a small block of bytes that tells that hop what to do next (forward how much, when to expire, which channel to send on). The hop payload's contents follow the **TLV (Type-Length-Value)** format, where every field gets a type byte, a length, and the value bytes. You'll see this format throughout the chapter; we'll come back to the byte-level details in chapter 7.

## Encryption is XOR with a keystream

First things first, encryption. In earlier sections we saw how Alice can use a shared secret to encrypt payment information so that only Bob can read it. Now let's review what that actually means.

Sphinx uses a stream cipher (specifically ChaCha20), which produces a deterministic stream of pseudo-random bytes given a key. The "deterministic" part is what makes this useful: Alice and Bob can each independently generate the *exact same* keystream from the same key, without ever having to send the keystream itself across the wire.

Once we have the keystream, encryption is just **XOR-ing** the plaintext with that keystream byte-by-byte. The neat thing is, the same operation works both ways. Alice XORs plaintext with the keystream to encrypt; Bob XORs the ciphertext with the *same* keystream to decrypt. The keystream cancels itself out. This is why Alice and any intermediate node only need to share one secret for encryption to work in both directions.

## The anatomy of an onion packet

Before we walk through how Alice builds the onion, let's review what's in the packet that she actually sends. Every Sphinx onion packet has three parts:

<onion-packet-anatomy></onion-packet-anatomy>

- **HEADER** holds the **version byte** (currently `0x00`) and the **ephemeral public key** for the current hop. The ephemeral pubkey is what each forwarder combines with their own node private key (via ECDH) to derive their shared secret with Alice. When Bob receives the packet, he uses the ephemeral pubkey from the header to derive `ss_AB`, and from that secret he derives every key he needs to peel his layer. Without the header, no shared secret can be derived, and the forwarder can't do anything with the rest of the packet.

- **PAYLOAD AREA** carries the per-hop instructions, wrapped in one layer of encryption per hop in the route. Each forwarder strips one layer off the entire buffer, exposing their hop payload at the front while the inner layers still cover the rest.

- **HMAC TAG** is a 32-byte authentication tag covering the payload area. HMAC (short for *Hash-based Message Authentication Code*) is a cryptographic primitive that combines a secret key with the message bytes to produce a fixed-size tag, which only someone holding the key can generate or verify. Before any decryption, each hop computes its own tag over the payload using a key derived from its shared secret with Alice, and checks it against the one in the packet. If they don't match, the packet was either altered or built wrong, and the receiving node rejects it immediately. (For a deeper dive into HMAC and how it's constructed, see the cryptographic primitives chapter of the Noise Protocol course.)

## Alice builds the onion

Now that we have an understanding of an onion packet, let's see how Alice constructs one for the route. Before reading on, ask yourself: whose payload goes into the onion first? Does she start with Bob (the first hop) or with Dave (the last hop)?

Since onions are peeled hop by hop along the payment path, the outermost layer is what the first hop sees, and it gets peeled first. The **innermost** layer is peeled last, but it has to be *added* first, because Alice builds each layer on top of the prior one, working from the inside out.

One crucial reason the packet is built in reverse is how the **per-hop HMACs nest into each other**. Each hop payload carries a 32-byte HMAC field that commits to the *next inner* layer. To understand why that matters, recall that every intermediate hop has its own shared secret with Alice (from chapter 4). The HMAC is what each hop uses to **authenticate** the packet they received: using a key derived from their shared secret, they recompute the HMAC over the encrypted payload and compare it to the one in the packet. The two match *if and only if* Alice built the packet honestly and no one tampered with the bytes in transit.

The catch is that Alice is the only party who can compute these HMACs ahead of time, because she's the only one with all three shared secrets. So she has to **bake every hop's HMAC into the onion in advance**, arranged so that each hop can find the next one inside their own decrypted hop payload. Bob's outer HMAC is `bob_hmac`; inside Bob's hop payload sits `charlie_hmac`; inside Charlie's hop payload sits `dave_hmac`. Each hop verifies their own outer HMAC, then extracts the next one and elevates it to the outer tag of the packet they forward.

That's why construction has to run inside-out. Bob's HMAC commits to Charlie's encrypted layer, which itself commits to Dave's. Each HMAC depends on the inner layers existing first, so the only order that works is innermost first, outermost last.

## Each hop peels its layer

Great, we've seen how an onion is built! Now, let's review how an onion is peeled! Broadly speeking, each hop performs the same three actions when they receive a payload:

1. **Verify the outer HMAC**: First, they will recompute the HMAC over the encrypted hop payload and compare against the packet's HMAC tag. If they don't match, they will drop the packet.
2. **Decrypt with XOR**: If the HMAC's match, they will XOR the hop payload with this hop's encryption keystream. This is the same operation Alice used to encrypt this payload! As we learned earlier, XORing it again with the same keystream will decrypt it.
3. **Read the hop payload at the front**: FInally, they will parse the TLV (amount, expiry, next channel, etc.) and extract the next-hop's HMAC. Then, they'll construct the new 1,300-byte packet, append the header and HMAC, and send it to the next hop.

Pay close attention to what happens at each XOR step... the hop's encryption layer comes off **every region** in the buffer, *not just the current hop payload*. More specifically, when Bob XORs his payload, it reveals his plaintext TLV, but Charlie's and Dave's are still encrypted within deper layers. Once Bob forwards the payload to Charlie, he'll decrypt and, once again, only see his payload, with Dave's remaining hidden.

When the packet finally reaches Dave, his hop payload's HMAC field contains 32 zero bytes. Per **BOLT 4**, an all-zero next-hop HMAC signals that the receiving node is the final destination: there's no next hop to verify against, and the payload is meant to be consumed in place. Dave checks the payment hash, verifies the amount, and claims the funds.

## How the per-hop HMACs hold it together

In case we haven't hammered the point enough yet, understanding the role the HMACs play is crucial to understanding onion routing, as each hop's HMAC commits to all layers within it. Below is a review of the **per-hop HMACs** from our example, outermost to innermost:

- The packet's outer HMAC is `bob_hmac`. Since this is the last HMAC Alice derives when creating the onion, it commits to all other payload data within the onion.
- Inside Bob's hop payload sits `charlie_hmac`, the tag Charlie will verify after Bob forwards.
- Inside Charlie's hop payload sits `dave_hmac`, visible only after Bob decrypts and forwards.
- Inside Dave's hop payload, the HMAC field is `0x00…` (32 zeros), signaling that Dave is the final hop.

Each one commits to *everything* below it in the onion: the hop payload, the next-inner HMAC, and all the encrypted layers underneath. A forwarder can't tamper with what's below their own hop payload without invalidating the HMAC the *next* forwarder will check. That's what makes onion routing safe in the presence of malicious intermediate hops: any modification anywhere in the inner layers gets detected at the next hop, and the packet is dropped.

## A small confession

Sorry in advance, but throughout this chapter we've been quietly abstracting away all of the keys that are used in the Sphinx protocol, leading the reader to believe all the magic happens with just the shared secret. In reality, Sphinx uses distinct named keys with specific roles: the forward and return-path per-hop keys are derived from each shared secret, while the `pad` key is derived from Alice's session key. If we reused one raw secret for all of the encryption and authentication jobs, we'd be opening ourselves up to some pretty bad attack vectors.
