# Peeling a Layer

In the last section, we implemented the code to build a <g>Sphinx</g> onion packet from a given route. Now, let's switch our perspective and learn how to peel an onion packet and get it ready to forward to the next hop. This chapter covers the **peel** from Bob's perspective, focusing on which bytes he extracts, what gets decrypted, and how he builds the next packet to send to Charlie. Chapter 10 will add the policy checks (<g>TLV</g> validation, fee/<g>CLTV</g> thresholds) and stitches the full processing loop together.

## What Bob already knows

Before we get too far, we should probably set the scene by reviewing what Bob actually knows. Since we've reviewed this payment example already from many different perspectives, it's easy to assume Bob knows more than he actually does.

First, we'll imagine Bob's been running his node for a few months. He's not a novice, but he's not a pro either. He has his long-lived **node private key** (`bob_privkey`) and a channel with Alice, but that's about it. He does **not** yet have any of the per-payment keys we've been working with throughout the course:

- No `ss_AB`, the 32-byte <g>ECDH</g> shared secret with Alice for this payment.
- No `rho_B`, `mu_B`, `um_B`, or `ammag_B` (the four per-hop keys, all derived from `ss_AB`).
- No knowledge of Alice's ephemeral pubkey `E_AB`. At least, not yet!
- No idea Alice is the original sender, or that Charlie and Dave exist downstream.

At some point, Bob receives an `update_add_htlc` from Alice carrying the 1,366 bytes of onion data. It's game on. But look back at the list above: he has his node private key and a channel with Alice, and *none* of the per-payment keys he needs to read the packet. So where does he start?

With the one thing Alice left in plain sight. The first 33 bytes of the packet (after the version byte) are <m>E_AB</m>, the ephemeral pubkey Alice generated for Bob as the first link in the **blinding chain** from chapter 4. Bob performs ECDH between his node private key and <m>E_AB</m> and recovers the very same `ss_AB` Alice used when she wrapped his layer. That one shared secret is the seed: every other key Bob needs (`rho_B`, `mu_B`, `um_B`, `ammag_B`) derives from it.

`ss_AB = SHA256(bob_privkey · E_AB)`

## Peel, step by step

Now, the moment you've been waiting for, let's review each step and look at some code!

#### Step 1. Parse the four packet fields
First, when Bob get's receives the 1,366 bytes on the wite, he'll divide them into the following four fixed-size fields:

- **`version`** (1 byte): must be `0x00`. Any other value gets rejected.
- **`ephemeral_pubkey`** (33 bytes): the <m>E_AB</m>. This is Alice's ephemeral public key, which Bob will perform ECDH with, using his node private key.
- **`hop_payloads`** (1,300 bytes): the encrypted blob Bob will decrypt in step 4.
- **`outer_hmac`** (32 bytes): the integrity tag Bob will verify in step 3.

<python-snippet id="peel-parse"></python-snippet>

#### Step 2. Derive `mu_B` and `rho_B` from `ss_AB`
Next, Bob will derive the below two keys, which he'll need for verifying and decrypting the message:

- `mu_B = HMAC('mu', ss_AB)`: 32-byte HMAC key for verifying the outer HMAC tag.
- `rho_B = HMAC('rho', ss_AB)`: 32-byte ChaCha20 key for the decryption keystream.

<python-snippet id="peel-derive-keys"></python-snippet>

#### Step 3. Verify the outer HMAC
Before Bob decrypts anything, he'll recompute the <g>HMAC</g> tag over the encrypted `hop_payloads` concatenated with `associated_data` (the 32-byte `payment_hash` from the <g>HTLC</g>). If it matches the `outer_hmac` from the packet, then Bob is able to validate the integrity of the message. If they don't match, then the packet may have been tampered with, so Bob rejects and sends back an <g>error onion</g> (chapter 11).

<python-snippet id="peel-verify-hmac"></python-snippet>

#### Step 4. Extend the buffer and XOR over 2,600 bytes
Next, Bob will append 1,300 zero bytes to the end of `hop_payloads`, making a 2,600-byte working buffer. From here, he'll generate a keystream using his `rho_B` key and XOR it across the full 2,600 bytes.

The front 1,300 bytes are now decrypted. Alice used the same XOR with the same `rho_B` key when she built Bob's layer, so Bob's XOR strips it cleanly, and he can read his hop payload at the start. The trailing 1,300 bytes are pure keystream (`0 ⊕ rho_B[1300:2600] = rho_B[1300:2600]`), but here's the catch: **Bob won't end up using most of them**. In step 6, he'll slice out Charlie's `hop_payloads` starting at byte `s_B` (right after his own hop payload) and running 1,300 bytes, so only the first `s_B` bytes of the extension (≈60 in our example) actually land in Charlie's view. The remaining ~1,240 bytes get computed and discarded.

So why generate the full 1,300 if Bob only needs `s_B`? **Because Bob doesn't yet know `s_B`**! That's the bigsize length prefix in his still-encrypted hop payload, which he can't parse until he's decrypted the front. BOLT 4 sidesteps the chicken-and-egg problem by specifying the worst-case extension up front: enough keystream that any possible hop size is covered.

<python-snippet id="peel-extend-xor"></python-snippet>

#### Step 5. Read Bob's hop payload at the front
Now that the leading bytes are plaintext, Bob can parse out the relevant information:

- **`bigsize_LEN`** (1-9 bytes): the length-prefix telling him how long his TLV section is.
- **TLV records** (`LEN` bytes): the routing instructions (`amt_to_forward`, `outgoing_cltv_value`, `short_channel_id`).
- **`next_hmac`** (32 bytes): the HMAC tag Bob writes into the outer HMAC field of his outgoing packet (this is `charlie_hmac`, which Alice committed to when she built Charlie's layer).

> Note: `bigsize` is BOLT 1's variable-length integer encoding. For TLV-section lengths under 253 bytes (which is essentially always, for routing TLVs), it's a single byte.

<python-snippet id="peel-read-payload"></python-snippet>

#### Step 6. Lift the next 1,300 bytes as Charlie's `hop_payloads`
Now that Bob has read his own hop payload, he turns to building the packet Charlie will receive. Starting right after his hop payload, he lifts the next 1,300 bytes from the working buffer. That slice becomes Charlie's `hop_payloads`.

<python-snippet id="peel-lift-next"></python-snippet>

#### Step 7. Advance the ephemeral pubkey
If you remember, Bob used <m>E_AB</m> to derive his shared secret and process the packet. However, `E_AB` only produces a useful shared secret when Bob performs ECDH between it and his node privkey. Luckily, since Sphinx uses <g>blinding factors</g> to advance the ephemeral pubkey forward along the route, Bob can derive Charlie's ephemeral pubkey, <m>E_AC</m>, and embed it in the packet's header. Below is how he'd do it.

- **`bf_AB = SHA256(E_AB ‖ ss_AB)`**: the 32-byte scalar that blinds the ephemeral forward.
- **`E_AC = bf_AB · E_AB`**: EC scalar multiplication on the curve.

This is the same `E_AC` Alice derived in chapter 4. Charlie will perform ECDH between his node privkey and `E_AC` and recover the same `ss_AC` Alice used, no additional coordination needed!

<python-snippet id="peel-advance-ephemeral"></python-snippet>

#### Step 8. Assemble the outgoing packet
Finally, Bob will assemble the packet for Charlie:

- **`version`** stays `0x00`.
- **`ephemeral_pubkey`** = `E_AC` (advanced).
- **`hop_payloads`** = the 1,300-byte slice from step 6.
- **`outer_hmac`** = `next_hmac` from step 5 (which equals `charlie_hmac`).

Bob hands the packet to Charlie over the BOLT 8 noise transport. From Charlie's perspective the packet looks identical in shape to what Alice would have sent if Charlie were the first hop, thanks to the fixed-size privacy property.
