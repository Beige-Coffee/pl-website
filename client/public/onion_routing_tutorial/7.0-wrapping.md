# Wrapping Layer by Layer

We have everything we need to build the onion. Shared secrets per hop, five named keys per hop, a precomputed filler, and a clear picture of the 1,366-byte packet shape. This chapter brings them all together. By the end you'll have written the function that produces a complete BOLT 4 onion packet from a route plus a list of hop payloads.

The construction has one big counterintuitive move: we build the onion **in reverse**, starting from the destination and working backward toward the sender. We'll see why in a moment. We'll also meet the **HMAC chain**, the integrity construction that makes each layer commit to the layer beneath it.

## The build order: from the inside out

Lightning's onion is nested. Bob's layer wraps Carol's layer, which wraps Dave's layer. To produce that nesting, the natural order is to start with the innermost (Dave's) and work outward.

Here's why. Each layer's HMAC has to be computed over the contents of *the next layer*. Bob's HMAC is over what Carol receives. Carol's HMAC is over what Dave receives. Dave's HMAC is over... nothing, since Dave is the last hop. So the HMAC chain has a fixed direction: each hop's HMAC depends on the inner hop's contents being already built.

Concretely, the algorithm is:

1. Start with a 1,300-byte buffer. Initialize it with **pad-key noise**: ChaCha20 keystream derived from `pad_key = HMAC-SHA256(b"pad", session_key)`. (Not zeros. We'll come back to why in a moment.)
2. For each hop in **reverse order** (Dave first, then Carol, then Bob):
   - Shift the buffer right by `(this hop's payload_size + 32)` bytes. This pushes the existing contents back to make room at the front.
   - Write this hop's bigsize-prefixed TLV payload at the front. Append the 32-byte HMAC that points to the next layer (this is the `next_hmac` value we computed in the previous iteration; for Dave, it's all zeros).
   - XOR the entire 1,300-byte buffer with this hop's `rho` keystream. This is the encryption step.
   - **If this was the innermost iteration (Dave's wrap)**: overwrite the trailing `len(filler)` bytes of the buffer with the filler value we computed in chapter 6. This places the filler exactly where each forwarder will expect to see it.
   - Compute the HMAC over the buffer concatenated with `associated_data` (the 32-byte payment_hash) using this hop's `mu` key. Save this as the `next_hmac` value for the iteration that builds the layer above.
3. After the last iteration (the outermost hop), the buffer is what goes in the packet's `hop_payloads` field, and the HMAC we just computed is what goes in the packet's `hmac` field.
4. Assemble the final packet: `version || E₀ || hop_payloads || hmac`.

Two BOLT 4 details that matter and don't show up in simpler descriptions:

- **`associated_data`** is a 32-byte value (the payment_hash for an HTLC) that gets concatenated with the buffer inside every HMAC. It binds the onion to a specific payment. If a forwarder sees an onion attached to a different payment_hash than the one Alice baked in, the HMAC won't verify and the packet is rejected. Without this, an attacker who captured an in-flight onion could attach it to a different HTLC.
- **Pad-key noise** initializes the buffer instead of zeros. If we started with all zeros, the destination's decrypted view would have detectable trailing structure (the parts of the buffer no hop ever wrote to would XOR back to zero against the pad keystream). Starting with pseudorandom bytes makes the unused tail look like random noise to the destination, hiding "this packet was built for a short route."

## Why this works

The shift-right-then-write pattern places each hop's data at the front of the buffer at the time that hop is being processed. After we encrypt with `rho_i`, that data is now in encrypted form. When the actual forwarder later receives this packet and applies its `rho_i` keystream (which XORs with the same bytes), the encryption is undone and the hop sees its TLV payload at the front. From its perspective, peeling is just "decrypt and read."

The reason we go in reverse is that the inner layers have to exist before the outer layers can wrap them. Bob's HMAC commits to Carol's layer, so Carol's layer must already be built when Bob's HMAC is computed. Carol's HMAC commits to Dave's layer, so Dave's layer must be built first. The HMAC chain forces the build order.

## Walking through a 3-hop build

Let's trace this for our route Bob → Carol → Dave with payload sizes 60, 60, and 100 bytes (including HMAC). The buffer is 1,300 bytes throughout.

**Initial state.** The buffer is filled with 1,300 bytes of pad-key keystream (pseudorandom noise derived from the session key).

```
[ ... 1300 bytes of pad-key keystream ... ]
```

**Iteration 1: Dave (the innermost).** Shift right by `100`. The leading 100 bytes are about to be overwritten:

```
[ 100 bytes about to be written | 1200 bytes of shifted noise ]
```

Write Dave's bigsize-prefixed TLV payload (68 bytes) at the front. Append the 32-byte all-zero `next_hmac` (Dave is the destination; there's no further layer to commit to). The buffer's first 100 bytes now contain `dave_payload || 0x00 * 32`.

XOR the whole 1,300-byte buffer with Dave's `rho` keystream. Now overwrite the **trailing 120 bytes** of the buffer with the filler value we computed in chapter 6:

```
[ encrypted Dave layer | encrypted noise | 120 bytes of filler ]
```

Now compute the HMAC: `dave_hmac = HMAC-SHA256(dave_mu, buffer || associated_data)`. Save it as `next_hmac` for Carol's iteration.

**Iteration 2: Carol.** Shift the buffer right by `60`. The original contents shift back, and the leading 60 bytes are now garbage. Write Carol's TLV payload (28 bytes) at the front, then append `dave_hmac`. The buffer's first 60 bytes now contain `carol_payload || dave_hmac`.

XOR the whole buffer with Carol's `rho` keystream. Compute Carol's HMAC over the result concatenated with `associated_data`, using `carol_mu`. Save as `next_hmac` for Bob. (We don't overlay filler again; that step only happens on the innermost iteration.)

**Iteration 3: Bob.** Shift right by `60`. Write Bob's TLV payload (28 bytes) and append `carol_hmac`. XOR with Bob's `rho` keystream. Compute Bob's HMAC with `bob_mu` over the buffer concatenated with `associated_data`. Done.

The buffer at this point is the `hop_payloads` field of the packet. The HMAC we just computed is the packet's `hmac` field. Bob will be the first to receive this onion, and his HMAC verification will use the packet `hmac` (which is `bob_hmac`).

## What the forwarder sees

When Bob receives the packet, he'll:

1. Verify `HMAC-SHA256(bob_mu, hop_payloads || associated_data) == packet.hmac`. (Yes, because we just computed exactly that. Bob already knows the `associated_data` value because it's the payment_hash of the inbound HTLC he's about to process; the same payment_hash is on Bob's outbound HTLC to Carol, and so on through the route.)
2. XOR the `hop_payloads` field with his `rho` keystream. This undoes the encryption from iteration 3.
3. Read his TLV payload from the front. Read `carol_hmac` from bytes 28..60. Discard everything else for now.
4. Shift the buffer left by 60 bytes (to remove his own slot). Extend the trailing position with bytes from his rho keystream (more on this in chapter 8).
5. The packet he forwards to Carol has the shifted-left `hop_payloads`, the same `version` byte, the next ephemeral pubkey `E_1`, and `carol_hmac` as the `hmac` field.

When Carol receives Bob's forwarded packet, the `hmac` field is now `carol_hmac`. Her `mu` key will verify that against the `hop_payloads` she sees (concatenated with the same `associated_data`), and the chain continues.

The clever observation is that the HMAC chain is a **commitment from the outside in**. Bob's HMAC commits to Carol's layer. Once Bob has verified his HMAC, he knows the packet is authentic and unmodified. By extension, anything Bob unwraps and forwards is what Alice intended for the next hop.

<details>
<summary>Why is the HMAC over the encrypted bytes, not the plaintext?</summary>

If the HMAC were over the plaintext, the forwarder would have to decrypt before it could verify integrity. That's bad practice: the forwarder would be processing untrusted data with its `rho` keystream before knowing whether the packet was tampered with. By computing the HMAC over the encrypted bytes (the `hop_payloads` field as it appears on the wire), the forwarder can verify integrity *first* and only proceed to decrypt if verification passes.

This matches the AEAD pattern from the Noise course: encrypt-then-MAC, with the MAC verified before any further processing. (See the [HKDF chapter](/noise-tutorial/crypto-primitives) for the underlying motivation.)

## The first wrap: where filler matters

The `if i == innermost: overwrite trailing bytes with filler` step in iteration 1 is the filler kicking in. Here's why it works AFTER the rho XOR rather than before.

When Dave's `rho` keystream XORs the buffer, it XORs *everything*: Dave's payload, the previously-noise-filled trailing positions, all of it. After that XOR completes, we directly overwrite the trailing `len(filler)` bytes with the filler value. From this point onward, those trailing bytes are the exact bytes that need to appear at the back of each forwarder's view of the buffer (the filler we computed in chapter 6 was designed to be those bytes).

Why not pre-place the filler in the initial buffer and let the wraps propagate it? Because Dave's right-shift drops the last `slot_dave` bytes off the back of the buffer. If `slot_dave` is large (say, the destination payload is much bigger than the forwarder payloads), those dropped bytes can include the filler we tried to pre-place. The filler ends up at the wrong offset and the HMAC chain breaks.

By overwriting the trailing positions *after* Dave's XOR, we put the filler exactly where it needs to be regardless of `slot_dave`.

After Dave's wrap and the filler overwrite, when the packet flows along the route, the trailing positions of each downstream hop's view are exactly the cumulative XOR of subsequent hops' rho keystreams over the filler. The chapter-6 filler algorithm precomputed values that make all those XORs collapse to the right bytes for each hop's HMAC to verify.

## Wrap one hop, then loop

The construction has two pieces. First, a per-hop function `wrap_hop` that does the shift-write-XOR-HMAC dance for a single iteration. Second, a top-level `build` function that runs `wrap_hop` for each hop in reverse order, threading the `next_hmac` through the loop, and finally assembles the 1,366-byte packet.

The first exercise implements `wrap_hop`. The second implements `build` and uses your earlier `derive_shared_secrets`, the per-hop key derivation, your `generate_filler`, and now `wrap_hop` to produce a complete spec-compliant packet.
