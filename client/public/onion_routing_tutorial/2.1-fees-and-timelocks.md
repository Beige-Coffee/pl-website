# Fees and Timelocks

Alice has her route: Alice -> Bob -> Carol -> Dave. She knows each node's fee policy. Now she needs to calculate the exact amount and timelock for every hop.

Here's the key insight: **route construction works backward**, starting from the final hop (Dave) and working toward Alice. This might feel counterintuitive at first, but by the end of this chapter you'll see why it's the only way that makes sense.

## Why Backward?

Each forwarding node charges a fee based on the amount it forwards. To know how much Bob needs to receive, we first need to know how much he sends to Carol. To know that, we need to know Carol's fee. And Carol's fee depends on how much she sends to Dave.

In other words, every hop's incoming amount depends on the downstream hops. The only place we can start without any dependencies is the end: Dave, the final recipient. From there, we work backward hop by hop, adding fees and timelocks as we go.

<details>
  <summary>Why does route construction work backward?</summary>

Because each hop's fee depends on the amount it forwards. To know what Bob must receive, we first need to know what he forwards to Carol. To know that, we need to know Carol's fee. To know Carol's fee, we need Dave's amount. So we start at Dave and work backward. If we tried to go forward from Alice, we'd be stuck in a circular dependency: Alice's amount depends on Bob's fee, which depends on Carol's amount, which depends on Carol's fee, which depends on Dave's amount. Working backward breaks this chain.
</details>

## The Fee Formula

Every forwarding node uses the same fee formula, defined in BOLT 7. The node receives `amount_forwarded + fee`, and forwards `amount_forwarded` to the next hop.

## CLTV Timelock Calculation (Backward)

Fees aren't the only thing Alice needs to calculate backward. She also needs to set the **CLTV timelocks** for each hop.

<details>
  <summary>What are CLTVs and why do they matter?</summary>

CLTV stands for **CheckLockTimeVerify**, a Bitcoin Script opcode that prevents funds from being spent until a certain block height. In Lightning, each HTLC (Hashed Time-Locked Contract) has a CLTV expiry. This gives each hop a window of time to claim the payment by revealing the preimage. If the next hop doesn't reveal the preimage before the CLTV expires, the current hop can reclaim their funds on-chain. Each forwarding node adds a `cltv_expiry_delta` to ensure it has enough time between its incoming and outgoing timelocks. The further from the destination, the longer the timelock.
</details>

Each forwarding node requires a gap between the CLTV on its incoming HTLC and the CLTV on its outgoing HTLC. This gap is the `cltv_expiry_delta` from the node's routing policy. It gives the node enough time to claim funds if something goes wrong downstream.

**Summary:**

| Hop | Incoming CLTV | Outgoing CLTV | Delta |
|-----|---------------|---------------|-------|
| Alice -> Bob | - | 700,088 | - (sender) |
| Bob -> Carol | 700,088 | 700,048 | 40 |
| Carol -> Dave | 700,048 | 700,018 | 30 |
| Dave | 700,018 | - (recipient) | - |

Notice how the CLTVs decrease at each hop. This is by design: each forwarding node has a safety window to claim funds on-chain if the downstream node stalls. Bob has 40 blocks of breathing room, Carol has 30 blocks. The further upstream (closer to the sender), the longer the timeout.
