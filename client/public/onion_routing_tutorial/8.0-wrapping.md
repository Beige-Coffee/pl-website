## Wrap, step by step

Now that we've got a solid intuition, let see how we can translate the visual into an algorithm we can code. Before Alice adds any payload data to the onion packet, she'll need to initialize a 1,300-byte buffer with `pad_key`. To do this, she derives a `pad_key` from her <g>session key</g> and uses it as a ChaCha20 key to generate 1,300 bytes of pseudorandom noise.

<python-snippet id="wrap-pad-init"></python-snippet>

<details>
<summary>Why noise instead of zeros?</summary>

If she started with zeros, then the destination's decrypted view would decrypt to a sequence of zeros, which would leak information about how long the route was! By starting with pseudorandom bytes, the destination would be unable to decipher how long the route was.

</details>

Now that the buffer is initialized, Alice can iterate over each hop (in reverse order: Dave → Charlie → Bob). Each iteration will follow the same algorithm below. One quick heads-up before we start: when we compute the <g>HMAC</g> in Step 5, we'll concatenate the buffer with a 32-byte value called `associated_data` (the `payment_hash` for this HTLC). It binds the onion to a specific payment, and we'll dig into the "why" in Step 5.

#### Step 1. Right-shift the buffer
Before adding any new bytes to the buffer, Alice needs to make room at the front for the new hop's data, without growing the buffer past 1,300 bytes. She does this with a **right-shift** by `hop_size = len(payload) + 32` bytes, which does two things at once:

- The last `hop_size` bytes fall off the end (they're permanently dropped).
- `hop_size` bytes of placeholder space appear at the front (they will be overwritten in step 2).

> Note: `payload` is the bigsize-prefixed TLV records (`bigsize_LEN ‖ TLV_bytes`), so `len(payload)` already includes the LEN prefix; the `+ 32` accounts for the HMAC field that comes after.

<python-snippet id="wrap-shift"></python-snippet>

#### Step 2. Write this hop's payload at the front
Next, Alice writes the bigsize-prefixed TLV records into the slot in the front that just opened up, followed by the 32-byte `next_hmac`. Note, `next_hmac` is **not this hop's own HMAC**! It's the HMAC of the layer *inside* this hop, which this hop's payload commits to. Or, from an algorithmic perspective, it's the HMAC from the previous iteration. For example:

- **Dave's iteration** (innermost, runs first): `next_hmac` is 32 zero bytes. Dave is the destination; there's no inner layer for his hop payload to commit to.
- **Charlie's iteration**: `next_hmac` is `dave_hmac`, computed in Dave's Step 5 (below).
- **Bob's iteration**: `next_hmac` is `charlie_hmac`, computed in Charlie's Step 5 (below).

Each iteration's Step 5 produces *this* hop's HMAC, which then becomes the *next* iteration's `next_hmac`. That's the HMAC chain: each layer commits to the encrypted contents of the one beneath it.

<python-snippet id="wrap-write"></python-snippet>

#### Step 3. XOR the buffer with this hop's `rho` keystream
Now that Alice has added the payload and `next_hmac`, she's ready to perform some cryptography! She XORs the entire 1,300-byte buffer with the keystream generated from this hop's **rho** key (`rho_i`, derived from the shared secret `ss_AX` with the current hop). Remember, one XOR pass will both encrypts the bytes Alice just wrote AND adds a fresh encryption layer on top of everything written in inner iterations.

<python-snippet id="wrap-xor"></python-snippet>

#### Step 4. Overlay the filler (innermost iteration only)
If you're wondering where the <g>filler</g> comes in, this is it! Before Alice is ready to calculate the HMAC for this packet, she will need to overwrite the trailing `len(filler)` bytes of the buffer with the filler we built in chapter 7. Note, this only needs to happen in the innermost layer because the filler was precomputed (back in chapter 7) to anticipate every later iteration's `rho` XOR. We place it once here, and Charlie's and Bob's subsequent XOR passes transform it into the bytes each forwarder will see at the trailing positions when they peel.

<python-snippet id="wrap-filler-overlay"></python-snippet>

#### Step 5. Compute this hop's HMAC
Finally, Alice will compute the HMAC over the buffer concatenated with `associated_data` (the 32-byte `payment_hash`) using this hop's `mu` key. Save it as `next_hmac` for the iteration that builds the layer above.

<python-snippet id="wrap-hmac"></python-snippet>

The `associated_data` parameter is worth a closer look. It's the 32-byte `payment_hash` for the HTLC, concatenated with the encrypted buffer before hashing. It binds the onion to a specific payment: if a forwarder sees this onion attached to a different `payment_hash` than the one Alice baked in, the HMAC verification fails and the packet is rejected. Without `associated_data`, an attacker who captured an in-flight onion could re-attach it to a different HTLC, and forwarders would have no way to notice the swap, so they'd forward it instead of rejecting it.

After the outermost iteration finishes, the buffer is what goes in the packet's `hop_payloads` field, and the HMAC we just computed is the packet's outer `hmac` tag. Alice prepends the 1-byte version (`0x00`) and her 33-byte ephemeral pubkey `E_AB`, appends the 32-byte `bob_hmac`, and ships the 1,366-byte packet to Bob.

## Wrap one hop, then loop

Let's get to coding! For this exercise, we'll complete the following two functions.
- First, a per-hop function `wrap_hop` that runs through the steps we just reviewed for a single iteration.
- Second, a top-level `build` function that calls `wrap_hop` for each hop in reverse order, threading the `next_hmac` through the loop, applying the filler overlay on the innermost iteration, and, finally, assembling the 1,366-byte packet.

You'll write both. The first exercise (`wrap_hop`) gives you a clean primitive for one iteration. The second exercise (`build`) uses your earlier `derive_shared_secrets`, your `derive_keys`, your `generate_filler`, and your new `wrap_hop` to produce a complete spec-compliant packet. The `build` exercise comes with a structured starter code skeleton so you can focus on the substantive moves (the reverse loop and the filler overlay) rather than bookkeeping. The test suite checks your output against the official BOLT 4 onion test vector, so if your packet matches, you can be confident every detail is right.
