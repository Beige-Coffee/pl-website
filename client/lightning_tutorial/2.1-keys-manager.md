# Key Management

## Why Key Management Matters

A Lightning channel requires multiple cryptographic keys for different purposes: signing funding transactions, constructing revocable commitment outputs, creating HTLC scripts, and more. Managing these keys securely and deterministically is the foundation of everything we will build.

Rather than generating random keys for each operation, Lightning uses **hierarchical deterministic (HD) key derivation** based on BIP32. This means all keys can be regenerated from a single seed, making backup and recovery straightforward.

## The Six Key Families

BOLT 3 defines six **key families**, each serving a distinct purpose in channel operations. Every channel derives one key from each family, giving each channel a complete set of six keys.

### 1. MultiSig Key (Family 0)

The `multisig_key` is used in the 2-of-2 funding script that locks funds into the channel. Both parties contribute their multisig public key to create the funding output. This key signs commitment transactions to authorize state updates.

### 2. Revocation Base Key (Family 1)

The `revocation_basepoint` is one of two components used to derive revocation keys. The revocation mechanism is what allows old channel states to be penalized. When combined with the counterparty's per-commitment point, it produces the revocation public key used in commitment transaction scripts.

### 3. HTLC Base Key (Family 2)

The `htlc_basepoint` is used to derive keys for signing HTLC (Hashed Time-Locked Contract) transactions. HTLCs are the conditional payments that enable multi-hop routing across the Lightning Network.

### 4. Payment Base Key (Family 3)

The `payment_basepoint` is used for the `to_remote` output in commitment transactions. With `option_static_remotekey` (now mandatory), this key is used directly (not derived per-commitment), which simplifies fund recovery if a node loses its channel state.

### 5. Delay Base Key (Family 4)

The `delayed_payment_basepoint` is used to derive keys for the `to_local` output, which has a CSV (CheckSequenceVerify) delay. This delay gives the counterparty time to detect and punish cheating attempts.

### 6. Commitment Seed (Family 5)

The `commitment_seed` is the seed from which all per-commitment secrets are derived. These secrets are revealed when old states are revoked, enabling the revocation mechanism that enforces channel fairness.

## The Derivation Path

Lightning uses a specific BIP32 derivation path to generate these keys:

```text
m / 1017h / 0h / {key_family}h / 0 / {index}
```

Let us break down each level:

| Level | Value | Meaning |
|-------|-------|---------|
| `m` | root | The master key derived from the wallet seed |
| `1017h` | purpose | Registered BIP 43 purpose for Lightning (hardened) |
| `0h` | coin_type | Bitcoin mainnet (hardened). Use `1h` for testnet |
| `{key_family}h` | family | One of the 6 families: 0-5 (hardened) |
| `0` | branch | Always 0 (non-hardened) |
| `{index}` | index | Channel index, incremented for each new channel |

The `h` suffix indicates **hardened derivation**, which we will explain in the next chapter. For now, the important thing is that hardened derivation at the family level means that compromising one key family does not compromise the others.

### Example Paths

For the first channel (index 0) on mainnet:

```text
MultiSig key:        m/1017h/0h/0h/0/0
Revocation base:     m/1017h/0h/1h/0/0
HTLC base:           m/1017h/0h/2h/0/0
Payment base:        m/1017h/0h/3h/0/0
Delay base:          m/1017h/0h/4h/0/0
Commitment seed:     m/1017h/0h/5h/0/0
```

For a second channel (index 1):

```text
MultiSig key:        m/1017h/0h/0h/0/1
Revocation base:     m/1017h/0h/1h/0/1
...and so on
```

## Why This Design?

This key management scheme has several advantages:

- **Single backup**: Only the seed needs to be backed up. All keys can be regenerated.
- **Isolation**: Hardened derivation at the family level means a compromised key in one family does not reveal keys in other families.
- **Scalability**: New channels just increment the index. No new entropy is needed.
- **Standardization**: All Lightning implementations follow the same path structure, enabling interoperability.
