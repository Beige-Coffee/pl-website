# Shared Secrets per Hop

Hmm, so the last section left us with a bit of a dilemma. Even though Alice has identified the best path to route her payment through, she still doesn't have a way to encrypt the payment instructions so that only the intended audience can read them. Or, does she!?

The *secret* (pun intended) to making this work is for Alice to derive a unique shared private key for *each hop* in the payment path that. Crucially, only Alice and the unique hop will be able to derive the shared private key. By doing this, she can ensure that each message stays encrypted, and only the intended recipient is able to decrypt it. To make this trick work, we'll use a tool that we reviewed in the Noise handshake course: **Elliptic Curve Diffie-Hellman (ECDH)**.


## A quick refresher on ECDH

Recall that a public key is just a large, private number multiplied by the elliptic curve generator point: `A = a · G`. The resulting point `A` is also a point on the curve, but it has a special property such that you cannot recover `a` from `A`, even though both `A` and `G` are publicly known. To do so, you'd have to solve the **elliptic curve discrete log problem**, which is computationally infeasible.

Another neat property of elliptic curves is that scalar multiplication on the curve is **commutative**. In other words, for some mathematical operations, the order of the components does not matter. For example, `3 x 5` is the same as `5 x 3`. If we extend that example to elliptic curves, given Alice's keypair `(a, A)` and Bob's `(b, B)`, the point `a · B` equals `b · A`, because both expand to `ab · G`. This means that, **if Alice and Bob trade public keys**, they can both compute a shared point that only they know. An observer sees `A` and `B` on the wire but can't combine them into `ab · G` without one of the private numbers, which the discrete log problem keeps safely hidden. If Alice and Bob take the SHA256 hash of this point, they end up with a **shared secret**, which they can use to encrypt and decrypt information sent between them.

## First attempt: Alice reuses her node key

Now, let's connect this back to our payment example. Armed with the ECDH toolkit, **one option** Alice has is to use her Lightning node's existing keypair. She'd take her node private key and combine it with each forwarder's node public key (already broadcast on the gossip network) to derive a shared secret with that forwarder.

Then she'd embed her own node public key at the top of the packet. Each forwarder reads it, runs ECDH against it with their own node's private key, and derives the same shared secret. With that secret, they can decrypt the slice meant for them.

The lesson: any reused, identity-tied key is a non-starter for the ephemeral role. Alice needs a **fresh, identity-detached keypair per payment**. Let's try that next.

## Second attempt: a fresh ephemeral key per hop

So our first attempt wasn't ideal. Let's see if we can do better!

Instead of using her Lightning node's private key, Alice can generate a fresh, throwaway keypair for each hop in the payment path. We call these **ephemeral** keypairs because they are short-lived, as they are only ever used for this specific payment.

For our example, Alice would generate one `(ephemeral private key, ephemeral public key)` pair per forwarder: `(e_Bob, E_Bob)`, `(e_Charlie, E_Charlie)`, and `(e_Dave, E_Dave)`. With the keypairs in hand, she runs ECDH against each hop's Lightning node public key (same as in attempt one) to derive a shared secret with that hop. Then she embeds `E_Bob` into Bob's slot, `E_Charlie` into Charlie's slot, and `E_Dave` into Dave's slot, alongside the encrypted payload for each hop.

Each hop reads its own ephemeral pubkey at the front of its slot, runs ECDH against it with its own node private key, and recovers the matching shared secret. The identity leak from the first attempt is now gone: the ephemeral keys aren't tied to any Lightning node. Yay!

## Third (and last!) attempt: a single ephemeral key, advanced by blinding

Okay, enough funny business. Let's review how things actually work in the Lightning Network.

Instead of creating a unique ephemeral keypair for each hop, Alice will generate **one** keypair, called the **session key**. This keypair is the starting point for a chain of key derivations. Each hop will deterministically derive its shared secret from the session key plus information that accumulates as the payment routes through the network. This way, only the *first* ephemeral key ever needs to be embedded in the packet. For every subsequent hop, the *next* hop's ephemeral key is computed on the fly by the current hop and written into the packet before forwarding.

This is going to get a little gnarly, so buckle up!

Alice begins by generating a random one-time **session key**. Then, for each hop in the route, she derives an ephemeral private key, an ephemeral public key, a shared secret with that hop, and (for every hop except the final one) a **blinding factor** that the next hop will use to advance the chain. Let's walk it through.

For Bob (hop 0):
- Alice sets the ephemeral private key equal to the session key: <m>e_AB</m> = sessionkey
- She derives the ephemeral public key by multiplying <m>e_AB</m> by the generator point: <m>E_AB</m> = <m>e_AB</m> · <m>G</m>
- She derives the shared secret with Bob's node public key: <m>ss_AB</m> = SHA256(<m>e_AB</m> · <m>B</m>)
- She derives the blinding factor for Bob: <m>bf_AB</m> = SHA256(<m>E_AB</m> ‖ <m>ss_AB</m>)

For Charlie (hop 1):
- Alice derives the next ephemeral private key by tweaking Bob's with the blinding factor: <m>e_AC</m> = <m>bf_AB</m> · <m>e_AB</m> (mod <m>n</m>)
- She derives Charlie's ephemeral public key by multiplying <m>e_AC</m> by the generator point: <m>E_AC</m> = <m>bf_AB</m> · <m>E_AB</m>
- She derives the shared secret with Charlie's node public key: <m>ss_AC</m> = SHA256(<m>e_AC</m> · <m>C</m>)
- She derives the blinding factor for Charlie: <m>bf_AC</m> = SHA256(<m>E_AC</m> ‖ <m>ss_AC</m>)

For Dave (hop 2):
- Alice derives the next ephemeral private key by tweaking Charlie's with the blinding factor: <m>e_AD</m> = <m>bf_AC</m> · <m>e_AC</m> (mod <m>n</m>)
- She derives Dave's ephemeral public key by multiplying <m>e_AD</m> by the generator point: <m>E_AD</m> = <m>bf_AC</m> · <m>E_AC</m>
- She derives the shared secret with Dave's node public key: <m>ss_AD</m> = SHA256(<m>e_AD</m> · <m>D</m>)
- No blinding factor for Dave: he's the final hop, so there's no successor that would need one.

Look at what this buys us. Alice computes (<m>E_AB</m>, <m>ss_AB</m>, <m>E_AC</m>, <m>ss_AC</m>, <m>E_AD</m>, <m>ss_AD</m>) in advance, working forward from her session key. She'll pack <m>E_AB</m> into the start of the onion. **That's the only ephemeral key she sends.** The forwarders are going to derive the rest themselves.

## Why the forwarders can advance the chain

Hopefully it's now clear how Alice can compute all the necessary cryptographic material (<m>E_AB</m>, <m>ss_AB</m>, <m>E_AC</m>, <m>ss_AC</m>, <m>E_AD</m>, <m>ss_AD</m>) in advance. But it's less obvious why Bob (or any other node in the payment path) can independently compute the same blinding factor and keep the rest of the packet routing on track.

Let's review what Bob does when he receives the onion:

1. First, Bob takes <m>E_AB</m> from the packet header and runs ECDH with his node private key: <m>ss_AB</m> = SHA256(bob_priv · <m>E_AB</m>). This is the same <m>ss_AB</m> Alice derived as <m>e_AB</m> · <m>B</m>, because scalar multiplication on the curve is commutative.
2. Next, Bob uses the shared secret (<m>ss_AB</m>) and the ephemeral public key (<m>E_AB</m>) to derive the blinding factor: <m>bf_AB</m> = SHA256(<m>E_AB</m> ‖ <m>ss_AB</m>). Both inputs are in scope: <m>E_AB</m> is right there in the packet, and <m>ss_AB</m> he just derived.
3. He then uses the blinding factor to compute the next ephemeral public key, which he'll hand to Charlie: <m>E_AC</m> = <m>bf_AB</m> · <m>E_AB</m>.

When Charlie receives the onion, the ephemeral key slot now contains <m>E_AC</m>. He runs the same routine with his own private key, derives <m>ss_AC</m>, computes <m>bf_AC</m>, advances to <m>E_AD</m>, and forwards to Dave. Dave does the same with his private key to derive <m>ss_AD</m> (he doesn't need to advance the chain, since there's no next hop).

At this point, you may be wondering: "what if Bob gives Charlie a fake ephemeral public key?!" Good question. The answer is mostly economic: Bob is incentivized to play fair. He's locking up his capital to route this payment, and he only earns a fee if the payment completes successfully. If he hands Charlie a bogus <m>E_AC</m>, Charlie's ECDH will derive the wrong <m>ss_AC</m>, his decryption will produce garbage, and the payment fails. Bob would lose his fee and tie up his channel for nothing. There's also a cryptographic check we'll add in chapter 7 (the **HMAC**), but the economic argument is enough for now.
