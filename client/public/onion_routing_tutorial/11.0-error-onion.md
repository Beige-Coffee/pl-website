# The Error Onion

So far we've spent a lot of time reviewing what happens when payments succeed. This is good, as nowadays (dare I say) most payments do succeed! However, sometimes they do not, so the protocol needs a way to send error messages back, allowing parties to quickly unwind the HTLCs. Additionally, error messages help provide the sender with more information as to why the payment failed. For example, a node might be offline, a channel might lack liquidity, a <g>CLTV</g> might be too tight, or an invoice might have expired. In any case, when something goes wrong, the failing hop should tell Alice why. And it should be able to tell Alice *privately*. 

In this chapter, we'll cover BOLT 4 and learn how a node can route an **<g>error onion</g>** backward through the route, privately informing a sender as to why a payment failed.

## How the error onion works

Every hop on the return path wraps the error with its own `ammag` keystream, layer by layer. The failing hop also computes an HMAC with its `um` key, which authenticates the error contents. By the time Alice receives the error, it's wrapped in multiple layers, and she uses her chain of shared secrets to peel them off in order.

This design buys us two important guarantees:

- **Alice can trust the error.** She can tell exactly which hop generated it, and that it wasn't forged or modified along the way back. Why? Because only the failing hop knows its own `um` key.
- **Nobody else can read it.** The hops between the failing one and Alice can't read what the error says, or even tell where it originated. Each hop only knows its own `ammag` and `um` keys, not anyone else's.

## What the failing hop builds

Let's trace what Charlie does when he fails the payment.

1. Construct the **failure message**: a structured byte sequence with the failure code plus any associated data (like a `channel_update` for `UPDATE`-flagged failures).
2. **Length-prefix and pad it to a fixed size** so the on-wire length doesn't leak which failure it is. In our worked example, the unencrypted payload comes out to 260 bytes:
   - `failure_len` (2 bytes): how long the real message is.
   - the `failure_message` itself (`failure_len` bytes).
   - `pad_len` (2 bytes): how many padding bytes follow.
   - that many zero bytes of padding.

   How much padding? Per BOLT 4, `failure_len + pad_len` must come to at least 256, and our example uses exactly 256 (the spec's classic recommendation). That way, the message plus its padding stays a constant size no matter how short the real message is, and the two length fields bring the total to 260. Now, what about failure messages *bigger* than 256 bytes, like one carrying a long `channel_update` plus extra TLV data? Those get padded to a bigger fixed total instead. In fact, the spec's own test vector pads to 1,024 bytes, making a 1,060-byte packet! The lesson for us: when decoding, always read the two length fields. Never assume the size.
3. **Compute an HMAC** over the 260-byte payload with his `um` key: `error_hmac = HMAC-SHA256(um_charlie, payload)`.
4. **Assemble the error packet:** `error_hmac (32 bytes) || payload (260 bytes)`, 292 bytes in all for our example.
5. **Encrypt it** by XOR-ing with his `ammag` keystream, generated to the packet's full length: `wrapped = error_packet ⊕ ChaCha20(ammag_charlie, len(error_packet))`.
6. **Send it back** to Bob, his upstream hop, as the failure data on the <g>HTLC</g>.

Why the explicit length prefix instead of just trimming trailing zeros at decode time? Great question! Some failure messages contain binary data with internal zero bytes. For example, `temporary_channel_failure` carries a serialized `channel_update`, whose bytes can include zeros at any offset. If we just stripped trailing zeros, we'd corrupt those messages. The u16 `failure_len` at the front tells us exactly where the message ends, no guessing required.

## What the upstream hops do

Bob receives `wrapped` from Charlie. He doesn't know what's inside (he doesn't have Charlie's `ammag` key), and BOLT 4 says he doesn't need to. His only job is to wrap one more layer: `wrapped = wrapped ⊕ ChaCha20(ammag_bob, len(wrapped))`. He covers however many bytes arrived, then sends the new `wrapped` upstream to Alice on his own HTLC.

If there were more hops between Alice and Charlie (say, a 5-hop route), each one would wrap with its own `ammag` in turn. By the time the error reaches Alice, it's been XORed with every intermediate hop's keystream plus the originating Charlie's keystream.

Here's the neat part: Alice peels the layers in **the same order** they were applied on the trip back, hop by hop, until she finds the one whose HMAC verifies.

## What Alice does

When Alice receives the wrapped error, she has shared secrets <m>ss_AB</m>, <m>ss_AC</m>, <m>ss_AD</m> (one per hop) and can derive the corresponding `um` and `ammag` keys for each. She loops through her hops in order:

For each hop `i` in order (`i = 0, 1, 2, ..., N-1`), Alice does two things:

1. XOR the wrapped bytes with that hop's keystream, `ChaCha20(ammag_i, len(wrapped))`.
2. Check whether the leading 32 bytes are a valid HMAC over the rest under that hop's `um` key, i.e. whether `HMAC-SHA256(um_i, wrapped[32:])` equals `wrapped[:32]`. If it does, hop `i` is the one that failed: Alice reads the `failure_len` prefix at the front of the payload and pulls out exactly that many bytes as the failure message. If it doesn't, she moves on to the next hop's keys.

Why does this work? The wrapping order from the network's perspective is: Charlie XORs first, then Bob XORs, then... back to Alice. From Alice's perspective on the return, the *outermost* layer is the most recent hop's wrap (Bob's, in our 3-hop example), and the *innermost* is Charlie's (the failing hop). Alice peels in the order Bob → Charlie, which is `i = 0, 1, ...` since hop 0 is the first forwarder. After XOR-ing with hop <m>i</m>'s `ammag`, if hop <m>i</m> was the failing hop, the contents reveal a 32-byte HMAC that verifies under <m>um_i</m>. If not, Alice continues.

What if she peels through every layer and never finds a valid HMAC? Then the error packet is malformed, which means someone along the return path tampered with it (or sent garbage). Remember, nobody can forge a valid error without one of the `um` keys, so a no-match tells Alice the bytes themselves can't be trusted.

## A concrete walkthrough

For our route Alice → Bob → Charlie → Dave with Charlie failing:

1. Charlie builds: `error = HMAC(um_charlie, padded_msg) || padded_msg`. Then wraps: `wire = error ⊕ ammag_charlie_stream`. Sends `wire` back to Bob.
2. Bob wraps again: `wire = wire ⊕ ammag_bob_stream`. Sends to Alice.

Alice's peel:

3. **Iteration i=0 (Bob's keys):** `peeled = wire ⊕ ammag_bob_stream`. Check `HMAC(um_bob, peeled[32:]) == peeled[:32]`. **No**, because Bob isn't the failing hop, so his `um` doesn't authenticate Charlie's error. Continue.
4. **Iteration i=1 (Charlie's keys):** `peeled = peeled ⊕ ammag_charlie_stream`. Now `peeled` is exactly Charlie's original `error` packet. Check `HMAC(um_charlie, peeled[32:]) == peeled[:32]`. **Yes**, Charlie generated this error. Parse the length prefix at `peeled[32:34]` to find the message length, then read that many bytes starting at `peeled[34:]`.

So the failing hop is identified by **whose `um` key verifies**, which Alice checks by trial-and-decrypt as she peels. Trial-and-decrypt might sound expensive, but it's really not! There are at most N hops in the route, the keystream computations are cheap, and each HMAC check is a single SHA256 computation.

<details>
<summary>Why does the failing hop need to wrap with its own ammag if it's already authenticated with um?</summary>

Great question, and there are two reasons. First, an HMAC proves integrity, but it doesn't hide anything! Without the encryption layer, Bob could read "Charlie failed because of channel X" as the error passed through him, which leaks information about Charlie's state. Second, without per-hop encryption, a malicious upstream hop could swap one valid error for another (any old error from any failing hop), and the HMAC tag would still verify. The `ammag` wrap ties the error to its specific position in the route. Only after peeling Bob's layer can anyone inspect the contents, and only then does the `um_charlie` HMAC verify.

</details>

<details>
<summary>What if a malicious forwarder injects a fake error?</summary>

Let them try! A forwarder can write any bytes it likes onto the return HTLC, and Alice will dutifully try to peel them: XOR with `ammag_0` and check the `um_0` HMAC, then XOR with `ammag_1` and check `um_1`, and so on down the chain. But the forwarder doesn't have any of the `um` keys (those require knowing the corresponding shared secrets, which require knowing either the session key or a hop's node privkey). So no manufactured error will ever produce a valid HMAC at any layer. Alice reaches the end of the chain without a match and knows the error was tampered with. From there, she can disconnect the upstream peer or downgrade their reliability score.

</details>

Just one exercise this chapter, and it's a fun one: Alice's peel. You just watched the failing hop build its error packet in the walkthrough above, so we'll focus our coding on the harder (and more interesting!) half: Alice decrypting it. Then it parses the u16 length prefix to recover the exact failure message bytes, even when the message contains internal zeros.
