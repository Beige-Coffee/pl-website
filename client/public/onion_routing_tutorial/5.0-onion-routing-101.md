# Onion Routing 101

In the last chapter we saw how Alice can derive a shared secret with each hop in the payment path. Before we can put these secrets to work, we need to review the protocol that will use them: onion routing!

Lightning uses the **Sphinx** onion-routing construction. <g>Sphinx</g> is a refinement of earlier onion-routing schemes that fixes two problems they had: packet sizes leaking the route length, and header sizes growing with the route. Sphinx addresses the first vulnerability by making every packet exactly 1,366 bytes (we'll dig into this in chapter 7). The second issue is addressed by using a compact header, made possible because, as we saw in the last chapter, each hop can derive its shared secret using one single ephemeral Diffie-Hellman operation.

The rest of this course will dig into how Sphinx works in much more detail. For this chapter, our goal is to build our intuition first. This will allow us to properly contextualize key derivation (which we review next!). Then we'll circle back and dig into the nitty gritty.

Now, back to business! Each hop in the route gets its own **hop payload** - a small block of bytes that tells that hop what to do next (how much to forward, when to expire the HTLC, which channel to send it on, etc.). As we'll see, payloads are wrapped in layers of encryption so that only the intended recipient can read them, which is why it's called an "onion". Each hop payload's contents follow the **TLV (Type-Length-Value)** format, where every field gets a type byte, a length, and the value bytes. You'll see this format throughout the chapter; we'll come back to the byte-level details later, when we peel and parse a hop payload.

## Encryption is XOR with a keystream

First things first, encryption! In earlier sections learned that Alice will use a shared secret to encrypt payment information so that only Bob can read it. However, we didn't learn how that encryption actually happens. Let's do that now.

Sphinx uses a stream cipher (specifically ChaCha20), which produces a **deterministic** stream of pseudo-random bytes given a key. This determinism is very important. Specifically, Alice and Bob can each use the same *shared secret* to independently generate the *exact same* keystream, which can be used to encrypt or decrypt information without ever sending the keystream over the wire.

Once Alice (or Bob) has the keystream, they can encrypt the message by simply **XOR-ing** the plaintext with the keystream, byte-by-byte. The neat thing is, the same operation works both ways. Bob can then XOR the ciphertext with the *same* keystream to decrypt it, revealing the plaintext message. This is because the keystream cancels itself out.

## The anatomy of an onion packet

Before we walk through how Alice builds the onion, let's review what's in the packet that Alice will actually send. Note, in the below diagram, the payload sections for Bob, Charlie, and Dave are visible, but they would be hidden within layer of encryption in a "real" onion.

<onion-packet-anatomy></onion-packet-anatomy>

Every Sphinx onion packet has three parts:
- <anatomy-term region="header">HEADER</anatomy-term> holds the **version byte** (currently `0x00`) and a single **ephemeral public key** for the current hop. Alice puts the first hop's key (`E_AB`) here, and the forwarder combines it with its own node private key (via <g>ECDH</g>) to derive its shared secret with Alice, and from there every key it needs to peel its layer. Before forwarding, the hop advances this key with the <g>blinding factor</g> from chapter 4 and writes the *next* hop's ephemeral pubkey into the packet it sends on, so each hop can derive its own shared secret with no extra coordination.

- <anatomy-term region="payload">PAYLOAD AREA</anatomy-term> carries one **hop payload** per hop, each holding a short **length prefix** (how many bytes of instructions to read), the encrypted **routing instructions** (amount, <g>CLTV</g>, the next hop, etc.), and a **32-byte HMAC** that the next hop will check. The whole buffer is wrapped in one layer of encryption per hop: each forwarder strips one layer off the entire buffer, exposing its own hop payload at the front while the inner layers still cover the rest.

- <anatomy-term region="hmac">HMAC TAG</anatomy-term> is a 32-byte authentication tag that proves the **integrity of the payload area**. HMAC (*Hash-based Message Authentication Code*) combines a secret key with the message bytes to produce a fixed-size tag that only a holder of the shared secret can generate or verify. Before decrypting anything, each hop recomputes this tag over the payload with a key derived from its shared secret with Alice and checks it against the one in the packet; if they don't match, the packet was altered or built wrong and the hop rejects it immediately. (For a deeper dive into how HMAC is constructed, see the [cryptographic primitives chapter](/noise-tutorial/crypto-primitives) of the Noise Protocol course.)

## Alice builds the onion

Now that we have an understanding of an onion packet, let's see how Alice constructs one for the route. Before reading on, ask yourself: **whose payload goes into the onion first?** Does Alice start with Bob (the first hop) or with Dave (the last hop)?

Since onions are peeled hop by hop along the payment path, the outermost layer is what the first hop sees, *so it gets peeled first*. The **innermost** layer is peeled last, *so it has to be added first*. Why? Because Alice builds each layer on top of the prior one, working from the inside out!

There's also a cryptographic reason the order is forced: each hop's <g>HMAC</g> is computed over the encrypted layer *beneath* it, so an inner layer has to exist before Alice can authenticate the layer wrapped around it. We'll see why that nesting matters later in this chapter.

## Each hop peels its layer

Great, we've seen how an onion is built! Now, let's review how an onion is peeled! Broadly speaking, each hop performs the same three actions when they receive a payload:

1. **Verify the outer HMAC**: First, they recompute the HMAC over the encrypted hop payload and compare it against the packet's HMAC tag. If they don't match, they drop the packet.
2. **Decrypt with XOR**: If the HMACs match, they XOR the hop payload with the encryption keystream. This is the same operation Alice used to encrypt this payload! As we learned earlier, XORing it again with the same keystream decrypts it.
3. **Read the hop payload at the front**: Finally, they parse the <g>TLV</g> (amount, expiry, next channel, etc.) and *extract the next-hop's HMAC*. Then, they construct the new 1,300-byte packet, append the header and HMAC, and send it to the next hop.

Pay close attention to what happens at each XOR step... the encryption layer comes off **the entire payload**, *not just the current hop's payload*. More specifically, when Bob XORs his payload, it reveals his plaintext TLV, but Charlie's and Dave's are still encrypted within deeper layers. Once Bob forwards the payload to Charlie, he decrypts and, once again, only sees his payload, with Dave's remaining hidden.

When the packet finally reaches Dave, his hop payload's HMAC field contains 32 zero bytes. Per **BOLT 4**, an all-zero next-hop HMAC signals that the receiving node is the final destination: there's no next hop to verify against, and the payload is meant to be consumed in place. Dave checks the payment hash, verifies the amount, and claims the funds.

## How the per-hop HMACs hold everything together

Understanding the role the HMACs play is crucial to understanding onion routing, as each hop's HMAC commits to all layers within it. Below is a review of the **per-hop HMACs** from our example above, outermost to innermost:

- The packet's outer HMAC is `bob_hmac`. Since this is the last HMAC Alice derives when creating the onion, it commits to all other payload data within the onion.
- Inside Bob's hop payload sits `charlie_hmac`, the tag Charlie verifies after Bob forwards.
- Inside Charlie's hop payload sits `dave_hmac`, visible only after Bob decrypts and forwards.
- Inside Dave's hop payload, the HMAC field is `0x00…` (32 zeros), signaling that Dave is the final hop.

Each one commits to *everything* below it: the hop payload, the next-inner HMAC, and all the encrypted layers underneath.

## A small confession

Sorry in advance, but throughout this chapter we've been quietly abstracting away all of the keys that are used in the Sphinx protocol, leading the reader to believe all the magic happens with just the shared secret. In reality, Sphinx uses distinct named keys with specific roles. If we reused one raw secret for all of the encryption and authentication jobs, we'd be opening ourselves up to some pretty bad attack vectors.

Now that we've reviewed the mechanics of how onion routing works, we're ready to take on the next challenge, digging into each of the five keys that the Sphinx protocol uses.
