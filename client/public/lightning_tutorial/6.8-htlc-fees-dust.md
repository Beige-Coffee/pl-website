## Write A Function To Create HTLC Outputs

To begin this journey, we'll start by implementing a helper function that will produce a list of output dictionaries for our HTLCs. This is very similar to the `create_commitment_outputs` function we implemented earlier. As we'll see shortly, by producing a list of output dictionaries for our HTLCs, we'll be able to easily add them to the list containing our `to_local` and `to_remote` outputs so they can easily be sorted.

<details>
  <summary>Click to see the output dictionary structure</summary>

Each output dictionary represents a commitment transaction output. This structure is specific to the Programming Lightning course.

- `"value"`: An `int` representing the output amount in satoshis.
- `"script"`: `bytes` containing the P2WSH `scriptPubKey` for the HTLC output.
- `"cltv_expiry"`: An `int` representing the HTLC expiry (block height). This is used for sorting outputs.

</details>

For this exercise, let's implement `create_htlc_outputs` in the code editor below. This function takes the following inputs:

- `commitment_keys`: A `CommitmentKeys` object that holds all of the derived public keys for the current channel state. You can learn more about it below.
- `offered_htlcs`: A list of HTLC dicts that we offered to our counterparty. These will be encumbered with an offered HTLC script.
- `received_htlcs`: A list of HTLC dicts that our counterparty offered to us. These will be encumbered with a received HTLC script.

<details>
  <summary>Click to see the CommitmentKeys object</summary>

The `CommitmentKeys` object holds all of the public keys we'll need for any given channel state. In other words, these keys have already been tweaked by the **Per-Commitment Point** and are unique to a specific channel state. You access them as attributes (e.g., `commitment_keys.revocation_key`).

</details>

<details>
  <summary>Click to see the HTLC input dictionary</summary>

Each HTLC in the `offered_htlcs` and `received_htlcs` lists is a dictionary with these keys:

- `"amount_sat"`: `int` - the HTLC amount in satoshis.
- `"payment_hash"`: `bytes` - the 32-byte SHA256 hash of the payment preimage.
- `"cltv_expiry"`: `int` - the CLTV expiry block height for the HTLC.

</details>

## Update Our create_commitment_tx Function

Now, let's put everything together and update the `create_commitment_tx` function to include HTLC outputs. Earlier in the course, we implemented most of this function but left out the HTLC functionality. Now that we've built all the HTLC scripts and helper functions, let's add them to our commitment transaction!

For this exercise, head back to the code editor below and update the function to account for any `offered_htlcs` or `received_htlcs` that may be passed in.

To successfully complete this exercise, you'll need to:

1. Trim dust HTLCs first. For offered HTLCs, trim when `amount_sat < dust_limit + 663 * feerate_per_kw // 1000`. For received HTLCs, trim when `amount_sat < dust_limit + 703 * feerate_per_kw // 1000`.
2. Update the fee calculation to account for only the **untrimmed** HTLCs: `weight = 724 + 172 * num_untrimmed_htlcs`.
3. Create HTLC outputs using the `create_htlc_outputs` function we just implemented, but only for the untrimmed HTLCs.
4. Wrap channel outputs as dicts with `"cltv_expiry": 0`, then combine with HTLC output dicts.
5. Sort all outputs by `(nValue, scriptPubKey bytes, cltv_expiry)`.
6. Extract the `CTxOut` list from the sorted dicts.
