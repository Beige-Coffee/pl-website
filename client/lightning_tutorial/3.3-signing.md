# Transaction Signing

## Why Signing Matters

Every Bitcoin transaction needs signatures to prove that the spender is authorized to move the funds. In Lightning, signing happens in two critical places:

1. **Signing the funding transaction** -- to spend the opener's UTXOs into the channel.
2. **Signing commitment transactions** -- to authorize each state update of the channel.

Both use the same underlying mechanism: **BIP143 SegWit signing** with **ECDSA** signatures.

## BIP143: SegWit Signature Hashing

Before SegWit, the algorithm for computing what gets signed (the "sighash") was inefficient and had quadratic scaling issues for transactions with many inputs. BIP143 introduced a new sighash algorithm for SegWit transactions that is both more efficient and more secure.

The BIP143 sighash preimage is a serialization of the transaction data in a specific order:

```text
 1. nVersion        (4 bytes, little-endian)
 2. hashPrevouts    (32 bytes)
 3. hashSequence    (32 bytes)
 4. outpoint        (32+4 bytes)
 5. scriptCode      (variable)
 6. amount          (8 bytes, little-endian)
 7. nSequence       (4 bytes, little-endian)
 8. hashOutputs     (32 bytes)
 9. nLockTime       (4 bytes, little-endian)
10. nHashType       (4 bytes, little-endian)
```

Let us break down each field:

### hashPrevouts

The double-SHA256 of all input outpoints (txid + vout) concatenated together:

```python
def hash_prevouts(inputs):
    data = b''
    for txid, vout in inputs:
        data += txid + vout.to_bytes(4, 'little')
    return hashlib.sha256(hashlib.sha256(data).digest()).digest()
```

### hashSequence

The double-SHA256 of all input sequence numbers concatenated:

```python
def hash_sequence(inputs):
    data = b''
    for _, _, sequence in inputs:
        data += sequence.to_bytes(4, 'little')
    return hashlib.sha256(hashlib.sha256(data).digest()).digest()
```

### scriptCode

For P2WPKH inputs, the scriptCode is:

```text
0x1976a914 <20-byte-pubkey-hash> 0x88ac
```

For P2WSH inputs (like the funding output), the scriptCode is the full witness script (the 2-of-2 multisig script), length-prefixed as a Bitcoin script.

### amount

The value of the UTXO being spent, in satoshis. This is a critical security improvement over the pre-SegWit algorithm: the signer commits to the exact amount, preventing fee manipulation attacks.

### hashOutputs

The double-SHA256 of all outputs serialized together:

```python
def hash_outputs(outputs):
    data = b''
    for amount, script in outputs:
        data += amount.to_bytes(8, 'little')
        data += len(script).to_bytes(1, 'big') + script
    return hashlib.sha256(hashlib.sha256(data).digest()).digest()
```

## SIGHASH_ALL

The sighash type determines which parts of the transaction the signature commits to. Lightning uses `SIGHASH_ALL` (value `0x01`), which means the signature covers:

- All inputs (via hashPrevouts and hashSequence)
- All outputs (via hashOutputs)
- The specific input being signed (via outpoint, scriptCode, amount, nSequence)

This is the most restrictive sighash type: any modification to any part of the transaction invalidates the signature. This is appropriate for Lightning because both parties need to agree on the exact transaction before signing.

## The Signing Process

Once the sighash preimage is constructed, the signing process is:

1. **Hash the preimage**: Compute `hash = SHA256(SHA256(preimage))`
2. **Sign with ECDSA**: Use the private key to produce an ECDSA signature over the hash
3. **Encode as DER**: The signature is DER-encoded and the sighash type byte is appended

```python
from cryptography.hazmat.primitives.asymmetric import ec
from cryptography.hazmat.primitives.asymmetric.utils import decode_dss_signature

def sign_input(private_key_bytes: bytes, sighash_preimage: bytes) -> bytes:
    # Double-SHA256 the preimage
    sighash = hashlib.sha256(hashlib.sha256(sighash_preimage).digest()).digest()

    # Create ECDSA signature
    private_key = ec.derive_private_key(
        int.from_bytes(private_key_bytes, 'big'),
        ec.SECP256K1()
    )
    der_sig = private_key.sign(sighash, ec.ECDSA(Prehashed(SHA256())))

    # Append SIGHASH_ALL
    return der_sig + b'\x01'
```

## DER Encoding

ECDSA signatures consist of two integers, `r` and `s`. These are encoded in **DER** (Distinguished Encoding Rules) format:

```text
0x30 <total-length>
  0x02 <r-length> <r>
  0x02 <s-length> <s>
```

A few important details:

- If the high bit of `r` or `s` is set, a `0x00` byte is prepended to keep the number positive (DER uses signed integers).
- **Low-S normalization**: Bitcoin requires the `s` value to be in the lower half of the curve order. If `s > n/2`, replace it with `n - s`. This is enforced by BIP62 and prevents transaction malleability through signature manipulation.

## Witness Structure for P2WPKH

For a standard P2WPKH input (like spending a wallet UTXO to fund the channel), the witness is:

```text
[signature] [public_key]
```

The signature includes the sighash type byte appended at the end.

## Witness Structure for P2WSH (Funding Output)

When spending the funding output (a P2WSH 2-of-2 multisig), the witness is:

```text
[empty] [signature_1] [signature_2] [funding_script]
```

The empty element at the beginning is required due to the `OP_CHECKMULTISIG` off-by-one bug. We will cover this in detail when we build the commitment finalization.

## Putting It Together

The complete signing flow for a funding transaction input:

1. Construct the BIP143 sighash preimage with the input's details.
2. Double-SHA256 the preimage to get the sighash.
3. Produce an ECDSA signature using the appropriate private key.
4. DER-encode the signature and append the `SIGHASH_ALL` byte.
5. Place the signature in the witness field of the transaction.
