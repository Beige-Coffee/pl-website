# Session Key Blinding

We have a problem. In the previous chapter, Alice computed a shared secret with Bob using straightforward ECDH: she multiplied her session private key by Bob's public key. But what about Carol and Dave?

Alice could include a separate session public key for each hop, but that would make the onion packet grow with the route length. A larger packet for longer routes would leak information about how many hops remain, defeating the fixed-size privacy guarantee.

This is where the **Sphinx** construction gets clever. Instead of including multiple public keys, Alice derives a **chain of ephemeral keys** from a single session key. At each hop, the session key is deterministically "re-randomized" using a **blinding factor**. The result: one session key in the packet header, but each hop sees a different ephemeral public key.

## The Blinding Factor

After Alice computes the shared secret with a hop, she derives a **blinding factor** from the current ephemeral public key and the shared secret, then uses it to "twist" the session key into a new one:

## The Full Chain

Here's how Alice computes shared secrets for all three hops:

**Hop 1 (Bob):**
1. Ephemeral public key = session public key (the original one)
2. Shared secret = `SHA256(session_private_key * Bob_public_key)`
3. Blinding factor = `SHA256(ephemeral_pubkey || shared_secret)`
4. Next session key = `session_private_key * blinding_factor (mod n)`

**Hop 2 (Carol):**
1. Ephemeral public key = new session public key (after blinding)
2. Shared secret = `SHA256(blinded_session_key * Carol_public_key)`
3. Blinding factor = `SHA256(ephemeral_pubkey || shared_secret)`
4. Next session key = `blinded_session_key * blinding_factor (mod n)`

**Hop 3 (Dave):**
1. Ephemeral public key = twice-blinded session public key
2. Shared secret = `SHA256(twice_blinded_key * Dave_public_key)`

Each hop in the route uses a different ephemeral key, but they all trace back to Alice's original session key through the chain of blinding factors.

## Why Each Hop Can Verify

Here's the crucial insight: each hop can compute the shared secret and the blinding factor **without knowing Alice's private key**. When Bob receives the onion packet, he sees the ephemeral public key in the header. He computes:

1. `shared_secret = SHA256(bob_private_key * ephemeral_pubkey)` (standard ECDH)
2. `blinding_factor = SHA256(ephemeral_pubkey || shared_secret)`
3. `next_ephemeral_pubkey = ephemeral_pubkey * blinding_factor`

Bob then replaces the ephemeral public key in the packet header with `next_ephemeral_pubkey` before forwarding. Carol will see this new key and repeat the process. Nobody needs Alice's session private key to advance the chain.

## Why This Is Compact

This is what makes Sphinx packets efficient. Instead of including a separate public key for each hop (which would require 33 bytes per hop and make the packet grow with the route length), Alice includes a single 33-byte ephemeral public key. Each hop deterministically advances it. The packet stays the same size regardless of route length.

<details>
  <summary>Could each hop derive all the downstream ephemeral keys?</summary>

No. Each hop can only compute the **next** ephemeral key, not any further ones. To compute the key two hops ahead, you'd need the shared secret with the next hop, which requires the next hop's private key. Bob can compute the ephemeral key Carol will see, but he cannot compute the one Dave will see. He'd need Carol's private key for that. This is a critical privacy property: each hop can only see one step ahead in the blinding chain.
</details>
