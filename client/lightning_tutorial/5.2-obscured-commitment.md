# Obscured Commitment Numbers

## Why Obscure the Commitment Number?

Each commitment transaction in a Lightning channel has a sequential **commitment number** starting from 0. This number is important for the revocation mechanism: when a commitment is broadcast, the counterparty needs to identify which commitment it is so they can look up the corresponding per-commitment secret and derive the revocation key.

However, storing the commitment number in plain text would leak information to on-chain observers. Someone watching the blockchain could determine how many state updates a channel had undergone, which reveals information about the channel's activity level.

BOLT 3 addresses this by **obscuring** the commitment number using a shared secret derived from both parties' payment basepoints. The obscured number is encoded in the `nLocktime` and `nSequence` fields of the commitment transaction.

## The Obscure Factor

The obscure factor is computed as the lower 48 bits of the SHA256 hash of both parties' payment basepoints concatenated together:

```text
obscure_factor = SHA256(open_payment_basepoint || accept_payment_basepoint) & 0xFFFFFFFFFFFF
```

The **opener's** (funder's) payment basepoint comes first, followed by the **acceptor's** (non-funder's) payment basepoint. This ordering is determined during the channel open handshake and is consistent for the lifetime of the channel.

```python
import hashlib

def compute_obscure_factor(opener_payment_basepoint: bytes, acceptor_payment_basepoint: bytes) -> int:
    """Compute the 48-bit obscure factor from both parties' payment basepoints."""
    h = hashlib.sha256(opener_payment_basepoint + acceptor_payment_basepoint).digest()
    # Take the lower 48 bits (last 6 bytes)
    return int.from_bytes(h[26:32], 'big')
```

### Who Knows the Obscure Factor?

Both channel parties know both payment basepoints (they were exchanged during channel open), so both can compute the obscure factor. An outside observer who does not know the payment basepoints cannot determine the actual commitment number.

However, this is not strong encryption. If someone knows which two nodes own a channel (which is often public in the Lightning Network), they could potentially compute the obscure factor. The obscuring is meant to provide a layer of privacy, not cryptographic secrecy.

## Encoding in Locktime and Sequence

The obscured commitment number is a 48-bit value, but neither `nLocktime` (32 bits) nor `nSequence` (32 bits) can hold it alone. BOLT 3 splits it across both fields:

- **Upper 24 bits** go into the `nSequence` field of the first input.
- **Lower 24 bits** go into the `nLocktime` field.

### Setting the Obscured Values

```python
def set_obscured_commitment_number(commitment_number: int, obscure_factor: int):
    """Compute the locktime and sequence for a commitment transaction.

    Returns (locktime, sequence) tuple.
    """
    obscured = commitment_number ^ obscure_factor

    # Lower 24 bits -> locktime (with 0x20000000 base)
    locktime = 0x20000000 | (obscured & 0xFFFFFF)

    # Upper 24 bits -> sequence (with 0x80000000 base)
    sequence = 0x80000000 | ((obscured >> 24) & 0xFFFFFF)

    return locktime, sequence
```

### The Base Values

Notice the base values added to each field:

- **Locktime**: `0x20000000` is added to the lower 24 bits. The `0x20` prefix in the high byte signals that this is a commitment transaction (it is an arbitrary convention chosen by the BOLT authors).
- **Sequence**: `0x80000000` is added to the upper 24 bits. The `0x80` prefix sets the disable-flag for `nSequence`-based relative timelocks on this input, which is correct because the commitment transaction input spends the funding output (which does not need a relative timelock).

### Recovering the Commitment Number

Either channel party can recover the commitment number from an on-chain commitment transaction:

```python
def recover_commitment_number(locktime: int, sequence: int, obscure_factor: int) -> int:
    """Recover the commitment number from locktime and sequence."""
    lower_24 = locktime & 0xFFFFFF
    upper_24 = sequence & 0xFFFFFF
    obscured = (upper_24 << 24) | lower_24
    return obscured ^ obscure_factor
```

## Why This Encoding?

You might wonder why BOLT 3 encodes the commitment number in `nLocktime` and `nSequence` rather than in a dedicated output or OP_RETURN. There are several reasons:

1. **No extra output needed**: Using existing transaction fields avoids adding an extra output (which costs more in fees and is more visible).
2. **Compatible with relay rules**: The chosen values for locktime and sequence do not trigger any special behavior in Bitcoin's consensus or relay rules.
3. **Constant transaction size**: The commitment number encoding does not change the transaction's size regardless of the commitment number.

## Worked Example

Suppose:
- Commitment number: `42`
- Obscure factor: `0xA1B2C3D4E5F6`

```text
obscured = 42 XOR 0xA1B2C3D4E5F6
         = 0x2A XOR 0xA1B2C3D4E5F6
         = 0xA1B2C3D4E5DC

Upper 24 bits: 0xA1B2C3
Lower 24 bits: 0xD4E5DC

locktime = 0x20000000 | 0xD4E5DC = 0x20D4E5DC
sequence = 0x80000000 | 0xA1B2C3 = 0x80A1B2C3
```

To recover: XOR the extracted 48-bit value with the same obscure factor to get back `42`.

## Summary

The obscured commitment number scheme:

1. Takes the commitment number (0, 1, 2, ...).
2. XORs it with the obscure factor (derived from both payment basepoints).
3. Splits the result: lower 24 bits into locktime, upper 24 bits into sequence.
4. Adds base values (0x20000000 for locktime, 0x80000000 for sequence).

Both channel parties can compute and recover the commitment number. Outside observers cannot (easily) determine it.
