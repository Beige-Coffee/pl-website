# Build Our Funding Transaction
Now that we have our funding script, let's create our Funding Transaction so we can open our payment channel! Once created, we'll broadcast the transaction in our local Regtest environment to simulate operating a Lightning channel. It'll be fun!

For this exercise, we'll build our very own Funding Transaction using Python. The function will take the following inputs:
- `input_txid`: The TxID of Alice's UTXO (as bytes in internal byte order).
- `input_vout`: The output index of that UTXO.
- `funding_amount`: The amount (in satoshis) to lock in the 2-of-2 multisig.
- `pubkey1`: Alice's (channel opener) funding public key.
- `pubkey2`: Bob's (channel acceptor) funding public key.

To successfully complete this exercise, you'll need to return a `CMutableTransaction` object. For this exercise, we'll keep things simple and won't worry about adding any signature data to the witness, though we will dig into these details shortly!

> ⚠️ For simplicity, this exercise omits fees and change outputs. In practice, you'd add a **change output** to return excess funds and deduct miner fees.

<funding-diagram></funding-diagram>

We'll use the [`python-bitcoinlib`](https://github.com/petertodd/python-bitcoinlib) library to build our transaction. **Hover over the diagram above** to see which library types are used to build each part of the transaction. Here are the key types you'll need:

- **`COutPoint(txid_bytes, vout)`** — References a previous transaction output. The txid must be in internal byte order (the `input_txid` parameter is already in this format).
- **`CTxIn(outpoint, sequence=...)`** — A transaction input. Takes a `COutPoint` and an optional sequence number.
- **`CTxOut(amount, scriptPubKey)`** — A transaction output. Takes an amount in satoshis and a `CScript` for the locking script.
- **`CMutableTransaction(inputs, outputs)`** — Assembles a full transaction from lists of inputs and outputs.

To create the P2WSH output, you'll wrap the funding script hash in `CScript([OP_0, SHA256(funding_script)])`. The `hashlib` module is also available.

Your function should return the `CMutableTransaction` object directly.

#### Question: How can Alice lose all of her funds in this setup?
<details>
  <summary>Answer</summary>

If Bob stops responding or refuses to cooperate, then there's no way for Alice to retrieve her funds from this payment channel! This is because, to spend from a 2-of-2 multisig, you need signatures from *both* parties.
</details>


#### Question: How can we fix this security flaw?

<details>
  <summary>Answer</summary>

There are a few different ways to go about this, but the general solution is that we will need to create a way for Alice to receive a "refund" or unilateral exit from the channel. Can you think of how we can implement this?

</details>
